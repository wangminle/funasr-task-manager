# CLI 串行问题分析报告

**问题：** 用户配置了 3 台 FunASR 服务器（共 12 个并发槽位），但 CLI 工具却串行一个一个执行任务，没有利用后端的并行调度能力。

---

## 一、问题根因分析

### 1.1 CLI `transcribe` 命令的设计缺陷

**源码位置：** `3-dev/src/backend/cli/commands/transcribe.py`

```python
for fp in existing:
    # 1. 上传文件
    file_data = c.client.upload_file(fp)
    
    # 2. 创建任务（只创建一个）
    items = [{"file_id": file_data["file_id"], ...}]
    tasks = c.client.create_tasks(items)
    
    # 3. 等待任务完成（阻塞）
    task = wait_for_task(c.client, task_id, poll_interval=5.0, timeout=3600.0)
    
    # 4. 下载结果
    content = c.client.get_result(task_id, fmt=fmt)
```

**问题：**
- 这是一个 **for 循环串行处理**
- 每个文件都要完整走完 "上传→创建→等待→下载" 流程
- `wait_for_task` 会阻塞直到任务完成，才会处理下一个文件
- 即使传入 10 个文件，也是一个一个串行执行

### 1.2 后端设计是支持并行的

**调度器位置：** `3-dev/src/backend/app/services/scheduler.py`

后端调度器设计：
```python
class Scheduler:
    """LPT + Earliest-Finish-Time + capacity-aware scheduling."""
    
    def schedule(self, tasks, servers):
        # 1. 展开所有服务器的空闲槽位
        # 2. 按任务时长降序排列（LPT）
        # 3. 分配到最早空闲的槽位
        # 4. 多服务器并行执行
```

**API 支持批量创建任务：**
```python
# POST /api/v1/tasks
# body.items 可以包含多个任务
{
    "items": [
        {"file_id": "xxx", "language": "zh"},
        {"file_id": "yyy", "language": "zh"},
        ...
    ]
}
```

**后端能力：**
- 3 台服务器 × 4 槽位 = 12 个并发能力
- 任务调度器会并行分配到不同服务器
- 任务创建后立即进入调度队列

## 二、为什么 CLI 没有利用后端的并行能力？

### 2.1 CLI 的设计理念是"一键转写"

`transcribe` 命令的定位是：
> 一键转写：上传 → 创建任务 → 等待完成 → 下载结果

这是一个 **同步阻塞** 的设计，适合：
- 单个文件快速转写
- 脚本化流程
- 用户希望立即看到结果

**不适合：**
- 批量任务并行处理
- 利用多服务器并发能力

### 2.2 正确的批量处理方式

**方式一：分离提交和等待**

```bash
# 1. 批量上传文件
for file in *.mp4; do
    curl -X POST http://localhost:8000/api/v1/files/upload -F "file=@$file"
done

# 2. 批量创建任务（一次性提交）
curl -X POST http://localhost:8000/api/v1/tasks \
  -H "Content-Type: application/json" \
  -d '{
    "items": [
      {"file_id": "file1", "language": "zh"},
      {"file_id": "file2", "language": "zh"},
      ...
    ]
  }'

# 3. 轮询等待所有任务完成
# 后端会并行调度到 3 台服务器
```

**方式二：使用 `--no-wait` 先提交**

```bash
# 先提交所有任务（不等待）
python -m cli transcribe file1.mp4 --no-wait
python -m cli transcribe file2.mp4 --no-wait
...

# 然后查询任务状态
python -m cli task list --status RUNNING
```

**方式三：浏览器 E2E 方案（参考 Skill）**

Skill 文档描述的是浏览器方案：
1. 打开上传页面
2. **一次选择多个文件**（浏览器多选）
3. 点击提交 → 后端并行创建任务
4. 后端调度器并行分配
5. 轮询等待所有任务完成

## 三、问题总结

### 3.1 CLI 设计问题

| 问题 | 原因 | 影响 |
|------|------|------|
| 串行处理 | for 循环 + wait_for_task 阻塞 | 无法利用多服务器并发 |
| 没有批量模式 | 每次只创建一个任务 | 无法触发后端批量调度 |
| 文档不清晰 | 没有说明 CLI 适合单任务，批量需用其他方式 | 用户容易误用 |

### 3.2 后端设计是正确的

| 能力 | 设计 | 实际可用 |
|------|------|----------|
| 批量任务创建 | API 支持 items 数组 | ✅ |
| 并行调度 | LPT + EFT 算法 | ✅ |
| 多服务器管理 | 12 个并发槽位 | ✅ |
| 任务队列 | PREPROCESSING → DISPATCHED → TRANSCRIBING | ✅ |

### 3.3 正确的批量转写流程

```
1. 批量上传文件 → 10 个文件上传
2. 批量创建任务 → 后端创建 10 个任务，分配 task_group_id
3. 后端调度器 → LPT 排序 + 分配到 3 台服务器的 12 个槽位
4. 并行执行 → 最多 12 个任务同时转写
5. 轮询完成 → 检查所有任务状态
6. 批量下载 → 下载所有结果
```

## 四、改进建议

### 4.1 CLI 改进方案

**增加 `--parallel` 参数：**

```python
# 批量模式
for fp in existing:
    file_data = c.client.upload_file(fp)
    file_ids.append(file_data["file_id"])

# 一次性创建所有任务
if parallel:
    items = [{"file_id": fid, "language": language} for fid in file_ids]
    tasks = c.client.create_tasks(items)
    
    # 并行等待所有任务完成
    await asyncio.gather([wait_for_task(tid) for tid in task_ids])
else:
    # 串行模式（当前实现）
    ...
```

### 4.2 文档改进

README.md 应该明确说明：
- CLI `transcribe` 适合单个或少量文件
- 批量任务应使用 API 批量创建 + `--no-wait` 模式
- 或使用浏览器界面上传多个文件

**结论：**
- Repo 设计没有问题，后端调度器支持并行
- CLI 的 `transcribe` 命令设计是串行的，不适合批量并行任务
- 正确方式是使用 API 批量创建任务，让后端调度器并行分配
- 或使用浏览器界面上传多个文件
