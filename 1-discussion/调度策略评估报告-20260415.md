# 调度策略评估报告-20260415

> 评估日期：2026-04-15
> 评估对象：当前批量调度主链路 `schedule_batch() -> slot_queues -> dispatch -> calibrate_after_completion()`
> 评估范围：`scheduler.py`、`task_runner.py` 以及 2026-04-14 前后的 CLI 批量测试结论

---

## 一、当前策略全景

```text
任务到达 -> schedule_batch() -> slot_queues -> dispatch

1. Quota 分配
   基于 throughput_rtf 计算服务器速度占比

2. LPT + EFT
   任务按估算时长降序
   在 quota 约束内选择最早完成的 slot

3. Slot Queue 流水线补位
   首轮生成每个 slot 的有序队列
   任务完成后直接从本 slot 队列头部补位

4. Work Steal 兜底
   空闲 slot 从其他服务器队列尾部偷取短任务

5. Online Calibration
   calibrate_after_completion() 更新 RTFTracker p90
   并按偏差调整 penalty_factor
```

对应实现：

- `get_throughput_speed()` 使用 `throughput_rtf` 计算配额速度
- `schedule_batch()` 负责 Quota + LPT + EFT
- `build_slot_queues()` 把决策组织为每个虚拟 slot 的队列
- `_dispatch_queued_tasks()` 负责按队列下发和 work steal
- `calibrate_after_completion()` 负责执行后在线校准

---

## 二、优势

### 1. Benchmark 驱动的容量感知较强

- `throughput_rtf` 把单线程时延和并发吞吐区分开，调度器已把它直接用于配额计算。
- 从现有测试观察，三台服务器都守在 benchmark 推荐的并发上限内，没有出现超发导致的明显退化。
- 这说明当前方案在“并发守限”这个底线目标上是达成的。

### 2. LPT + EFT 组合对 makespan 有明显帮助

- LPT 先把大任务压到前面，避免长尾任务最后才开始执行。
- EFT 再在 quota 允许范围内选“最早完成”的 slot，能减少空槽等待。
- 在 12 视频测试中，预估 makespan 151s，实际 120s，虽然偏保守，但排序方向基本正确。

### 3. Slot Queue 机制让调度链路更像流水线

- 当前不是“每轮重新从零调度”，而是首轮规划后，后续优先从 slot queue 补位。
- 这样能显著减少重复调度的开销，也避免空闲 slot 因重新计算而停顿。
- 从已有两批测试结果看，补位过程基本连续，没有明显的“等下一轮再发”的空档。

### 4. Work Steal 提供了运行时兜底能力

- 当某台服务器提前清空自己的队列且仍有空闲 slot 时，可以从其他服务器队列尾部偷任务。
- 2026-04-07 的旧数据里确实触发过多次 work steal，证明这条路径不是摆设。
- 即使在新环境里没有触发，它仍然是抵抗计划偏差和节点差异的安全网。

### 5. 在线校准具备自适应潜力

- `RTFTracker` 用滚动窗口和 P90 统计修正基线。
- `calibrate_after_completion()` 在预测偏慢时增加惩罚，在连续偏快时降低惩罚。
- 这意味着系统不是完全依赖一次性 benchmark，而是允许执行数据逐步纠偏。

## 三、劣势与问题

### 1. Quota 分配的吞吐量量纲有问题

这是当前最核心的问题。

当前 `get_throughput_speed()` 的逻辑是：

```python
tp_rtf = server.throughput_rtf
if tp_rtf and tp_rtf > 0:
    return 1.0 / tp_rtf
```

按当前测试数据举例：

| 服务器 | max_concurrency | throughput_rtf | 当前 speed=1/RTF |
| --- | ---: | ---: | ---: |
| asr-10095 | 2 | 0.0031 | 322.6 |
| asr-10096 | 4 | 0.0185 | 54.1 |
| asr-10097 | 1 | 0.0061 | 163.9 |

问题在于：`throughput_rtf` 已经是并发条件下测得的吞吐指标，直接取 `1 / throughput_rtf` 只反映“单位吞吐指标的倒数”，没有把并发数纳入同一量纲。  
如果目标是比较服务器的**总吞吐能力**，更合理的速度定义应当是：

- `max_concurrency / throughput_rtf`
- 或 benchmark 原始结果中的 `total_audio_sec / wall_clock_sec`

否则会产生一个偏差：并发高但 `throughput_rtf` 数值更大的服务器，可能被系统性低估；并发较低但 `throughput_rtf` 极小的服务器，则可能被高估。

这也是为什么当前现象更像是：

- quota 的理论比例有偏差
- 但后续 LPT + EFT 在第二层把结果“补回来了一部分”

这属于“结果还能工作，但推导并不严谨”的状态。

### 2. Quota 只看当前空闲 slot，调度视野偏短

`_allocate_quotas()` 内部有：

```python
effective_count = min(task_count, total_free)
```

这意味着 quota 首轮只按当前空闲 slot 数量分配，而不是按整个批次任务数做全局规划。

这不是 bug，但会带来两个限制：

- 当任务数远大于当前空闲 slot 时，首轮 quota 更像“局部配额”而非“全局计划”
- 后续任务只能等下一轮 dispatch 时重算，调度器的首轮视野是近视的

因此，当前策略更准确地说是“分轮次局部最优”，而不是“一次性全局最优”。

### 3. LPT 对时长为 0 的任务处理不优雅

调度日志里存在 `dur=0s` 的任务，这通常来自：

- 媒体元数据未成功提取
- `ffprobe` 未获取到时长
- 极短文件被记录为 0

这类任务会在 LPT 排序中堆到最后，随后更容易被放到相对较慢的 slot。  
因为它们通常确实很短，所以影响不大，但它说明调度输入数据还不够稳定。

### 4. Work Steal 触发条件偏保守

当前逻辑是：

- 某服务器有空闲 slot
- 并且 `free_slots > own_active_queues`
- 并且其他服务器的 slot queue 中仍然保留可偷任务

这会导致一个保守约束：

- 如果本机队列还剩 1 个任务，即便该任务预计很慢
- 另一台服务器已经空出来了
- 也未必会触发 steal

换句话说，当前 work steal 更像“自己的队列基本耗尽后再偷”，而不是“谁更早完成谁就主动重平衡”。

### 5. ETA 目前系统性偏保守

从现有两组测试结果看：

| 批次 | 预估 makespan | 实际 makespan | 偏差 |
| --- | ---: | ---: | --- |
| 5x-copies（25 任务） | 53.2s | 41s | 预估偏大约 23% |
| 12 视频 | 151s | 120s | 预估偏大约 21% |

偏大不影响正确性，而且方向上是安全的，但也说明：

- `DEFAULT_OVERHEAD=5s`
- `penalty_factor`
- `P90` 基线

三者叠加后，整体估算有较明显的保守偏置。

| 预期目标 | 状态 | 说明 |
| --- | --- | --- |
| 任务 100% 完成 | 达成 | 两批共 37 个任务全部成功完成 |

综合判断：当前策略是**可工作的、总体合理的**，但还不能称为“理论和工程上都完全闭环”。

## 五、是否需要修改

### 必须修改（P0）

#### 1. 修复 throughput speed 的量纲问题

这是优先级最高的改动。

建议方向：

- 将 `get_throughput_speed()` 从 `1 / throughput_rtf` 改为更接近总吞吐的指标
- 优先考虑 `max_concurrency / throughput_rtf`
- 如果 benchmark 保存了原始吞吐数据，则更建议直接使用 `total_audio_sec / wall_clock_sec`

原因很直接：quota 分配是整个调度第一层，如果第一层的速度比有偏差，后续算法只能被动补偿。

### 建议改进（P1）

#### 2. Quota 分配增加全局任务视角

当前 quota 只按首轮空闲 slot 计算。  
如果要让调度器“一次看到全局”，可以考虑：

- 让 quota 先按总任务数计算全局期望占比
- 再按当前空闲槽位决定首轮能下发多少
- 剩余比例信息保留给后续轮次使用

这样能减少“每轮只看眼前空位”的短视效应。

#### 3. Work Steal 增加进度感知或剩余时间感知

如果某服务器虽然还有自己的 queue，但剩余完成时间明显更长，而另一台服务器已经空闲，那么理论上应该允许更积极地 steal。

可以考虑的方向：

- 基于 `estimated_finish` / `source_remaining` 做更主动的跨队列重平衡
- 扩展 steal 的候选范围，不只盯住“自己的 queue 空了”这一种条件

### 暂不需要修改（P2）

#### 4. ETA 偏大

目前方向安全，用户不会因为过度乐观而被误导，短期内不属于 correctness 问题。

#### 5. 0 秒任务的 LPT 排序

影响面小，优先级低于 quota 和 work steal。

#### 6. 在线校准的 penalty 机制

现阶段整体运行正常，没有证据表明这里是主要瓶颈。

---

## 六、结论

当前调度策略的总体评价是：

- 主体框架合理
- 实际运行稳定
- 并发控制准确
- 两批测试都证明系统可以把任务完整跑通

但当前最关键的问题也非常明确：

> `throughput_speed` 的定义没有把并发能力放进统一量纲，导致 quota 分配比例偏离真实总吞吐能力。

因此，当前系统更像是：

- 第一层 quota 理论上有缺口
- 第二层 LPT + EFT 在执行层面把结果“补得还不错”

这能解释为什么现状“看着还行”，但不应被视为最终正确答案。

最终判断：

- 当前策略可以继续使用
- 但必须先修复 P0 的吞吐量量纲问题
- 修完后，再决定是否继续推进全局 quota 和更积极的 work steal

换句话说，这不是“推翻重做”的问题，而是“先把第一层速度模型算对”。
