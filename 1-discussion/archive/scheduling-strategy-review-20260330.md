# FunASR Task Manager 调度策略评审意见

> **日期**：2026-03-30  
> **评审对象**：`1-discussion/funasr-scheduling-analysis-20260329.md`  
> **评审依据**：现有代码实现、真实测试数据、外部工程实践调研  
> **结论**：文档可作为讨论基线，但不能直接作为实施蓝图。多级反馈队列提案需要降级，ETA 预测和分波调度应提前。

---

## 一、总体评价

这份分析文档的学术框架是扎实的——操作系统调度理论讲解清楚，与 FunASR 场景的映射也基本准确。对 vLLM、Kaldi、ESPnet 等开源项目的横向对比提供了有价值的视野扩展。

但它存在三个结构性问题：

1. **理论先行，落地分析不足**：文档花了大量篇幅介绍经典算法，但对当前系统「已经有什么」的描述不够精确。部分「建议改进」实际上已经存在于代码中（如 RTF 滚动统计、并发惩罚、benchmark 测速），容易误导团队认为需要从零开始。

2. **核心提案（MLFQ）与问题域不匹配**：文档最强烈推荐的「多级反馈队列」恰恰是最不适合当前场景的方案。原因在下文详细展开。

3. **开源项目对比的可迁移性被高估**：vLLM 的 PagedAttention 解决的是 GPU 显存碎片化问题，Kaldi 的 Online Endpointing 解决的是流式解码端点检测问题——这些和我们的「异构服务器池批量任务分派」是完全不同的工程问题。直接类比容易产生错误的优化方向。

## 二、对文档核心提案的逐项评审

### 2.1 多级反馈队列（MLFQ）——建议降级为「待实验验证」

文档给出 ✅✅ 强烈推荐，我的评价是 ⚠️ **不建议在当前阶段引入**。

**核心论点**：MLFQ 要解决的问题（短任务被长任务阻塞的「护航效应」），在我们的并发槽位架构下**基本不存在**。

推理如下：

- 当前系统有 3 台服务器 × 4 并发 = 12 个并行槽位
- LPT 排序后，长任务先占槽位，短任务填充剩余槽位——它们是**同时并行执行**的，不是串行排队
- 在 8 任务 E2E 测试中，最短任务（24s 音频）在 10097 上只花了 21s 完成，并没有等待 1548s 长任务跑完
- 文档举的反例「100 个 1 分钟任务 + 1 个 3 小时任务」中的「其他任务等 3 小时」——这是**单服务器单并发**场景下的推理，不适用于有 12 个并行槽位的真实环境

MLFQ 在操作系统中的核心优势是**时间片抢占**：短进程可以在一个时间片内完成并释放 CPU，而长进程被分到更长时间片的低优先级队列。但 ASR 转写任务是非抢占式的——一旦开始就无法中断。MLFQ 退化成了一个固定优先级的多队列系统，和简单的「优先级 + LPT」没有本质区别，却增加了队列管理、老化规则、阈值调优的复杂度。

Celery 的工程实践也印证了这一点：在真实分布式系统中，异构任务调度通常用**多队列 + 专用 worker** 模式（按任务类型路由），而不是 MLFQ。因为后者的 gaming 和 misclassification 问题在非 OS 场景下更难控制。

**如果未来确实出现短任务被饥饿的现象**，正确的解法不是 MLFQ，而是：
- **保留槽位策略**：预留 1-2 个槽位给短任务，类似 Ray Serve 的 `max_replicas_per_node` 思路
- **分波调度**：每波只放出有限任务，让短任务有机会在下一波被优先调度（设计文档已提到，下文详述）

### 2.2 优先级调度——建议采纳，但需要先想清楚产品语义

文档的 ✅ 评价我同意，优先级是值得做的方向。但实施顺序需要调整。

**关键问题不是代码怎么写，而是产品怎么定义**：

1. **谁决定优先级？** 用户自己标注？系统根据音频时长自动分类？管理员指定？
2. **优先级影响什么？** 是插到队列前面？还是抢占更好的服务器？还是预留专用资源？
3. **用户是否需要感知优先级概念？** 当前产品形态是「上传 → 等待 → 获取结果」，加入优先级后用户心智模型会变复杂

我建议先用最小 MVP 验证价值：

```
Phase 0: 在 Task 模型中增加 priority 字段（默认 NORMAL）
Phase 1: schedule_batch 内部按 priority 排序后再做 LPT
Phase 2: 只有 Phase 1 跑通且真的被用到了，再考虑 UI 暴露和高级规则
```

**模型层改动**（这是真正的前置依赖）：

| 字段 | 类型 | 说明 |
|------|------|------|
| `priority` | `SmallInteger, default=2` | 0=URGENT, 1=HIGH, 2=NORMAL, 3=LOW |
| `queued_at` | `DateTime` | 精确入队时间，用于 aging 和 ETA |

当前 Task 模型没有 `priority` 和 `queued_at`（只有 `created_at`），这两个字段是后续所有调度改进的基础设施。

### 2.3 ETA/ETC 预测——**建议大幅提前优先级**

文档放在阶段 3（低优先级），我建议提到**阶段 1**。

理由：

1. **用户可感知最强**。DoorDash 的实践表明，ETA 准确度是用户信任和留存的核心指标。我们的 SSE 进度推送已有 `eta_seconds` 字段，但当前写入的是 `decision.estimated_duration`（纯处理时长），不包含排队等待时间。

2. **对现有架构侵入最小**。只需要修改两处：
   - `task_runner.py:198`：`task.eta_seconds = int(decision.estimated_duration)` → 改为包含 `estimated_start`
   - `progress.py:calculate_eta()`：QUEUED 状态下使用调度器的 `calculate_task_eta()` 而非简单的 `duration * rtf`

3. **当前 ETA 的具体问题**：

   | 任务状态 | 当前 ETA 来源 | 问题 |
   |---------|-------------|------|
   | QUEUED | `duration_sec * rtf + overhead` | 不含排队等待时间 |
   | DISPATCHED | `decision.estimated_duration` | 不含调度延迟 |
   | TRANSCRIBING | `estimated_total - elapsed` | 这个是准确的 ✅ |

   也就是说，**任务还没开始转写时的 ETA 是不可靠的**，而这恰恰是用户最焦虑想知道答案的时刻。

4. **改进方案**（低风险，可以在下一个迭代直接做）：

   ```
   ETA(QUEUED) = estimated_start + estimated_duration
   ETA(DISPATCHED) = estimated_duration + startup_overhead
   ETA(TRANSCRIBING) = remaining_time  // 已有，准确
   ```

   其中 `estimated_start` 已经在 `ScheduleDecision` 中计算好了，只是没有持久化到 Task 模型。

### 2.4 老化机制——同意延后，但给出明确的触发条件

文档放在阶段 3，我同意这个排序。但要补充**什么时候应该启动老化**的决策标准：

**当以下两个条件同时成立时，才需要引入老化**：
1. 系统引入了优先级机制（否则所有任务优先级相同，不存在饥饿）
2. 实际运行中观察到低优先级任务等待时间超过 SLA 上限（需要先定义 SLA）

在此之前，老化就是过度工程。

### 2.5 资源事件驱动重调度——方向正确，但范围必须限定

文档提到了服务器下线、RTF 变化触发重调度。方向正确，但我要强调一个**绝对限制**：

> **只重排 QUEUED 状态的任务，绝不重排已经 DISPATCHED 或 TRANSCRIBING 的任务。**

当前任务状态机已经编码了这个约束：

```
DISPATCHED → TRANSCRIBING | FAILED
TRANSCRIBING → SUCCEEDED | FAILED
```

没有 `DISPATCHED → QUEUED` 或 `TRANSCRIBING → QUEUED` 的回退路径。如果要加「资源变化重调度」，它只能影响 `QUEUED → DISPATCHED` 这一步的服务器选择，不能撤回已派发的任务。

### 2.6 长音频切块调度——不是调度问题，是产品架构问题

文档从 vLLM Chunked Prefill 和 ESPnet Blockwise Processing 类比到「长音频分块调度」。

我认为这是一个**完全不同的课题**，不应该放在调度优化路线图里。原因：

1. **音频切分涉及内容语义**：在哪里切？句子边界？静音段？切错了会导致转写质量下降
2. **结果合并涉及时间戳对齐**：多个片段的时间戳需要偏移合并，SRT/JSON 格式都要适配
3. **断点续跑涉及状态管理**：一个父任务分成 N 个子任务，父任务的状态应该怎么流转？
4. **FunASR 本身支持长音频**：当前已经成功处理了 5400s（90 分钟）的音频，不存在「无法处理长音频」的问题

这个方向如果要做，应该单独立项为「长音频分片处理架构」，与调度算法优化解耦。

## 三、文档遗漏的关键问题

### 3.1 RTF 基线可信度——当前最大的实际瓶颈

设计文档 `修复迭代开发计划-20260329.md` 已经精确诊断了这个问题：

> 三台服务器部署在同一台物理主机（100.116.250.20），RTF 差异（0.124 vs 0.656 vs 0.737，相差 5-6 倍）极可能不反映真实硬件差异，而是来自 benchmark 时序差异和 GPU 资源竞争。

**这才是当前调度不均的根本原因**——不是算法有问题，是输入数据失真。

quota 分配已经缓解了最严重的症状（保证每台至少分 1 个任务），但根源没解决。如果 RTF 基线不准，任何基于 RTF 的调度优化（包括文档提的所有方案）效果都会打折。

**建议的改进路径**：

1. **Benchmark 改进**：多次采样取中位数，而非单次测量。排除冷启动首次
2. **并行 benchmark 校正**：同时对所有服务器发起 benchmark，消除时序差异
3. **运行时自适应**：当前 `calibrate_after_completion` 已经在做 RTF 滚动更新，但至少需要 3 个样本才生效。可以考虑降低到 2 个样本即开始使用
4. **共享 GPU 场景的 penalty 建模**：如果多个 FunASR 实例共享同一 GPU，`penalty_factor` 应该考虑跨实例的总并发，而不仅是单实例内部的 `running_tasks`

### 3.2 分波调度（Wave-Based Dispatch）——最高 ROI 的下一步

设计文档已经在「后续方向」中提到了分波调度，但分析文档完全没有涉及。我认为这是**当前最值得优先做的调度改进**。

**问题**：当前 `_dispatch_queued_tasks()` 一次性把所有 QUEUED 任务全部调度出去。如果同时来了 20 个任务但只有 12 个槽位，前 12 个按 LPT 分配后，后 8 个直接无法调度——它们要等到下一个 poll 周期（1s 后），此时可能还没有槽位释放。

**更严重的问题**：一次性全量调度时，调度器不知道哪些任务会很快完成。假设有 3 个 10s 短任务和 9 个 1h 长任务，12 个槽位全部被占满。3 个短任务 10s 后完成，释放 3 个槽位——但这时候队列里已经没有待调度任务了，因为全部在第一波就派发出去了。

**分波调度的核心思路**：

```
Wave 1: 派发 N 个任务（N = 可用槽位数或更少）
→ 等待 M 个任务完成
Wave 2: 用新释放的槽位 + 最新的 RTF 数据 重新调度下一批
→ RTF 估算越来越准，后续波次的分配质量越来越高
```

这解决了三个问题：
1. **RTF 冷启动**：第一波的完成数据直接校准后续波次
2. **短任务快速周转**：短任务完成后立即释放槽位给下一波
3. **资源动态适应**：服务器状态变化（负载上升、网络波动）在下一波被自然感知

**实现难度低**：只需要在 `_dispatch_queued_tasks()` 中加上 `limit` 参数，并让任务完成事件触发一次额外的调度检查。

### 3.3 调度决策可观测性——被完全遗漏的工程需求

无论未来怎么改调度算法，当前缺少一个关键能力：**看到调度器为什么这么决策**。

现有的日志（`batch_schedule_plan`、`quota_allocation`）只输出到后端 log，没有结构化存储。当用户问「为什么我的任务等了这么久」时，无法回溯。

建议在 Task 模型或关联表中记录：

| 字段 | 用途 |
|------|------|
| `schedule_reason` | 为什么分配到这台服务器（RTF 对比、EFT 计算） |
| `queue_position_at_dispatch` | 派发时在队列中的位置 |
| `estimated_start_at_dispatch` | 派发时预估的开始时间 |
| `estimated_finish_at_dispatch` | 派发时预估的完成时间 |

这些数据既可用于用户侧的 ETA 展示，也可用于调度算法的离线评估。

| 项目 | 触发条件 | 方法 |
|------|---------|------|

| **MLFQ 评估** | 上述两项都不够用 | 用真实批次数据回放对比 |

### 明确不做

| 项目 | 理由 |
|------|------|
| RR / 抢占式调度 | 任务不可中断 |
| PagedAttention 类比 | 问题域不匹配 |

## 七、结论

这份分析文档的最大价值是提供了一个**讨论框架**：它把调度优化从「感觉哪里不对」提升到了「可以用经典理论逐项评估」的层次。

但它的主要风险是**理论驱动的过度设计**。MLFQ、PagedAttention、Chunked Prefill 这些都是学术上正确的方向，但在我们当前 3 台服务器 × 4 并发 × 非抢占式任务的约束下，性价比远不如「修好 ETA、做好分波、校准 RTF」这些贴地的工程改进。

**推荐的下一步行动**：

1. 把本文档和另一位评审者的意见合并，形成一份「调度优化共识文档」
2. 以「分波调度 + ETA 修正 + 调度日志」为第一个迭代目标
3. 用当前仿真测试框架建立调度质量基线（makespan、短任务 P95、ETA 偏差率）
4. 在有了基线数据后，再决定是否引入优先级和更复杂的调度策略

## 附录 A：外部调研摘要

### A.1 异构批处理调度的工程实践

学术界最新的异构调度研究（2025-2026）主要集中在三个方向：

- **深度强化学习调度**（AQESF, AAAI 2025）：用 Multi-Action PPO 同时优化任务排序和放置，在阿里巴巴真实 trace 数据上验证。但这类方法需要大量历史数据训练，不适合我们当前规模。
- **多目标分解优化**（QLSA-MOEAD, Nature 2026）：同时最小化 makespan 和最大化资源利用率。核心洞察是用 Q-learning 自适应调整搜索方向——这个思路可以简化为我们的「分波调度 + RTF 自适应」。
- **Neural Hyper-Heuristic**（2025）：用遗传算法选择任务优先级规则，再用 EFT 分配资源。与我们的 LPT+EFT 框架高度吻合，主要提升在于任务排序策略的自适应选择。

**关键共识**：所有论文都确认异构调度是 NP-hard 问题，启发式方法（如 LPT+EFT）在实际规模下已经接近最优（84-96% 的最优解近似度）。

### A.2 Ray Serve 的调度策略

Ray Serve 提供了与我们场景最相近的参考：

- **默认 Power of Two Choices**：随机选两个 replica，取负载更低的。简单有效，避免了全局最优搜索的开销。
- **Custom Request Router API**：允许按 throughput、cache affinity 等自定义路由逻辑。这个扩展点设计值得借鉴。
- **Prefix Cache Affinity Routing**：对 LLM 场景，将相似请求路由到缓存命中率高的 replica，减少 60% 的首 token 延迟。

**对我们的启发**：如果未来引入模型缓存或特定语言模型的调度倾向，可以参考 Ray 的 affinity routing 模式。当前阶段不需要。

### A.3 Celery 的优先级和饥饿防护

Celery 的生产实践确认了几个重要原则：

- **多队列 + 专用 worker** 优于 MLFQ：按任务类型/优先级路由到不同队列，各队列有独立的 worker pool
- **prefetch 限制是防饥饿的关键**：控制每个 worker 预取的任务数，避免单个 worker 吃光所有高优先级任务
- **队列长度监控是必须的**：当队列增长速度超过消费速度时触发告警或动态扩容

**对我们的启发**：如果引入优先级，用「多队列路由」模式比 MLFQ 更简单可控。

### A.4 ETA 预测的工程实践

DoorDash 和 HPC 领域的 ETA 预测实践提供了两个层次的方法：

- **简单公式**（我们当前可以直接改进的）：`ETA = queue_wait + processing_time`，其中 `queue_wait = queue_position × avg_task_duration / available_slots`
- **ML 模型**（长期方向）：用历史完成数据训练预测模型，考虑任务特征（音频时长、格式、语言）和系统状态（服务器负载、时段）

**用户体验关键洞察**（来自 Particula 的 AI 任务 UI 研究）：

> 超过 45 秒的任务，标准加载动画会导致高弃出率。有效的做法是：阶段化进度（预处理 → 排队 → 转写 → 完成）、流式更新、可见的队列位置。

我们的 SSE + `format_progress_message()` 已经实现了阶段化进度，但 QUEUED 状态下的 ETA 不准确会降低用户信任。
