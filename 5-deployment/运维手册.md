## 二、部署操作

### 2.1 首次部署

```bash
# 1. 克隆代码
cd /opt/asr-task-manager

# 2. 构建并启动
cd 3-dev/src/backend
docker-compose up -d --build

# 3. 验证服务健康
curl http://localhost:8000/health
# 期望: {"status":"ok"}

# 4. 初始化数据库（首次）
docker-compose exec web alembic upgrade head
```

### 2.2 版本更新

```bash
# 1. 拉取最新代码
git pull origin main

# 2. 重新构建并滚动更新
docker-compose up -d --build web

# 3. 执行数据库迁移（如有）
docker-compose exec web alembic upgrade head

# 4. 验证
curl http://localhost:8000/health
```

### 2.3 回滚

```bash
# 回滚到上一个版本
git checkout <previous-tag>
docker-compose up -d --build web

# 回滚数据库（如需）
docker-compose exec web alembic downgrade -1
```

## 三、配置管理

### 3.1 环境变量

| 变量 | 默认值 | 说明 |
|------|--------|------|
| `ASR_DATABASE_URL` | `sqlite+aiosqlite:///./data/asr_tasks.db` | 数据库连接串 |
| `ASR_REDIS_URL` | `redis://localhost:6379/0` | Redis 连接串 |
| `ASR_UPLOAD_DIR` | `./data/uploads` | 上传文件目录 |
| `ASR_RESULT_DIR` | `./data/results` | 转写结果目录 |
| `ASR_TEMP_DIR` | `./data/temp` | 临时处理目录 |
| `ASR_LOG_LEVEL` | `INFO` | 日志级别 |
| `ASR_LOG_FORMAT` | `console` | 日志格式 (`json`/`console`) |
| `ASR_HOST` | `0.0.0.0` | 监听地址 |
| `ASR_PORT` | `8000` | 监听端口 |
| `ASR_MAX_UPLOAD_SIZE_MB` | `2048` | 最大上传文件大小 (MB) |
| `ASR_TASK_TIMEOUT_SECONDS` | `3600` | 任务超时时间 |
| `ASR_MAX_RETRY_COUNT` | `3` | 最大重试次数 |
| `ASR_HEARTBEAT_INTERVAL_SECONDS` | `30` | 心跳间隔 |
| `ASR_HEARTBEAT_TIMEOUT_SECONDS` | `90` | 心跳超时阈值 |
| `ASR_RATE_LIMIT_ENABLED` | `false` | 是否启用限流器 |
| `ASR_RATE_LIMIT_MAX_CONCURRENT` | `10` | 单用户最大并发任务数 |
| `ASR_RATE_LIMIT_MAX_DAILY` | `100` | 单用户每日最大任务数 |
| `ASR_RATE_LIMIT_MAX_UPLOAD_MB_PER_MIN` | `50` | 单用户每分钟最大上传流量（MB） |

> 说明：限流默认关闭。只有设置 `ASR_RATE_LIMIT_ENABLED=true` 后，429 拒绝和限流指标才会反映真实限流行为。
>
> 说明：`ASR_DATABASE_URL`、`ASR_UPLOAD_DIR`、`ASR_RESULT_DIR`、`ASR_TEMP_DIR` 默认都是相对路径，解析结果取决于进程启动时的当前工作目录。容器内按 `/app` 解析；本地开发建议始终从 `3-dev/src/backend` 启动，避免在仓库根目录生成额外的顶层 `data/` 目录。

## 四、日常运维

### 4.2 日志查看

```bash
docker-compose logs -f web --tail=100

# JSON 格式日志查询（生产环境使用 jq）
docker-compose logs web | jq 'select(.level=="ERROR")'
```

### 4.3 ASR 服务器管理

```bash
# 注册服务器
curl -X POST http://localhost:8000/api/v1/servers \
  -H "Content-Type: application/json" \
  -d '{"server_id":"asr-01","host":"192.168.1.10","port":10095,"protocol_version":"funasr-main","max_concurrency":4}'

# 查看服务器列表
curl http://localhost:8000/api/v1/servers

# 注销服务器
curl -X DELETE http://localhost:8000/api/v1/servers/asr-01
```

### 4.4 数据清理

系统内置文件清理策略，按 TTL 自动清理过期文件。手动清理：

```bash
# 清理超过 7 天的临时文件
docker-compose exec web python -c "
from app.services.cleanup import cleanup_files
import asyncio
asyncio.run(cleanup_files())
"
```

> 补充：调用批量删任务接口时，系统只会物理删除未被任何保留任务引用的源文件；多个任务共享同一 `file_id` 时不会误删共享文件。

开发和测试环境如果需要重置 SQLite 数据和评估当前空间占用，可使用仓库自带技能脚本：

```bash
cd /path/to/funasr-task-manager

# 只读评估 backend/data 当前状态
python 6-skills/reset-asr-db-before-test/scripts/reset_db.py --dry-run

# 重置测试数据库并保留现有服务器配置
python 6-skills/reset-asr-db-before-test/scripts/reset_db.py
```

该脚本面向开发和测试环境，默认针对 `3-dev/src/backend/data/`，不要在生产数据目录上直接使用。

### 4.5 回调 Outbox 运维

回调采用事务型 Outbox：任务状态变更时先写入 `callback_outbox`，事务提交后才尝试立即投递；后台任务执行器随后会周期性重试仍为 `PENDING` 的记录。

建议排查方式：

```bash
# 查看回调投递相关日志
docker-compose logs -f web --tail=200 | grep callback

# 进入容器后检查 callback_outbox 表（SQLite 示例）
docker-compose exec web sqlite3 /app/data/asr_tasks.db "select outbox_id, task_id, status, retry_count, last_error from callback_outbox order by created_at desc limit 20;"
```

重点关注：

- `status=PENDING`：待后台重试
- `status=SENT`：已成功送达
- `status=FAILED`：已达到最大重试次数，需要人工排查下游地址、网络或签名校验

## 五、监控与告警

### 5.1 关键指标

| 指标 | 类型 | 说明 | 告警阈值 |
|------|------|------|----------|
| `asr_tasks_total` | Counter | 任务总数 | — |
| `asr_queue_depth` | Gauge | 队列深度 | > 100 (Warning), > 500 (Critical) |
| `asr_task_duration_seconds` | Histogram | 任务耗时 | P95 > 30min (Warning) |
| `asr_circuit_breaker_state` | Gauge | 断路器状态 | — |
| `asr_task_retries_total` | Counter | 重试总数 | > 5/s (Warning) |
| `asr_server_slots_used` | Gauge | 已用槽位 | — |
| `asr_rate_limit_rejections_total` | Counter | 限流拒绝数 | — |
| `asr_callback_deliveries_total` | Counter | 回调投递次数（按状态） | — |

### 5.2 告警规则

| 告警名 | 条件 | 严重度 |
|--------|------|--------|
| QueueBacklogWarning | queue_depth > 100 持续 5min | Warning |
| QueueCritical | queue_depth > 500 持续 2min | Critical |
| HighFailureRate | 失败率 > 10% 持续 3min | Critical |
| ServerOffline | 心跳超 60s | Critical |
| SlowProcessing | P95 > 30min 持续 10min | Warning |
| HighRetryRate | 重试率 > 5/s 持续 5min | Warning |

### 5.3 Grafana 面板

访问 http://localhost:3001，默认账号 `admin` / `admin`。

推荐面板行：
1. **概览行**：任务总数、成功率、队列深度、活跃服务器数
2. **性能行**：任务耗时分布、RTF 趋势、ETA 偏差
3. **服务器行**：各节点状态、槽位使用、断路器状态
4. **错误行**：失败率、重试率、限流拒绝率

## 六、故障处理

### 6.1 常见故障与处理

| 故障现象 | 可能原因 | 处理方式 |
|----------|----------|----------|
| API 返回 500 | 数据库锁冲突 | 重启 web 容器；检查 SQLite 文件权限 |
| 任务卡在 TRANSCRIBING | ASR 服务器超时 | 检查 ASR 连接；断路器可能已触发 |
| 队列持续积压 | ASR 处理能力不足 | 增加 ASR 节点；提高 max_concurrency |
| Redis 连接失败 | Redis 容器异常 | `docker-compose restart redis` |
| 断路器 OPEN | ASR 节点连续失败 | 检查 ASR 服务；等待恢复后自动 HALF_OPEN |
| 上传返回 413 | 文件超过大小限制 | 调整 `ASR_MAX_UPLOAD_SIZE_MB` |
| 返回 429 | 已启用限流且超出阈值 | 确认 `ASR_RATE_LIMIT_ENABLED=true`；减少请求频率或调整限流配置 |
| 回调长时间未送达 | 下游 webhook 异常或签名校验失败 | 检查 `callback_outbox` 状态、`last_error` 和下游日志 |

### 6.2 紧急恢复

```bash
# 重启全部服务
docker-compose restart

# 仅重启后端
docker-compose restart web

# 强制重建
docker-compose down && docker-compose up -d --build
```

## 七、备份与恢复

### 7.1 数据备份

```bash
# SQLite 数据库备份
docker-compose exec web sqlite3 /app/data/asr_tasks.db ".backup '/app/data/backup_$(date +%Y%m%d).db'"

# 上传文件备份
tar czf uploads_backup_$(date +%Y%m%d).tar.gz -C /var/lib/docker/volumes/ asr-task-manager_app-data

# Redis 备份
docker-compose exec redis redis-cli BGSAVE
```

### 7.2 数据恢复

```bash
# 恢复 SQLite
docker-compose exec web cp /app/data/backup_YYYYMMDD.db /app/data/asr_tasks.db
docker-compose restart web
```

## 八、性能调优

| 参数 | 建议值 | 说明 |
|------|--------|------|
| Uvicorn workers | CPU 核心数 × 2 | `--workers N` |
| max_concurrency | 每 ASR 节点 4-8 | 根据 GPU 性能调整 |
| heartbeat_interval | 30s | 生产环境不低于 15s |
| Redis maxmemory | 512MB+ | 根据任务量调整 |
| SQLite WAL mode | 开启 | `PRAGMA journal_mode=WAL` |
