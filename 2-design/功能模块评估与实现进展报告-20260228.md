### 4.2 需要评估的项（涉及架构或后端扩展）

#### 4.2.1 D-03 阶段指示器（任务状态流）

- **冲突评估**：**无冲突**。后端的 Task 状态机已设计了完整的 6 阶段（PENDING → PREPROCESSING → QUEUED → DISPATCHED → TRANSCRIBING → SUCCEEDED/FAILED），只需前端渲染。
- **技术方案**：
  - 在 TaskDetailView.vue 的状态卡片下方增加 `el-steps` 组件
  - 根据 `task.status` 映射到步骤索引，高亮当前阶段
  - 已完成阶段显示绿色 ✓，当前阶段显示蓝色脉冲动画，未到达阶段灰色

```vue
<el-steps :active="statusToStep(task.status)" finish-status="success" align-center>
  <el-step title="待处理" />
  <el-step title="预处理" />
  <el-step title="排队中" />
  <el-step title="已分配" />
  <el-step title="识别中" />
  <el-step title="已完成" />
</el-steps>
```

#### 4.2.2 D-04 分配服务器 RTF + 处理时间

- **冲突评估**：**轻微冲突**。当前 `TaskResponse` schema 只返回 `assigned_server_id`，不返回服务器的 RTF 数据和已处理/预计时间。
- **技术方案**：
  1. 扩展 `GET /api/v1/tasks/{task_id}` 响应，关联查询 `server_instances` 表的 `rtf_baseline`
  2. 或在前端 TaskDetailView.vue 中额外调用 `listServers()` 获取服务器信息，根据 `assigned_server_id` 匹配
  3. 已处理时间 = `now() - task.started_at`（前端计算即可）
  4. 预计总时间 = `file.duration_sec × server.rtf_baseline`（前端计算）
- **推荐**：方案 2（前端关联），避免修改后端 schema

#### 4.2.3 D-07 转写结果实时预览

- **冲突评估**：**中等冲突**。当前架构下，转写结果是任务完成后一次性返回的完整 JSON。原型设计的是**转写过程中的实时预览**——需要 ASR 服务器在转写过程中返回中间结果。
- **技术方案**：
  - **方案 A（推荐，渐进实现）**：仅在 SUCCEEDED 状态时显示完整结果预览。调用 `getTaskResult(taskId, 'json')` 获取全文后渲染到右侧卡片。时间戳从 `stamp_sents` 字段提取。
  - **方案 B（完整版，需后端改造）**：
    1. 在 `adapters/funasr_ws.py` 的 WebSocket 转写流程中，收到中间结果（FunASR 的 `text` 字段在 2pass 模式下会返回中间结果）时，通过 SSE 推送给前端
    2. 扩展 `ProgressEvent` schema 增加 `partial_text` 字段
    3. 前端在 SSE 连接中接收 `partial_text` 并实时渲染
  - **当前建议**：先实现方案 A（成本低，1 天），方案 B 作为 M2+ 迭代目标

  **后端侧**（以下三种方案选一）：

  | 方案 | 说明 | 复杂度 | 推荐度 |
  |------|------|--------|--------|
  | **A. 内存环形缓冲** | 在 Python 进程中维护 deque，存最近 1 小时的采样点（每分钟一个），重启丢失 | 低 | ⭐⭐⭐⭐⭐（MVP） |

  **推荐方案 A（内存环形缓冲）**：

  ```python
  # services/metric_history.py
  from collections import deque
  from dataclasses import dataclass, field
  from datetime import datetime

  @dataclass
  class MetricSnapshot:
      timestamp: datetime
      queue_depth: int
      avg_rtf: float
      server_rtf: dict[str, float]       # server_id → rtf
      completed_count: int
      failed_count: int

  class MetricHistory:
      def __init__(self, max_points: int = 60):
          self._buffer: deque[MetricSnapshot] = deque(maxlen=max_points)

      def record(self, snapshot: MetricSnapshot):
          self._buffer.append(snapshot)

      def get_series(self, minutes: int = 60) -> list[MetricSnapshot]:
          cutoff = datetime.utcnow() - timedelta(minutes=minutes)
          return [s for s in self._buffer if s.timestamp >= cutoff]
  ```

## 五、FunASR 协议线程数/IO 数探测能力分析

### 5.1 当前协议能力

经过对以下代码和文档的深入分析：

- 本项目 `server_probe.py`（ServerCapabilities 数据类）
- 本项目 `adapters/funasr_ws.py`（FunASR WebSocket 客户端）
- [funasr-client-python](https://github.com/wangminle/funasr-client-python) 的 `server_probe.py`
- [modelscope/FunASR](https://github.com/modelscope/FunASR) 的 runtime 服务端代码

**结论：FunASR 现有 WebSocket 协议不支持探测服务器的线程数和 IO 数。**

### 5.2 详细分析

#### 5.2.1 FunASR WebSocket 协议消息格式

FunASR 的 WebSocket 通信仅支持两种消息方向：

**客户端 → 服务端**：
```json
{
  "mode": "offline",
  "wav_name": "test.wav",
  "wav_format": "pcm",
  "audio_fs": 16000,
  "is_speaking": true,
  "itn": true,
  "chunk_size": [5, 10, 5],
  "hotwords": "..."
}
```
+ 二进制音频数据

**服务端 → 客户端**：
```json
{
  "text": "识别结果文本",
  "mode": "offline",
  "wav_name": "test.wav",
  "is_final": false,
  "timestamp": [[100, 200], [300, 500]],
  "stamp_sents": [{"text_seg": "...", "punc": "...", "ts_list": [...]}]
}
```

协议中**不存在**"查询服务器信息"类型的消息。WebSocket 连接建立后，服务端期望接收音频数据并返回识别结果，没有 `info` / `status` / `capabilities` 等查询命令。

#### 5.2.2 FunASR 服务端启动参数

FunASR C++ Runtime 的服务端启动时确实有线程相关参数：

| 启动参数 | 说明 |
|----------|------|
| `--worker_threads` | 工作线程数 |
| `--io_threads` | IO 线程数 |
| `--concurrent_vad` | VAD 并发数 |
| `--concurrent_asr_online` | 在线 ASR 并发数 |
| `--concurrent_asr_offline` | 离线 ASR 并发数 |
| `--decoder_thread_num` | 解码线程数 |

但这些都是**启动参数**，仅在服务端进程初始化时设置，**不通过 WebSocket 协议暴露给客户端**。

#### 5.2.3 当前的探测能力

通过 `server_probe.py` 的分级探测机制，我们目前能获取的信息：

| 能力 | 可探测 | 方式 |
|------|--------|------|
| 服务器是否在线 | ✅ | WebSocket 连接测试 |
| 支持的模式 (offline/online/2pass) | ✅ | 发送探测音频，分析响应 |
| 协议版本 (新版/旧版) | ✅ | 分析 `is_final` 语义推断 |
| 时间戳能力 | ✅ | 检查响应中的 `timestamp`/`stamp_sents` |
| 探测延迟 | ✅ | `probe_duration_ms` |
| **线程数** | ❌ | 协议不支持 |
| **IO 数** | ❌ | 协议不支持 |
| **GPU 信息** | ❌ | 协议不支持 |
| **当前负载** | ❌ | 协议不支持 |
| **并发限制** | ❌ | 协议不支持（只能靠配置指定 `max_concurrency`） |

### 5.3 要支持线程数/IO 探测的技术路径

| 方案 | 说明 | 复杂度 | 是否需要修改 ASR 服务端 |
|------|------|--------|------------------------|
| **A. 静态配置** | 在 `server_instances` 表的 `labels_json` 中手工配置 `{"threads": 16, "io_threads": 4, "gpu": "A100"}` | 最低 | **否** |
| **B. Sidecar HTTP API** | 在每台 ASR 服务器旁部署一个轻量 HTTP 服务，读取 `/proc` 信息或服务端配置文件，暴露 JSON API | 中 | **否**（但需要部署额外服务） |
| **C. 修改 FunASR 服务端** | Fork FunASR runtime，添加特殊 WebSocket 消息类型（如发送 `{"type": "server_info"}`，返回 `{"threads": 16, ...}`） | 高 | **是** |
| **D. Prometheus + Node Exporter** | ASR 服务器部署 Node Exporter + DCGM Exporter，任务管理器通过 Prometheus API 查询 CPU/GPU/线程指标 | 中-高 | **否**（但需要部署监控栈） |

**推荐路径**：

1. **短期（当前阶段）**：**方案 A 静态配置**。注册服务器时由管理员手动填写 `max_concurrency`（已实现），并在 `labels_json` 中补充硬件信息。前端编辑对话框增加这些字段。
2. **中期**：**方案 D Prometheus**。如果已经部署了 Prometheus 监控栈（设计方案 M3 已规划），可以直接查询 Node Exporter 的 `node_cpu_seconds_total`、进程级线程数等指标。
3. **长期**：如果 FunASR 上游社区接受 PR，可以提交方案 C 的协议扩展。

### 5.4 设计文档中 `max_threads` 字段的处理建议

设计方案 7.1 节定义了 `ASRServer.max_threads` 字段（该服务器最大并发线程数），在当前实现中已映射为 `ServerInstance.max_concurrency`。这个值目前只能通过**手动配置**或**推断**获得：

- **手动配置**：注册/编辑服务器时由管理员填入
- **经验推断**：通过测速（`offline_light` 探测）获取 RTF 基线后，结合文件时长和已知的并行上限，间接推断最大并发

**`max_concurrency` 的含义**应该从"服务器线程数"转化为"任务管理器允许同时向该服务器发送的最大任务数"。这是调度层面的软限制，比底层线程数更实用——因为即使知道服务端有 16 个 worker_threads，也不意味着能同时发 16 个大文件（内存/显存可能不够）。实际最优并发数需要通过测速和监控来确定。
