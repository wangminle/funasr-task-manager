| 维度 | 核心诉求 |
|------|----------|
| **定位** | ASR 任务中转适配层，连接多用户并行输入与多台 ASR 服务器资源 |

### 2.2 各方案优势分析

#### 报告①的优势
1. **容量估算严谨**：明确给出了"500任务/天、50槽位、12h窗口清空"的 SLA 推导，RTF 上限计算（≤1.2），这是其他报告缺失的关键决策依据
2. **调度算法有深度**：提出了 LPT（最长处理时间优先）+ 最早完工分配的调度策略，比简单 FIFO 更优，并给出了数学依据
3. **已有项目复用分析到位**：明确指出 funasr-client-python 中 `protocol_adapter.py` 和 `server_probe.py` 的复用价值和迁移方式
4. **大文件上传建议务实**：推荐 tus 可续传上传协议，对长音视频场景非常关键
5. **窗口化批处理思路**：提出"夜间批处理"调度模式，适合"头天准备→次日出结果"的业务场景

#### 报告②的优势
1. **接口设计最完整**：TypeScript 类型定义覆盖了 Task/Node/Batch/Event/Result 全量类型，代码可直接参考
2. **双模式调度创新**：Push/Pull 双模式 + 自适应选择，适合不同规模场景
3. **容错机制最全面**：断路器 + 幂等性保证 + 事件溯源三层保障，工业级设计
4. **可插拔队列设计**：内存队列 / Redis Streams 统一接口，渐进式升级路径清晰
5. **多种负载均衡策略**：轮询、加权轮询、最少连接、响应时间、能力感知、一致性哈希六种策略可选

#### 报告③的优势
1. **执行计划最务实**：M1-M4 里程碑拆解到天级，验收标准明确可量化
2. **渐进式演进策略**：以方案B（轻量级）为起点，方案A（工业级）为演进目标，风险可控
3. **决策矩阵量化**：五维度评分（可行性/复杂度/成本/扩展性/风险），结论有据可依
4. **YAML 配置驱动协议适配**：新增协议无需改代码，降低维护成本
5. **切换阈值明确**：队列长度>500、写冲突率>5%、日任务量>2000 等具体指标触发迁移

#### 报告④的优势
1. **外部任务ID风险洞察最深**：明确提出"下游 ASR 的 TaskId 不能作为业务唯一 ID"（腾讯云 TaskId 24h 有效且可跨日重复），要求维护 `external_task_id ↔ task_id` 映射——这是其他三份报告均未充分关注的关键风险
2. **两条链路解耦设计**：将"文件入库链路"与"任务创建链路"显式拆分，前者做 I/O 密集的上传/校验/元信息抽取，后者做轻量的任务参数绑定和入队，系统弹性更好
3. **Webhook 回调 + Outbox 模式**：引入 Outbox 表 + event_id 幂等投递，解决"事务已提交但回调未发出"的一致性问题，工程成熟度最高
4. **数据库设计最完整**：files / tasks / task_events / server_instances / server_metrics 五表结构，字段级索引建议，ULID/UUIDv7 编号策略，task_events 审计表可做完整状态溯源
5. **两层进度策略**：状态进度（阶段区间）+ 估算进度（RTF×时长）双层推进，比单一百分比更可解释
6. **结果 URL 过期风险处理**：明确指出阿里云等 result_url 可能 24h 过期，管理器必须"尽快拉取→持久化"
7. **三种架构方案对比**：集中式调度器 / 队列微服务 / K8s 原生批处理三方案用表格对比可扩展性/容错/复杂度/运维/延迟/吞吐六维度，决策依据最全
8. **安全设计最具体**：API Token、HMAC 回调签名、用户级权限隔离、三维度限流（上传带宽/并发任务/日任务量），其他报告几乎未触及
9. **开源项目调研最全**：覆盖 13 个项目（Celery/RQ/Dramatiq/Prefect/Airflow/Argo/Kueue/Volcano/Ray/Triton/Whisper/FunASR/WeNet/Kaldi），含活跃度、许可证、借鉴点与局限，可操作性强

### 2.3 各方案不足分析

#### 报告①的不足
1. **未给出具体代码结构和项目骨架**：偏宏观设计，缺乏落地的目录结构和模块拆分
2. **任务队列推荐 Taskiq 但缺乏论证**：Taskiq 社区相比 Celery/Dramatiq 更小，生产案例不足
3. **进度推送未充分展开**：提到 SSE/WebSocket 但未给出事件模型的具体字段设计

#### 报告②的不足
1. **技术栈偏差**：使用 TypeScript/Node.js，而需求明确"以 Python 起步"——整个代码设计需要翻译为 Python
2. **过度设计风险**：Gossip 协议、事件溯源、一致性哈希等对初期场景过于复杂
3. **未考虑文件管理**：缺少文件上传、存储、清理等设计
4. **未考虑音频元信息提取**：缺少 ffprobe 集成、文件时长检测等关键环节
5. **FunASR 协议适配过于简化**：WebSocket 客户端代码未处理 `is_final` 语义差异（已有项目已解决的核心问题）

#### 报告③的不足
1. **初期不使用任务队列中间件**：用 SQLite + 文件锁做队列，在并发场景下可靠性存疑
2. **调度策略偏简单**：加权公平随机选择，缺乏报告①的 LPT+最早完工等优化
3. **前端方案未展开**：未提及前端框架选型和核心页面设计
4. **可观测性设计偏轻**：仅 structlog + Prometheus 端点，缺乏实际告警策略

#### 报告④的不足
1. **MVP 工时偏乐观**：里程碑 A 仅估 2-3 人周即完成"任务最小闭环"，但含文件入库+对象存储+状态机+前端轮询，实际可能 3-4 周
2. **默认推荐 PostgreSQL**：未给出 SQLite → PG 的渐进路径，初期部署依赖偏重（需 PG + Redis + MinIO 三个外部服务）
3. **协议适配层偏理论**：Adapter 接口定义（submit/query/cancel/normalize）清晰，但未处理 FunASR `is_final` 语义差异、SenseVoice 参数兼容等已有项目已验证的实战问题
4. **调度算法未深入**：给出了 Least-Load + Min-ETA 伪代码但缺少 LPT 全局优化思路（报告①更优）
5. **未分析已有项目 funasr-client-python 的复用价值**：错过了最关键的研发加速来源
6. **过度覆盖 K8s/Volcano/Kueue 场景**：对当前项目规模（几台到十几台服务器）属于过度设计，引入不必要的决策负担

## 四、已有项目复用分析：funasr-client-python

> 项目地址：https://github.com/wangminle/funasr-client-python

### 4.1 项目概述

funasr-client-python 是一个基于 Tkinter 的 FunASR GUI 客户端（V3），虽然 UI 层不适用于本项目，但其中**与 GUI 无关的核心模块**已经解决了本项目最关键的技术难题——FunASR 新旧协议适配。

### 4.2 可直接复用的模块（不需要重新设计和实施）

#### (1) `protocol_adapter.py` → 新项目 `adapters/funasr_ws.py`

**已解决的核心问题**：
- **`is_final` 语义差异修复**：新版 FunASR runtime 在 offline 模式下 `is_final` 始终为 `False`，旧版为 `True`。该模块通过 `_should_complete()` 方法统一了结束判定逻辑，避免客户端无限等待
- **消息构建兼容**：`build_start_message()` 根据服务端类型自动决定是否下发 SenseVoice 参数（避免旧服务端拒绝未知字段）
- **宽容结果解析**：`parse_result()` 兼容 `text`、`stamp_sents`、`text_2pass_offline` 等多种响应格式
- **布尔值宽容转换**：`_coerce_bool()` 处理 bool/int/str 各种类型的 `is_final` 字段

**复用方式**：
- 将 `ProtocolAdapter` 类重构为异步版本，去除 GUI 依赖
- `ServerType` 枚举、`RecognitionMode` 枚举、`ParsedResult` 数据类可直接使用
- `MessageProfile` 作为 ASR 请求参数的标准格式

#### (2) `server_probe.py` → 新项目 `services/probe.py`

**已解决的核心问题**：
- **分级探测**：三级探测（仅连接 / 离线轻量 / 2pass 完整），耗时从 <1s 到 3-5s 可控
- **能力发现**：自动检测服务器支持的模式（offline/online/2pass）、时间戳能力、协议语义
- **服务端类型推断**：基于 `is_final` 语义推断新版/旧版服务端，输出 `inferred_server_type`
- **探测缓存**：24 小时有效期，减少重复探测开销
- **序列化/反序列化**：`ServerCapabilities.to_dict()` / `from_dict()` 支持持久化

**复用方式**：
- `ServerProbe` 类可直接用于新项目的"节点注册与健康检查"模块
- `ServerCapabilities` 映射到资源注册表的节点能力描述
- 分级探测逻辑映射到调度器的"快速健康检查"和"深度能力评估"两种策略
- 探测结果中的 `is_final_semantics` 直接驱动协议适配器的策略选择

#### (3) 测速与 ETA 机制 → 新项目 `services/benchmark.py`

**已实现的能力**：
- 上传速度测试 + 转写速度测试
- 按文件真实时长动态估算等待超时
- 无法获取时长时的 20 分钟兜底策略

## 五、行业架构方案调研

### 5.1 核心架构模式：Web-Queue-Worker

经调研，**几乎所有**生产级离线批量转写系统都采用 **Web-Queue-Worker** 三层架构：

```
┌─────────────┐     ┌──────────────┐     ┌────────────────┐
│  Web Layer  │────▶│ Message Queue│────▶│  Worker Layer  │
│  (FastAPI)  │     │ (Redis/MQ)   │     │ (Celery/等)    │
│             │◀────│              │◀────│                │
│ - 文件上传  │ SSE │              │     │ - ASR调度      │
│ - 进度查询  │     │              │     │ - 协议适配     │
│ - 任务管理  │     │              │     │ - 结果回写     │
└─────────────┘     └──────────────┘     └────────────────┘
```

**各层职责**:

| 层级 | 职责 | 关键设计点 |
|------|------|-----------|
| **Web 层** | 接收用户请求（上传文件、查询进度、管理任务）；验证输入后快速入队返回 | 不做重计算，保持API响应 < 200ms |
| **Queue 层** | 持久化任务消息；解耦生产者与消费者；支持优先级（预留） | 任务不丢失，服务重启后可恢复 |
| **Worker 层** | 从队列取任务；选择最优ASR服务器；执行协议适配；管理转写流程；回写结果 | 可水平扩展Worker数量 |

**行业验证**:
- **AssemblyAI**：异步批处理架构，提交数千文件后总耗时取决于最长单文件而非所有文件之和。RTF 低至 0.008x。
- **微软 batch-processing-kit**：通用批处理框架 + 多端点分发 + 容错监控。
- **SaladCloud**：GPU推理进程 + I/O进程分离，AWS SQS 队列调度。
- **印度法院系统（arXiv 2507.01021）**：VAD 分段后并行转写，覆盖 15% 的印度法庭。

### 5.2 本项目的特殊性：中转适配层

与上述项目的关键区别在于：**本项目自身不做ASR推理，而是作为中转适配层调度外部ASR服务器**。这意味着：

1. **Worker 不消耗 GPU**：Worker 的工作是"把文件发给 ASR 服务器 → 等待结果 → 收集返回"，是 I/O 密集型而非计算密集型。
2. **瓶颈在 ASR 服务器**：调度的核心是"把对的任务分给对的服务器"，而非"如何高效利用本地 GPU"。
3. **需要协议适配**：不同 ASR 服务器可能运行不同版本的 FunASR（新旧 WebSocket 协议），需要统一抽象。

因此，架构可以简化为：

```
                          ┌──────────────────────────────────────┐
                          │      ASR 任务中转适配层              │
┌──────────┐              │                                      │         ┌────────────┐
│ 用户 A   │──上传──┐     │  ┌─────────┐  ┌────────┐  ┌──────┐ │    ┌───▶│ ASR Server │
│ 用户 B   │──上传──┤────▶│  │ Web API │─▶│Task    │─▶│调度器│─┼────┤    │ #1 (新协议)│
│ 用户 C   │──上传──┘     │  │(FastAPI)│  │Queue   │  │      │ │    │    └────────────┘
│  ...     │              │  └─────────┘  │(Redis) │  │      │ │    │    ┌────────────┐
│          │◀──进度/结果──│  ┌─────────┐  └────────┘  │      │─┼────┼───▶│ ASR Server │
│          │   (SSE)      │  │ 文件    │  ┌────────┐  │      │ │    │    │ #2 (旧协议)│
│          │              │  │ 存储    │  │任务 DB │  └──────┘ │    │    └────────────┘
└──────────┘              │  └─────────┘  └────────┘           │    │    ┌────────────┐
                          │                                      │    └───▶│ ASR Server │
                          └──────────────────────────────────────┘         │ #3 (新协议)│
                                                                           └────────────┘
```

### 5.3 进度推送方案对比

| 方案 | 方向 | 复杂度 | 浏览器兼容 | 适用场景 |
|------|------|--------|-----------|----------|
| **轮询 (Polling)** | 单向 | 最低 | 100% | 简单场景，对实时性要求不高 |
| **SSE (Server-Sent Events)** | 服务端→客户端 | 低 | 97% | 进度更新、日志流（**推荐**） |
| **WebSocket** | 双向 | 中 | 99% | 需要客户端主动发消息的场景 |

**推荐方案**: **SSE 作为主方案，轮询 API 作为降级方案**。理由：
- 进度推送是**单向**的（服务端→客户端），不需要 WebSocket 的双向能力
- SSE 基于标准 HTTP，天然兼容防火墙/代理，浏览器内置自动重连
- 实现复杂度远低于 WebSocket，FastAPI 原生支持 `StreamingResponse`
- fun-asr-docker 项目已验证此方案可行

## 七、核心模块详细设计思路

### 7.1 ASR 服务器资源管理器 (Resource Manager)

```python
# 概念模型
class ASRServer:
    id: str
    host: str
    port: int
    protocol_version: str    # "v1" | "v2" 新旧协议标识
    max_threads: int          # 该服务器最大并发线程数
    current_load: int         # 当前已占用线程数
    avg_rtf: float            # 该服务器的平均实时因子
    status: str               # "online" | "offline" | "degraded"
    last_heartbeat: datetime  # 最后心跳时间
```

**资源探测机制**:
1. **心跳检测**：定时（每 10-30 秒）向每台 ASR 服务器发送探测请求，确认在线状态
2. **负载上报**：ASR 服务器返回当前线程占用情况（如果 FunASR 服务端支持），或由适配层自行记账
3. **状态缓存**：服务器状态缓存在 Redis 中，避免每次调度都做探测

### 7.2 协议适配器 (Protocol Adapter)

采用**策略模式**，为每种协议版本实现独立的适配器：

```python
class ASRProtocolAdapter(ABC):

        ...

class FunASRWebSocketV2Adapter(ASRProtocolAdapter):
    """FunASR 新版 WebSocket 协议适配器"""
    ...

class FunASRWebSocketV1Adapter(ASRProtocolAdapter):
    """FunASR 旧版 WebSocket 协议适配器"""
    ...

# 未来可扩展：
# class WhisperRESTAdapter(ASRProtocolAdapter): ...
# class FunASRgRPCAdapter(ASRProtocolAdapter): ...
```

### 7.3 任务调度器 (Task Scheduler)

#### 容量估算与 SLA（采纳自报告①）

假设稳定后：每天 ~500 个任务，平均 2 小时音频/任务，总槽位 50。

| SLA 窗口 | 需要的总体倍速 | 对 RTF 的要求 |
|-----------|---------------|--------------|
| 24h 清空 | 200h/24 = 8.3x | RTF ≤ 50/8.3 ≈ 6.0 ✅ 宽裕 |
| 16h 清空 | 1000h/16 = 62.5x | RTF ≤ 50/62.5 ≈ 0.8 |
| **12h 清空** | 1000h/12 = 83.3x | RTF ≤ 50/83.3 ≈ **0.6** |

> 关键指标 RTF（Real-Time Factor）= 处理耗时 / 音频时长。RTF=0.5 表示 2 倍速处理。
> 建议明确 SLA 窗口后，以实测 RTF 验证算力是否足够。

#### 调度算法（采纳报告①的 LPT + 最早完工，优化版）

**v1 推荐策略：LPT + Earliest-Finish-Time**

```
调度流程:
1. 为每个任务估算处理时间：p(i,s) = duration(i) × rtf_p90(s) + overhead(s)
2. 将每个 server 的 slots 拆为 N 个"虚拟机器"
3. 按预计耗时从大到小排序（LPT: Longest Processing Time First）
4. 依次将任务分配给"预计最早空闲"的虚拟机器
5. 在线修正：每完成一个任务，用实际耗时更新 rtf 的滚动统计
6. 若某 server 健康下降（超时/失败率上升），拉高其 rtf_p90 或临时剔除
```

**为什么比简单 FIFO 更好**：
- 大幅降低"最后剩下几个超长任务导致清空延迟"的风险（尾部延迟）
- 计算开销极低，窗口开始时一次性生成分配计划即可
- 不需要切分文件，完整文件直接调度

**预估完成时间算法（增强版）**:

```
eta(task_i) = queue_time + asr_time + overhead
其中:
  queue_time = Σ(前方任务预估耗时) / available_slots   (队列前方等待)
  asr_time = duration(i) × rtf_p90(server)               (本任务处理)
  overhead = upload_time + transcode_time + protocol_time  (固定开销)
```

RTF 统计采用**滚动窗口 p90**（取近 N 次任务的 90 分位），调度用 p90 做保守估计。

**ETA 预测与实测偏差的自动校准机制**：

```
校准流程（每完成一个任务自动触发）:
1. 记录 actual_duration（实际耗时）和 predicted_duration（预估耗时）
2. 计算偏差比: deviation = actual / predicted
3. 更新服务器的 RTF 滚动窗口:
   rtf_window[server_id].append(actual_duration / audio_duration)
   保留最近 N=50 次记录
4. 重新计算 rtf_p90 = percentile(rtf_window, 90)
5. 若 |deviation - 1.0| > 0.3（偏差超过 30%），触发告警:
   - 可能原因: 服务器负载变化、并发数增加、网络波动
   - 自动操作: 将该服务器的 penalty_factor 上调 20%
6. 若连续 10 次 deviation < 0.7（实际比预测快 30%+），下调 penalty_factor 10%
7. 定期（每 4 小时）对空闲节点执行 benchmark 任务，用标准音频校正 rtf_baseline
```

> ETA 误差目标：稳态下 ≤ 30%。初期可用简单线性校准，后期可引入加权指数平滑（EWMA）。

**后期扩展（本期不实现，仅预留接口）**:
- 优先级队列：高优先级任务插队（双队列方案，参考报告①6.4.4）
- 亲和性调度：特定类型文件（如方言）分配给特定服务器
- 窗口化批处理：如果业务模式是"头天准备→次日出结果"，可在固定窗口统一调度

### 7.4 文件预处理模块

```
用户上传文件 → 格式检测 → 元信息提取 → 格式转换(可选) → 入库 → 入队
```

- **格式检测**：使用 `ffprobe`（ffmpeg 套件）检测文件格式、时长、编码、采样率
- **元信息提取**：提取并存储文件名、时长、文件大小、音视频类型、声道数等
- **格式转换**：如果 ASR 服务器要求特定格式（如 16kHz 单声道 WAV），在此阶段使用 `ffmpeg` 转换
- **依赖**: `ffmpeg`（系统级）+ `python-ffmpeg` 或 `pydub`（Python 封装）

### 7.5 数据库模型设计（增强版，采纳自报告④）

> **关键设计决策**（采纳自报告④）：
> 1. **内部 ID 使用 ULID/UUIDv7**：时间有序 + 全局唯一，便于分页和排序
> 2. **File 与 Task 分离**：一个文件可关联多个任务（重试/不同模型），避免冗余
> 3. **external_task_id 不做业务主键**：下游 ASR 的 TaskId 可能 24h 过期或跨日重复（如腾讯云），必须维护映射
> 4. **task_events 审计表**：完整状态溯源，支持 event_id 幂等去重
> 5. **两层进度模型**：状态进度（阶段区间）+ 估算进度（RTF×时长）

#### files 表（文件与元信息）

| 字段 | 类型 | 约束 | 说明 |
|------|------|------|------|

**Task.status 状态机**：

```
PENDING → PREPROCESSING → QUEUED → DISPATCHED → TRANSCRIBING → SUCCEEDED
                                                              → FAILED
                                        ← (重试回退) ←──────────┘
PENDING / QUEUED → CANCELED（用户取消）
```

| 状态 | 进度区间 | 说明 |
|------|----------|------|
| PENDING | 0% ~ 5% | 已上传，等待预处理 |
| PREPROCESSING | 5% ~ 15% | ffprobe 元信息提取 / ffmpeg 格式转换 |
| QUEUED | 15% ~ 20% | 已入任务队列，等待调度 |
| DISPATCHED | 20% | 已分配服务器，等待 ASR 开始 |
| TRANSCRIBING | 20% ~ 95% | ASR 执行中（按 RTF×时长 线性推进） |
| SUCCEEDED | 100% | 转写完成 |
| FAILED | 保持最后值 | 转写失败 |
| CANCELED | 保持最后值 | 用户取消 |

> **两层进度策略**（采纳自报告④）：阶段性的"状态进度"给出可解释的区间，TRANSCRIBING 阶段内按 `elapsed / (duration × rtf_p90)` 做时间推进，ASR 返回最终结果后校正为 100%。

### 7.6 容错与恢复机制（采纳自报告②，简化版）

#### 断路器模式（Circuit Breaker）

为每个 ASR 服务器维护断路器状态：

| 状态 | 行为 | 转换条件 |
|------|------|----------|
| **CLOSED**（正常） | 正常派单 | 连续失败 ≥ 5 次 → OPEN |
| **OPEN**（熔断） | 拒绝派单 | 等待 60s → HALF_OPEN |
| **HALF_OPEN**（试探） | 允许 3 次试探 | 3 次全成功 → CLOSED，否则 → OPEN |

#### 重试策略

```python
# 指数退避 + 抖动
delay = min(base_delay * (2 ** retry_count), max_delay) + random_jitter
# 默认: base=2s, max=60s, jitter=±25%
```

- 单个任务最多重试 3 次
- 重试时优先选择**不同的** ASR 服务器（避免重复命中故障节点）
- 重试次数耗尽后标记为 `failed`，保留错误信息供人工排查

#### 幂等保障

- 同一 `task_id` 重复提交到 ASR 服务器时，适配层应检查是否已有结果
- 通过 Redis 键 `idempotent:{task_id}` 做分布式锁，避免并发重复处理

#### 告警规则（具体化，含 PromQL 模板）

| 告警名 | PromQL 表达式 | 持续时间 | 严重度 | 处置建议 |
|--------|--------------|----------|--------|----------|
| **QueueBacklog** | `asr_queue_depth > 100` | 5min | Warning | 检查 Worker 是否存活；考虑扩容 Worker 或 ASR 节点 |
| **QueueCritical** | `asr_queue_depth > 500` | 2min | Critical | 立即扩容或暂停接收新任务 |
| **HighFailureRate** | `rate(asr_tasks_total{status="FAILED"}[5m]) / rate(asr_tasks_total[5m]) > 0.1` | 3min | Critical | 检查 ASR 节点健康；查看 task_events 最近失败原因 |
| **ServerOffline** | `time() - asr_server_last_heartbeat > 60` | 即时 | Critical | 断路器应已触发；排查节点网络/进程状态 |
| **SlowProcessing** | `histogram_quantile(0.95, asr_task_duration_seconds) > 3 * avg_over_time(histogram_quantile(0.95, asr_task_duration_seconds)[24h])` | 10min | Warning | RTF 可能漂移；触发 benchmark 重测 |
| **HighRetryRate** | `rate(asr_task_retries_total[10m]) > 5` | 5min | Warning | 某节点可能不稳定；检查断路器状态 |
| **DiskUsageHigh** | `node_filesystem_avail_bytes{mountpoint="/data"} / node_filesystem_size_bytes < 0.15` | 5min | Warning | 触发文件清理；检查归档策略 |

> Grafana 面板建议配置 4 个 Panel Row：①总览（任务量/成功率/队列深度）②服务器（各节点 RTF/槽位/断路器）③延迟（P50/P95/P99 耗时分布）④系统（磁盘/内存/CPU）。

```json
  "result_ref": "/api/v1/tasks/01J...T1/result",
  "timestamp": "2026-02-26T12:34:56Z",
  "signature": "HMAC-SHA256(secret, payload)"
}
```

> 回调签名使用 HMAC-SHA256，接收方以 `event_id` 做幂等去重。投递语义为"至少一次（at-least-once）"。

## 八、关键技术决策与权衡

### 8.1 为什么不直接用 Celery 而推荐 Dramatiq？

| 对比维度 | Celery | Dramatiq |
|----------|--------|----------|
| 20k 任务吞吐 | 11.68s (threads) | 4.12s |
| API 复杂度 | 高（大量配置项） | 低（合理默认值） |
| 重试机制 | 需手动配置 | 内置指数退避 |
| 内存效率 | 一般 | 优秀 |
| 代码热重载 | 不支持 | 内置支持 |
| 社区生态 | 最大 | 中等但活跃 |
| Broker 支持 | Redis/RMQ/SQS | Redis/RMQ |

本项目是 **I/O 密集型**（等待 ASR 服务器返回），任务本身不重（不做 GPU 推理），Dramatiq 的高性能和简洁 API 更匹配。Celery 的企业级特性（如 SQS 支持、Canvas 复杂编排）在本项目中暂时用不到。

### 8.2 为什么选 SSE 而不是 WebSocket 做进度推送？

- 进度推送是**单向**的：服务端 → 客户端
- SSE 基于标准 HTTP，无需升级协议，兼容性更好
- 浏览器 `EventSource` API 内置自动重连
- FastAPI 通过 `StreamingResponse` 可直接实现 SSE
- fun-asr-docker 项目已在生产中验证了 SSE 进度推送的可行性

### 8.3 为什么初期不引入 Airflow/Prefect/Ray 等编排工具？

- 本项目的任务流水线很**短**：上传 → 预处理 → 转写 → 返回结果，不是多阶段复杂 DAG
- 初期 ASR 服务器数量有限（几台到十几台），自建调度器足够
- 引入编排工具会增加部署复杂度和运维成本
- 如果后期需要加入 NLP 摘要、翻译等多阶段后处理，再考虑引入工作流引擎

### 8.4 数据库 SQLite → PostgreSQL 的渐进策略

**设计保障**：使用 SQLAlchemy ORM，数据库无关的模型定义；避免使用 SQLite 不支持的特性（数组、JSONB 用 TEXT 替代）。

**切换触发条件**（采纳报告③的量化阈值）：

| 指标 | 阈值 | 监测方式 |
|------|------|----------|
| 写冲突率 | > 5%（SQLite BUSY 错误比例） | structlog 统计 |
| 队列长度 | 持续 > 500 | Prometheus `asr_queue_depth` |
| 日任务量 | > 2000 | 数据库 COUNT 统计 |
| 多实例部署需求 | 需要 2+ API 实例共享数据 | 架构决策 |

**迁移操作流程**（`scripts/migrate_to_pg.py`）：

```
迁移步骤（计划停机窗口 30 分钟）:
1. 预检（迁移前 1 天）:
   - 确认 PostgreSQL 实例已部署并可访问
   - 确认 asyncpg 驱动已安装
   - 在 PG 上执行 alembic upgrade head 创建空表结构
   - 用测试数据验证 ORM 在 PG 上正常工作

2. 数据迁移:
   - 停止接收新任务（API 返回 503 + Retry-After）
   - 等待所有进行中任务完成（最长等待 30min，超时则强制保存状态）
   - 执行 scripts/migrate_to_pg.py:
     a. 连接 SQLite，逐表读取全量数据
     b. 批量 INSERT 到 PostgreSQL（每批 1000 条）
     c. 校验行数一致 + 抽样校验关键字段
   - 更新 config/settings.yaml: database_url 指向 PG

3. 验证:
   - 启动应用，执行端到端冒烟测试（上传→转写→查询）
   - 检查 /health 和 /metrics 正常
   - 确认日志无 ORM 报错

4. 回滚方案:
   - 保留 SQLite 文件不删除（至少保留 7 天）
   - 若 PG 出问题，改回 database_url 指向 SQLite 即可立即回退
   - 回退期间新产生的 PG 数据需手动同步回 SQLite（或接受丢失）
```

## 十一、风险与注意事项

### 11.1 技术风险

| 风险 | 影响 | 缓解措施 |
|------|------|----------|
| FunASR 新旧协议差异未充分文档化 | 适配器开发受阻 | 提前阅读 FunASR 源码，搭建新旧版本测试环境；**复用 funasr-client-python 已验证的逻辑** |
| ASR 服务器宕机/假死 | 任务卡死 | 心跳检测 + 超时机制 + 断路器 + 自动重试/重新分配 |
| 大文件上传导致内存溢出 | 服务崩溃 | 流式上传（分块读取），限制单文件大小；后期可引入 tus 续传协议 |
| SQLite 并发写入瓶颈 | 任务状态更新延迟 | WAL 模式 + 明确切换阈值（写冲突率 > 5%）+ 预留 PostgreSQL 迁移路径 |
| Worker 异常退出导致任务丢失 | 用户数据丢失 | Dramatiq 内置消息确认机制（ack on success） |
| **下游 TaskId 不唯一**（采纳自报告④） | 任务关联错乱 | 内部 ULID 为主键，`external_task_id` 仅做映射；不依赖下游 ID 的唯一性和生命周期 |
| **下游结果 URL 过期**（采纳自报告④） | 转写结果丢失 | 任务完成后立即拉取并持久化到本地存储，不依赖短期 URL |
| **回调投递失败导致客户端不知道结果** | 业务流程中断 | Outbox 模式 + 至少一次投递 + 最多 5 次重试 + 人工重试入口 |

### 11.2 设计原则建议

1. **关注点分离**：Web 层、调度层、协议适配层各司其职，通过接口解耦
2. **渐进式复杂度**：初期用最简架构跑通全链路，再逐步加入高级特性（报告③的核心理念）
3. **可观测性优先**：从第一天起就加入结构化日志、关键指标（队列深度、平均RTF、服务器利用率）
4. **幂等设计**：任务重试不会导致重复转写或数据不一致；回调投递以 event_id 去重
5. **配置外化**：ASR 服务器列表、调度参数等全部配置化，不硬编码
6. **内部 ID 自治**（采纳自报告④）：不依赖下游 ASR 的 TaskId 作为业务唯一 ID，内部 ULID 为准
7. **控制平面与数据平面解耦**（采纳自报告④）：管理器只管调度和状态，音频数据走独立的存储/传输通道
8. **正确性优先于性能**（采纳自报告④）：先保证可恢复、可追踪、可审计，再优化吞吐和延迟

### 11.3 故障演练操作手册（Runbook）

> 建议在 M3 阶段完成后执行一轮完整故障演练，验证容错机制有效性。

#### 场景 1：ASR 节点宕机

| 步骤 | 操作 | 预期结果 |
|------|------|----------|
| 1 | 手动停止一台 ASR 节点进程 | 30s 内心跳超时，节点状态变为 OFFLINE |
| 2 | 观察该节点上正在执行的任务 | 断路器 → OPEN；任务触发重试，分配到其他健康节点 |
| 3 | 提交新任务 | 调度器跳过 OFFLINE 节点，不分配 |
| 4 | 恢复节点 | 60s 后断路器 → HALF_OPEN → 试探成功 → CLOSED；新任务可分配 |
| 5 | 检查最终结果 | 所有任务最终 SUCCEEDED 或 FAILED（重试 3 次后），无任务永久卡死 |

#### 场景 2：队列积压

| 步骤 | 操作 | 预期结果 |
|------|------|----------|
| 1 | 短时间内批量提交 200+ 个任务 | 队列深度升高，Grafana 可见 |
| 2 | 观察告警 | QueueBacklog Warning 在 5min 内触发 |
| 3 | 观察调度 | 任务按 LPT 有序分配，不乱序 |
| 4 | 检查 ETA | 前端显示的 ETA 随队列消化逐步减少 |
| 5 | 全部完成后 | 队列深度归零，告警自动恢复 |

#### 场景 3：Worker 进程崩溃

| 步骤 | 操作 | 预期结果 |
|------|------|----------|
| 1 | kill -9 一个 Dramatiq Worker 进程 | 进程异常退出 |
| 2 | 观察任务状态 | 未 ack 的任务自动重新入队（Dramatiq 保障） |
| 3 | Docker 自动重启 Worker | 新 Worker 取回未完成任务继续执行 |
| 4 | 检查结果 | 任务最终完成，无数据丢失 |

#### 场景 4：Redis 短暂不可用

| 步骤 | 操作 | 预期结果 |
|------|------|----------|
| 1 | 暂停 Redis 容器 30s | API 层和 Worker 出现连接错误 |
| 2 | 恢复 Redis | 连接自动重建（redis-py 内置重连） |
| 3 | 检查任务状态 | 队列中的任务不丢失（Redis AOF 持久化）；进行中任务状态从 DB 恢复 |

#### 管道（Pipeline）与脚本化能力

CLI 的 `--quiet` + `--output json` 模式专为脚本/Agent 设计：

```bash
# 管道：上传 → 提取 file_id → 创建任务 → 等待 → 下载
FILE_ID=$(asr-cli upload meeting.wav -q -o json | jq -r '.file_id')
TASK_ID=$(asr-cli task create $FILE_ID -q -o json | jq -r '.[0].task_id')
asr-cli task wait $TASK_ID
asr-cli task result $TASK_ID --format srt --save meeting.srt

# 批量处理脚本
for f in /data/audio/*.wav; do
  asr-cli transcribe "$f" --format txt --output-dir /data/results/ -q
done

# 监控脚本：检查队列深度
DEPTH=$(asr-cli stats -q -o json | jq '.queue_depth')
if [ "$DEPTH" -gt 100 ]; then
  echo "ALERT: Queue depth $DEPTH exceeds threshold"
fi
```

#### Skills 集成接口规范

CLI 的 `--quiet --output json` 模式是 AI Agent Skills 的标准调用接口：

```python
# Skills 调用示例（伪代码）
import subprocess, json

def transcribe_audio(file_path: str, language: str = "zh") -> dict:
    """Agent Skill: 调用 ASR CLI 完成音频转写"""
    result = subprocess.run(
        ["asr-cli", "transcribe", file_path,
         "--language", language, "--format", "json",
         "-q", "-o", "json"],
        capture_output=True, text=True
    )
    return json.loads(result.stdout)
```

Agent 可调用的核心 Skill 动作：

| Skill 动作 | CLI 命令 | 输入 | 输出 |
|-----------|---------|------|------|
| 转写音频 | `asr-cli transcribe <file> -q -o json` | 音频文件路径 | JSON 结果 |
| 查询任务状态 | `asr-cli task info <id> -q -o json` | task_id | 任务详情 JSON |
| 批量转写 | `asr-cli transcribe *.wav -q -o json` | 文件列表 | 批量结果 JSON |
| 系统健康 | `asr-cli health -q -o json` | 无 | 健康状态 JSON |
| 节点状态 | `asr-cli server list -q -o json` | 无 | 节点列表 JSON |
