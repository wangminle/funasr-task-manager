# 速度模型修正分析：从 throughput_rtf 到 N/rtf_baseline

**日期**：2026-04-14
**触发**：运筹学分析揭示配额分配与生产最优分布严重偏离

---

## 1. 问题发现

从 12 视频批量转写测试的生产数据中观察到：

| 服务器 | slots | per-slot RTF | throughput_rtf | 1/tp_rtf (旧speed) | N/rtf (新speed) |
|--------|-------|-------------|----------------|---------------------|-----------------|
| 10095  | 2     | 0.063       | 0.0031         | 322.6               | 31.7            |
| 10096  | 4     | 0.083       | 0.0185         | 54.1                | 48.2            |
| 10097  | 1     | 0.078       | 0.0061         | 163.9               | 12.8            |

**关键偏差**：`throughput_rtf` 是用 **6 秒的 test.mp4** 测出来的，而生产任务是 **4.5~23 分钟**的长音频。两者的 RTF 特征偏差高达 **10~13 倍**（10095、10097），导致：

- 旧速度模型严重高估 10095/10097 的相对速度
- 4-slot 的 10096 被错误标记为"最慢"，只分配 1 个任务

## 2. 配额分配对比

12 个任务的配额分配（旧 vs 新 vs 实测最优）：

| 模型 | 10095 (2 slots) | 10096 (4 slots) | 10097 (1 slot) | 预估 makespan |
|------|-----------------|-----------------|----------------|--------------|
| 旧: 1/throughput_rtf | **7** | **1** | 4 | ~290s |
| 新: N/rtf_baseline | **4** | **6** | 2 | ~145s |
| 实测最优分布 | 3 | 8 | 1 | ~120s |

旧模型给 4-slot 的 10096 只分了 1 个任务，3 个 slot 闲置；新模型的分布接近实测最优。

## 3. 为什么 1/throughput_rtf 公式"数学正确但数据有毒"

`throughput_rtf = wall_clock / (audio_duration × N_concurrent)`

分母已含并发，所以 `1/throughput_rtf` = 整机总吞吐。**公式逻辑没错**。

问题在于：
- Benchmark Phase 2 用 6s 的 test.mp4 做并发测试
- 短样本的 RTF 和长音频的 RTF 差异非线性（可能因编解码预热、VAD 启动开销等）
- 10095 的 tp_rtf=0.0031，意味着 6s 音频并发×2 只需 0.037 秒处理完
- 但实际处理 1400s 音频时 per-slot RTF 是 0.063，完全不在一个量级

## 4. 修正方案

```python
# 旧公式
def get_throughput_speed(self, server):
    return 1.0 / server.throughput_rtf  # 被短样本污染

# 新公式
def get_throughput_speed(self, server):
    base_rtf = self.get_base_rtf(server)  # P90 RTF from production or rtf_baseline
    return server.max_concurrency / base_rtf
```

**优势**：
- `base_rtf` 来自 Benchmark Phase 1（3 分钟 tv-report-1.wav），更接近生产负载
- 运行一段时间后，`rtf_tracker.get_p90()` 用真实任务数据替代 benchmark，持续自我校准
- `max_concurrency` 显式参与，正确反映 "多 slot 多吞吐" 的物理事实

## 5. Fluid Lower Bound（运筹学验证）

系统总吞吐 = 31.7 + 48.2 + 12.8 = **92.7 audio-s/wall-s**
总音频 = 10365s
理论下界 = 10365 / 92.7 = **111.7s**

实测 120s，达到理论下界的 **93%**，说明原算法 (LPT+EFT) 本身优秀，瓶颈确实在速度模型输入。
