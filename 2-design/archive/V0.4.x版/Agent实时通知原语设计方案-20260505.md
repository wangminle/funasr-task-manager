> 日期：2026-05-05  
> 状态：待实施  
> 优先级：P0（直接影响用户体验）

## 1. 问题描述

### 现象

用户在飞书群内发送音频文件后，Agent 处理过程中产生的 8 条中间状态消息（下载中、预检查、提交任务等）全部在转写完成时（约 4 分钟后）一股脑送达。用户在等待期间完全看不到任何进度反馈，造成焦虑。

批量转写场景同理：用户 14:14 发出指令，Agent 执行了多轮 webos/工具调用，但所有中间通知在 14:25 才集中到达。

### 根因分析

```
用户发消息 → OpenClaw 触发 Agent → Agent 执行 Skill（多轮工具调用）
                                    ├── 工具调用 1: 下载文件
                                    │   ├── Agent 生成文本"正在下载..."  ← 缓冲在响应流中
                                    ├── 工具调用 2: ffprobe 预检
                                    │   ├── Agent 生成文本"预检通过"    ← 缓冲在响应流中
                                    ├── 工具调用 3: 上传+提交
                                    │   ├── Agent 生成文本"已提交"     ← 缓冲在响应流中
                                    └── Turn 结束
                                         └── 平台将缓冲文本统一推送到飞书 ← 所有消息同时到达
```

**核心矛盾**：Skills 要求 Agent "通知用户"，Agent 将其理解为"在响应流中输出文本"。但 OpenClaw 等平台的行为是 **turn 级别的消息投递**——只有当 Agent 整个 turn 结束后，才把响应流中的文本推送到飞书。普通 assistant 文本输出 ≠ 实时飞书消息。

## 2. 解决方案概述

### 方案结论

本方案是当前阶段的推荐方案，但实现优先级需要明确调整：

1. **首选：调用平台原生实时消息工具**。在 OpenClaw/asr-agent 环境中，优先使用 runtime 暴露的 `message` tool。它已经持有当前飞书会话、线程、文件上传和发送能力，且 session 记录证明它能直接返回 `messageId/chatId`。
2. **兜底：调用 `python -m cli notify send`**。当运行环境没有暴露 `message` tool，或 message tool 不支持当前渠道/文本场景时，再用 CLI 直接调用飞书 Open API。
3. **不把普通 assistant 文本当作通知**。普通文本仍可作为最终总结，但不能承担实时进度同步职责。

理由是：问题发生在 Agent 执行 Skill 的 turn 内，普通 assistant 文本被 OpenClaw/Hermes 等聊天平台缓冲；而后端 `task_runner`、SSE、callback 只能覆盖任务创建后的转写状态，不能覆盖 Agent 在任务创建前执行的文件下载、预检查、扫描目录、manifest 生成等阶段。因此首选修复点应在 **Agent 可调用的外部副作用工具** 上，而不是先改后端事件系统。

备选方案对比：

| 方案 | 能解决下载/预检查阶段吗 | 能绕过 turn 缓冲吗 | 改动面 | 结论 |
|------|------------------------|-------------------|--------|------|
| 仅强化 Skill 文案 | 否 | 否 | 小 | 不足，仍可能被缓冲 |
| 扩展后端 callback/outbox | 否（任务创建前无后端任务） | 是 | 大 | 中期增强，不作为首修 |
| 调用 OpenClaw `message` tool | 是 | 是 | 小 | 首选；已在 asr-agent session 中验证可用 |
| Agent 调用飞书 curl | 是 | 是 | 中 | 可行但分散，难统一凭据与重试 |
| 新增 `notify` CLI 原语 | 是 | 是 | 中 | 兜底；用于没有平台 message tool 的环境 |

### 核心思路

将"通知用户"从文案要求升级为**强制通知原语** `send_user_notice()`。在 OpenClaw + 飞书环境下，该原语的首选实现是 **调用 OpenClaw runtime 暴露的 `message` tool**；没有可用 `message` tool 时，才 fallback 到 `python -m cli notify send` 直接调用飞书 IM API。两者都绕过普通 assistant 文本的 turn 缓冲。

### 架构对比

**修改前（被缓冲）：**

```
Agent 文本输出 → 平台响应流缓冲 → Turn 结束后推送飞书 → 用户集中收到
```

**修改后（实时送达）：**

```
Agent 调用 `send_user_notice()` → 优先 OpenClaw message tool / fallback CLI notify → 用户立即收到
```

### 设计原则

1. **先 notify，再做耗时操作**：每次耗时工具调用之前，必须先发送通知告知用户即将做什么
2. **优先复用平台 message tool**：OpenClaw/asr-agent 中已有 `message` tool，Skill 应优先调用它，而不是重新实现飞书发送。
3. **fallback 到 CLI notify**：仅在平台没有可调用 message tool 时，使用 `python -m cli notify send`。
4. **默认 soft-fail**：通知发送失败不阻塞主流程（message tool 返回失败或 CLI exit 0 + stderr warning）
5. **多平台适配**：飞书为首要实现，预留企业微信/Slack 扩展接口
6. **回复线程支持**：平台 message tool 优先使用当前 session/thread；CLI fallback 支持 `--reply-to` 参数实现消息线程关联
7. **上下文显式传递**：`chat_id` / `reply_to` 必须来自当前飞书事件，由 webhook/Agent 运行环境注入环境变量或配置，不能靠 CLI 猜测
8. **避免 shell 转义风险**：CLI fallback 除 `--text` 外必须支持 `--text-file` 或 stdin，供 Agent 发送多行通知和包含引号的内容

## 3. 详细设计

### 3.1 `send_user_notice()` 适配器选择

`send_user_notice()` 是 Skill 层唯一允许使用的用户可见通知原语。它按以下顺序选择实现：

```text
send_user_notice(text, optional file_path):
  1. 如果当前 runtime 暴露 OpenClaw/Hermes message tool：
       调用 message tool，要求 action=send，message=text，可选 filePath=file_path
  2. 否则如果 CLI notify 可用：
       调用 python -m cli notify send / send-file
  3. 否则：
       在普通 assistant 文本中说明通知通道不可用，并继续主流程
```

#### 3.1.1 OpenClaw `message` tool 首选实现

根据 asr-agent session 记录，OpenClaw runtime 暴露的 `message` tool 形态如下：

```json
{
  "name": "message",
  "arguments": {
    "action": "send",
    "message": "✅ 转写完成..."
  }
}
```

发送文件附件时可附带 `filePath`：

```json
{
  "name": "message",
  "arguments": {
    "action": "send",
    "filePath": "/tmp/funasr-task-manager/results/20240510_160113_result.txt",
    "message": "✅ 转写完成..."
  }
}
```

成功结果示例：

```json
{
  "ok": true,
  "channel": "feishu",
  "action": "send",
  "messageId": "...",
  "chatId": "..."
}
```

要求：

- Skill 不直接输出“正在下载/正在预检/已提交”等普通文本作为实时通知；必须先调用 `message` tool。
- 每次调用后检查 toolResult 的 `ok == true`。失败时记录到最终报告，但默认不阻塞转写主流程。
- 如果 runtime 暴露 `message` tool，优先使用它发送文本和文件；不要再通过 CLI 重复发送，避免用户收到双份消息。
- 普通 assistant 文本可以保留为最终总结，但不计入“实时通知已发送”。

#### 3.1.2 CLI notify fallback

仅当以下条件之一成立时使用 CLI fallback：

- runtime 没有可调用 `message` tool。
- `message` tool 不支持当前渠道或当前 action。
- 需要在非 OpenClaw 环境中运行相同 Skill。
- `message` tool 连续失败，且用户/部署方明确允许使用飞书 API 凭据兜底。

### 3.2 `notify` CLI 子命令（fallback）

#### 3.2.1 命令接口

```bash
# 发送文本消息
python -m cli notify send \
  --text "⏳ 正在从飞书下载文件..." \
  [--text-file <path>] \
  [--chat-id <chat_id>] \
  [--reply-to <message_id>] \
  [--channel feishu] \
  [--strict] \
  [--retry-unknown]

# 发送多行/复杂文本，避免 shell 转义问题
python -m cli notify send \
  --text-file /tmp/asr-notice.txt \
  [--chat-id <chat_id>] \
  [--reply-to <message_id>]

# 从 stdin 读取文本
python -m cli notify send --stdin [--chat-id <chat_id>]

# 发送文件附件（Phase 2 实现）
python -m cli notify send-file \
  --file /tmp/funasr-task-manager/result.txt \
  --filename "会议录音.txt" \
  [--chat-id <chat_id>] \
  [--channel feishu] \
  [--strict]

# 验证凭据可用性
python -m cli notify auth-check \
  [--channel feishu]
```

#### 3.2.2 参数说明

| 参数 | 必需 | 说明 |
|------|------|------|
| `--text` | 三选一 | 消息内容，支持中文 |
| `--text-file` | 三选一 | 从文件读取消息内容，优先用于多行文本和复杂引号内容 |
| `--stdin` | 三选一 | 从标准输入读取消息内容 |
| `--chat-id` | 否 | 目标会话 ID，缺省从配置/环境变量读取 |
| `--reply-to` | 否 | 回复的原消息 ID（实现飞书线程回复） |
| `--channel` | 否 | 渠道类型，默认 `feishu`，预留 `wecom`/`slack` |
| `--strict` | 否 | 严格模式：失败时 exit 1（用于 auth-check / 测试） |
| `--retry-unknown` | 否 | HTTP 超时等无法确认是否送达时仍重试；默认关闭以避免重复消息 |
| `--file` | send-file 时必需 | 本地文件路径 |
| `--filename` | 否 | 飞书上显示的文件名（默认取路径中的文件名） |

#### 3.2.3 退出码策略

`send` 默认使用 soft-fail，避免通知系统故障阻断 ASR 主流程。`auth-check` 默认等价于 strict，因为它用于初始化和部署检查，必须明确暴露配置问题。

| 场景 | 默认模式（soft-fail） | `--strict` 模式 |
|------|---------------------|----------------|
| 发送成功 | exit 0 + stdout 输出 message_id | exit 0 |
| 发送失败（重试 1 次后） | exit 0 + stderr warning | exit 1 |
| 凭据缺失 | exit 0 + stderr "凭据未配置" | exit 1 |
| 网络不可达 | exit 0 + stderr "网络错误" | exit 1 |
| `auth-check` 失败 | exit 1 | exit 1 |

#### 3.2.4 凭据管理

**配置来源优先级**：CLI 参数 > 环境变量 > `~/.asr-cli.yaml`

**环境变量**：

```bash
FEISHU_APP_ID=cli_xxx
FEISHU_APP_SECRET=xxx
FEISHU_CHAT_ID=oc_xxx         # 默认目标会话
FEISHU_REPLY_TO=om_xxx        # 可选，默认回复的消息 ID
```

**运行环境注入要求（仅 CLI fallback 必需）**：

OpenClaw/Hermes 等聊天平台的 webhook 接收到飞书事件后，必须把当前会话上下文注入到 Agent 可见环境中：

- `FEISHU_CHAT_ID`：当前群聊或私聊会话 ID
- `FEISHU_REPLY_TO`：用户原始消息 ID，用于将通知回复到原话题/线程

如果这两个值都不可用，`notify send` 不能猜测目标会话，只能 soft-fail 并在 stderr 中提示"缺少 chat_id"。OpenClaw `message` tool 通常已从当前 session 持有会话上下文；只有走 CLI fallback 时才必须显式注入这些环境变量。

**配置文件扩展** (`~/.asr-cli.yaml`)：

```yaml
server: "http://localhost:15797"
api_key: ""
output: "table"

# 新增 notify 配置段
notify:
  channel: feishu
  feishu_app_id: "cli_xxx"
  feishu_app_secret: "xxx"
  default_chat_id: "oc_xxx"
  default_reply_to: ""          # 可选
```

现有 `cli/config_store.py` 只支持顶层键读取，实施时需要补充嵌套配置读取能力，或约定用点分键兼容旧命令：

```bash
python -m cli config set notify.feishu_app_id cli_xxx
python -m cli config set notify.feishu_app_secret xxx
python -m cli config set notify.default_chat_id oc_xxx
```

推荐实现：`config_store.get("notify.feishu_app_id")` 同时兼容嵌套 YAML 和点分键，避免破坏现有 `server/api_key/output` 配置。

#### 3.2.5 Token 缓存

缓存文件：`~/.asr-cli-feishu-token.json`

```json
{
  "tenant_access_token": "t-g10xxx",
  "expires_at": 1746460800,
  "app_id": "cli_xxx"
}
```

- 有效期设为获取时刻 + 110 分钟（飞书 token 2 小时有效，留 10 分钟余量）
- app_id 变更时自动刷新
- 文件不存在/过期/解析失败时自动重新获取

#### 3.2.6 飞书 API 调用逻辑

```python
# 获取 token
POST https://open.feishu.cn/open-apis/auth/v3/tenant_access_token/internal
Body: {"app_id": "...", "app_secret": "..."}

# 发送文本消息
POST https://open.feishu.cn/open-apis/im/v1/messages?receive_id_type=chat_id
Headers: {"Authorization": "Bearer {token}", "Content-Type": "application/json"}
Body: {
  "receive_id": "{chat_id}",
  "msg_type": "text",
  "content": "{\"text\": \"{text}\"}"
}

# 回复消息（有 reply_to 时）
POST https://open.feishu.cn/open-apis/im/v1/messages/{reply_to}/reply
Headers: {"Authorization": "Bearer {token}", "Content-Type": "application/json"}
Body: {
  "msg_type": "text",
  "content": "{\"text\": \"{text}\"}"
}

# 发送文件（Phase 2）
# Step 1: 上传
POST https://open.feishu.cn/open-apis/im/v1/files
Headers: {"Authorization": "Bearer {token}"}
Body: multipart/form-data (file_type=stream, file_name=..., file=@...)

# Step 2: 发送
POST https://open.feishu.cn/open-apis/im/v1/messages?receive_id_type=chat_id
Body: {"receive_id": "...", "msg_type": "file", "content": "{\"file_key\": \"...\"}"}
```

响应处理要求：

- HTTP 2xx 不等于业务成功，必须解析飞书响应 JSON 中的 `code` 字段。
- 只有 `code == 0` 才视为成功，并从响应中提取 `message_id`。
- `code != 0` 时将 `code`、`msg` 写入 stderr；soft-fail 模式仍 exit 0，strict 模式 exit 1。
- 不在日志中写入 `app_secret`、tenant token 或完整通知正文；失败日志最多保留通知正文前 80 个字符或正文 hash。

#### 3.2.7 重试策略

- 首次失败后等待 1 秒重试 1 次
- 重试仍失败：soft-fail 模式下打印 stderr 警告并 exit 0
- 记录失败通知到 `~/.asr-cli-notify-failures.log`（追加模式，保留最近 100 条）
- 为避免偶发重复消息，重试只针对明确未送达的场景；如果 HTTP 请求超时但无法确认飞书是否已接收，默认不重试，记录为 `unknown_delivery`，除非用户使用 `--strict --retry-unknown`

### 3.3 共享通知规范文件

新增 `6-skills/_shared/CHANNEL-NOTIFICATION.md`，定义强制通知原语：

```markdown
# 渠道实时通知规范

## 强制规则

1. 所有用户可见状态同步必须通过 send_user_notice() 实现。
2. 禁止把阶段通知仅写入普通 assistant 文本回复。
3. 禁止在长耗时工具调用前不发送状态消息。
4. 每次 send_user_notice() 必须发生在下一次耗时操作之前。
5. send_user_notice() 等待返回后再继续下一步工具调用。

## send_user_notice() 实现

### OpenClaw / 飞书环境（首选）

调用 runtime 暴露的 `message` tool：

  Tool: message
  arguments: {"action": "send", "message": "{通知内容}"}

如果需要发送文件：

  Tool: message
  arguments: {"action": "send", "message": "{摘要通知}", "filePath": "{本地文件路径}"}

### CLI fallback

仅当 runtime 没有 `message` tool 或 message tool 不可用时，调用 CLI notify 命令：

  python -m cli notify send --text "{通知内容}"

### 其他环境

如果当前运行环境明确不是聊天渠道（例如纯本地终端），可使用当前平台的普通用户回复能力输出。只要存在 OpenClaw、Hermes、飞书、企业微信、Slack 等聊天平台桥接，就必须优先使用 notify 原语，不能假设普通 assistant 文本会实时送达。

## 失败处理

- 默认 soft-fail：发送失败不阻塞主流程
- 失败的通知在最终报告中注明"以下 N 条通知未送达"
- 关键通知（任务完成/失败）建议使用 `--strict`；如果 strict 失败，应在普通响应中明确说明通知通道异常
```

## 6. 风险与缓解

| 风险 | 影响 | 缓解措施 |
|------|------|---------|
| Skill 继续输出普通文本而不调用 message tool | 仍然被 turn 缓冲，问题不解决 | 执行检查清单必须以 toolResult `ok == true` 作为“通知已发送”的判断依据 |
| OpenClaw message tool 名称或参数在不同版本变化 | Skill 示例调用失败 | 在 `realtime-notify.md` 中记录 runtime 探测方式；失败时 fallback CLI |
| 同时调用 message tool 和 CLI notify | 用户收到重复通知 | 适配器必须单选；message tool 成功后禁止 fallback |
| Python 解释器启动开销（~0.5-1s/次） | 每次 notify 多约 1s | 可接受；未来可优化为 daemon 模式 |
| 飞书 API 限流（单 App 50 QPS） | 批量转写高频通知可能触限流 | 批量场景限制为每 30s 发一次，不逐文件发 |
| 凭据未配置时 Agent 不知道失败 | soft-fail 可能导致用户仍收不到消息 | 在 init Skill Phase 7 增加 `notify auth-check --strict` 验证步骤 |
| webhook 未注入 chat_id/reply_to | notify 不知道发到哪里 | OpenClaw/Hermes 运行环境必须注入 `FEISHU_CHAT_ID`，并尽量注入 `FEISHU_REPLY_TO` |
| 消息不在原话题线程内 | 用户体验割裂 | 强烈建议配置 `--reply-to` 或 `default_reply_to` |
| shell 引号/多行文本转义失败 | notify 命令执行失败或内容变形 | 支持 `--text-file` / `--stdin`，Skill 示例优先使用这两种方式发送复杂文本 |
| 多 Agent 并发时消息顺序 | 消息到达顺序可能乱 | 通知内容自带 Phase 标识，用户可辨识 |

## 9. 验收标准

1. 用户发送音频文件后，**30 秒内**收到第一条飞书通知（"收到文件"/"开始下载"）
2. 处理过程中每个阶段转换时，用户在 **5 秒内**收到对应状态通知
3. OpenClaw/asr-agent session 记录中能看到显式 `message` tool 调用，而不是只有普通 assistant 文本
4. `message` tool 成功时不得再调用 CLI fallback，避免重复发送
5. 没有平台 message tool 时，CLI fallback 通知凭据未配置不应中断 Agent 主流程，仅 stderr 输出警告
6. `notify auth-check --strict` 可作为 init 流程的前置检查点
7. 所有通知消息出现在正确的飞书话题线程内（message tool 当前会话或配置 reply-to 时）

## 10. 附录：Agent 调用模式对照

### 修改前（延迟送达）

```
Agent: [思考] 用户发了音频，我来处理
Agent: [文本输出] "收到 1 个音频文件"          ← 缓冲
Agent: [工具调用] curl 获取 token              ← 执行 5s
Agent: [文本输出] "正在下载文件..."            ← 缓冲
Agent: [工具调用] curl 下载文件                ← 执行 30s
Agent: [文本输出] "下载完成"                   ← 缓冲
Agent: [工具调用] ffprobe + 上传 + 创建任务     ← 执行 20s
Agent: [文本输出] "已提交任务"                 ← 缓冲
... 轮询转写 ...
Agent: [文本输出] "转写完成"                   ← 缓冲
Agent: [Turn 结束]                            ← 平台把所有文本推到飞书，用户一次收到 8 条
```

### 修改后（实时送达）

```
Agent: [思考] 用户发了音频，我来处理
Agent: [工具调用] message(action=send, message="收到 1 个音频文件")  ← 飞书立即收到
Agent: [工具调用] message(action=send, message="⏳ 正在获取 token 并下载文件...")  ← 飞书立即收到
Agent: [工具调用] curl 获取 token + 下载文件    ← 执行 35s
Agent: [工具调用] message(action=send, message="✅ 下载完成（5.9MB）")  ← 飞书立即收到
Agent: [工具调用] ffprobe + 上传 + 创建任务     ← 执行 20s
Agent: [工具调用] message(action=send, message="✅ 已提交任务，预计 3 分钟完成")  ← 飞书立即收到
... 轮询转写 ...
Agent: [工具调用] message(action=send, message="✅ 转写完成！", filePath="result.txt")  ← 飞书立即收到文件
```

关键区别：**`message` tool / CLI notify 都是一次工具调用**，工具调用的副作用立即生效，不受平台响应流缓冲影响。OpenClaw/asr-agent 中优先使用 `message` tool；CLI notify 仅作为 fallback。
