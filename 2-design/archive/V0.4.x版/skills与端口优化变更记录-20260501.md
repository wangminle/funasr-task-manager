## 二、飞书文件下载：明确禁用 drive API

### 问题来源

测试报告 §2.2 记录：Agent 首次尝试 `drive/v1/files/{file_key}/download` 接口（返回 403 / 错误码 99991672），探索约 2 分钟后才切换到 `im/v1/messages` 接口成功下载。

### 根因

Skill 文档写了正确的 `im` 接口，但没有明确告诉 Agent "drive API 不能用于对话文件"，Agent 会自行探索更"通用"的 API 路径。

### 修改

在以下两个文件的飞书下载部分添加醒目警告：

- `6-skills/funasr-task-manager-channel-intake/SKILL.md`（Phase 1.5 Step 3）
- `6-skills/funasr-task-manager-channel-intake/references/channel-file-apis.md`（飞书 → 下载消息中的文件）

警告内容：

> ⚠️ 不要使用 drive API（`/open-apis/drive/v1/files/{file_key}/download`）——那是云文档/网盘接口，需要 `drive:file:readonly` 权限，对聊天中发送的文件附件返回 403（错误码 99991672）。对话文件**必须**用 `im/v1/messages` 接口。

## 四、后端服务改为用户级 systemd

### 问题来源

在 OpenClaw 部署机器上检查发现 `funasr-task-manager-backend.service` 被安装为系统级服务（`/etc/systemd/system/`），导致：
- 管理服务需要 `sudo`，Agent 无法直接操作
- 与 OpenClaw、Hermes、Cursor 等运行在用户空间的 Agent 工具不协调

### 评估结论

完全适合用户级服务：
1. 已经以 `wangminle` 用户身份运行，系统级切用户多此一举
2. 所有文件在用户目录下（`$HOME`）
3. 监听非特权端口（15797），不需要 root
4. 统一到用户级后 Agent 可直接管理生命周期

### 修改

**init/SKILL.md**（Phase 8）：

- 标题改为"systemd 用户级服务守护"
- 新增醒目警告："必须使用用户级服务（`systemctl --user`）"
- 新增旧系统级服务检测与迁移流程
- 所有命令改为 `systemctl --user` + `loginctl enable-linger`
- Phase 5 选项描述更新为"用户级服务（无需 sudo）"
- 错误处理表新增"已有系统级服务"场景

**references/systemd-setup.md**：

- 完全重写为用户级服务模板
- unit 文件差异：去掉 `User=` 行、`WantedBy=default.target`、写入 `~/.config/systemd/user/`
- 新增 Step 0（旧系统级服务迁移）
- 新增 `loginctl enable-linger` 步骤及说明
- 验证输出和常用命令全部改为 `--user` 形式

**_shared/ASR-WORKFLOW.md**：

- Phase 2 后端启动命令改为 `systemctl --user start`

## 五、默认端口迁移：8000 → 15797

### 动机

8000 是 Python 开发中最常用的端口（FastAPI/Django/http.server 等），本地开发冲突概率高。改为 5 位数端口避免冲突，同时规划连续端口组。

### 新端口规划

| 服务 | 新端口 | 旧端口 | 说明 |
|------|--------|--------|------|
| 后端 API | **15797** | 8000 | FastAPI / Uvicorn |
| 前端 Dev | **15798** | 5173 | Vite dev server |
| 预留 | **15799** | — | 未来扩展（调试代理等） |
| Redis | 6379 | 6379 | 标准端口，不变 |
| Prometheus | 9090 | 9090 | 标准端口，不变 |
| Alertmanager | 9093 | 9093 | 标准端口，不变 |
| Grafana | 3001 | 3001 | 已避开默认 3000，不变 |

## 七、部署注意事项

### 已部署实例迁移

如果线上已有运行中的 `funasr-task-manager-backend` 服务：

1. **端口迁移**：更新 systemd unit 文件中的 `--port` 参数，或设置环境变量 `ASR_PORT=15797`
2. **系统级 → 用户级迁移**：
   ```bash
   # 停止并禁用旧的系统级服务
   sudo systemctl disable --now funasr-task-manager-backend
   sudo rm /etc/systemd/system/funasr-task-manager-backend.service
   sudo systemctl daemon-reload

   # 创建用户级服务（无需 sudo）
   mkdir -p ~/.config/systemd/user
   # 编辑 ~/.config/systemd/user/funasr-task-manager-backend.service
   # （模板见 init/references/systemd-setup.md）
   systemctl --user daemon-reload
   systemctl --user enable --now funasr-task-manager-backend
   loginctl enable-linger $USER
   ```
3. **CLI 配置更新**：`python -m cli config set server http://localhost:15797`
4. **前端代理**：如果有 nginx 反向代理，更新 upstream 端口

### Docker 部署

Docker 用户只需 `docker compose pull && docker compose up -d`，新端口映射自动生效。

## 八、Playwright start-backend.js 端口遗漏修复（P1）

### 问题来源

Code Review 发现：端口迁移时遗漏了 `3-dev/src/frontend/scripts/start-backend.js` 第 53 行，仍硬编码 `--port 8000`。Playwright 的 `webServer[0].url` 检查 `http://127.0.0.1:15797/health`，但 start-backend.js 在 8000 启动 uvicorn，导致 15797 永远不可达，前端 E2E 测试必然超时失败。

### 修复

```diff
- ['-m', 'uvicorn', 'app.main:app', '--host', '127.0.0.1', '--port', '8000'],
+ ['-m', 'uvicorn', 'app.main:app', '--host', '127.0.0.1', '--port', '15797'],
```
