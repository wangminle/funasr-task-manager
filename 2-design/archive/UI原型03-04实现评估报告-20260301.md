## 3) 未完成模块与当前实现方案是否冲突，以及后续技术设计

## 3.1 结论先行

- 未完成模块**有必要继续实现**，否则监控页与详情页无法达到原型验收标准。
- 存在 4 类“方案级冲突/断层”，需要先做架构补齐再做页面补齐。

## 3.2 主要冲突点与设计建议

### 冲突 A：调度策略设计与实际执行链路脱节

现状：

- `scheduler.py` 有 LPT + earliest-finish + penalty 设计。
- `task_runner.py` 实际分配逻辑未调用 scheduler。

影响：

- 原型里的 ETA、节点负载、RTF 趋势可信度不足。

建议：

1. 在 `task_runner` 引入策略开关：`ASR_SCHED_STRATEGY=fifo|lpt`（默认 fifo，平滑迁移）。
2. 将 `TaskScheduler.assign_single_task` 或 `schedule_batch` 接入 dispatch。
3. 把 ETA 结果回写 `tasks.eta_seconds`，让列表/监控可复用统一口径。

### 冲突 B：断路器/重试模块孤岛化

现状：

- 有 `circuit_breaker.py`、`retry.py`、`callback.py`，但 task 执行主链路未接入断路器状态。

影响：

- 监控页“断路器”字段无法真实展示；故障场景的可解释性弱。

建议：

1. 在任务派发前做 `breaker.pre_check()`。
2. ASR 成功回调 `record_success()`，失败路径 `record_failure()`。
3. 将 breaker state 透出到 `ServerResponse`（新增 `circuit_breaker` 字段）。

### 冲突 C：监控图表需要时序数据，但当前只做瞬时查询

现状：

- 监控页主要基于 `listServers + listTasks(page_size=200)` 即时计算。
- 无队列深度、RTF、吞吐的时序存储层。

影响：

- 无法实现“最近 1 小时”图表与告警中心。

建议（两种路线）：

1. 快速路线（推荐当前阶段）：新增 `monitor_snapshots` 表，每 10 秒落一条快照；前端查 `query_range` 风格 API。
2. 标准路线（后续）：完善 Prometheus 指标写入，后端代理 Prometheus Query API 给前端图表用。

### 冲突 D：详情页“状态历史/结果预览”缺后端聚合接口

现状：

- 状态事件在 `task_events` 已落库。
- 前端只接了 SSE 流，页面重开后历史丢失；无结果预览 API。

建议：

1. 新增 `GET /api/v1/tasks/{id}/events`（或在详情接口中聚合返回最近 N 条）。
2. 新增 `GET /api/v1/tasks/{id}/result/preview?format=...&limit=...`。
3. 前端详情页补“格式切换 + 预览区 + 刷新按钮”。

## 4) 现有协议是否支持直接探测服务器线程数和 IO 数

## 4.1 结论

**当前协议与当前实现都不支持“直接探测线程数和 IO 数”。**

- 现有 websocket 请求/响应字段没有线程数、IO 线程相关返回字段。
- 你们当前 `server_probe` 也只探测可达性、模式支持、`is_final` 语义等能力。
- 因此要输出这些字段，通常需要**服务端改造或旁路指标接口**。

## 4.2 证据

1. 当前项目适配器与探测器字段中无 thread/io：
   - `adapters/funasr_ws.py`：start message 主要是 `mode/wav_name/wav_format/audio_fs/is_speaking/chunk_size/...`
   - `services/server_probe.py`、`schemas/server.py`：返回 `supports_offline/2pass`、`inferred_server_type`、`probe_duration_ms` 等，无 thread/io。

2. FunASR 官方 websocket 服务端源码显示：
   - 启动参数有 `--decoder-thread-num`、`--io-thread-num`（这是服务端启动配置）。
   - websocket 回包字段中常见 `text/mode/timestamp/is_final/stamp_sents`，未见 thread/io 输出字段。

## 4.3 设计建议（按改造成本排序）

1. **最低成本（无协议改动）**：在节点注册时人工配置 `max_concurrency` + `io_threads` 元数据，监控展示“配置值”。
2. **推荐（轻改服务端）**：FunASR 增加能力查询消息（如 `{"cmd":"capabilities"}`）并返回 thread/io。
3. **生产级**：FunASR 或 sidecar 暴露 Prometheus 指标，适配层定时采集并聚合到监控 API。
