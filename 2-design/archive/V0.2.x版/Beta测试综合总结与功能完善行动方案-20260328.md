## 2. 两位用户反馈的共性问题

综合两位用户的独立测试经验，提炼出以下 **高度一致** 的问题：

### 2.1 CLI 串行瓶颈（两位用户均遇到，最核心痛点）

| 维度 | 现状 | 影响 |
|------|------|------|
| `transcribe` 命令 | `for` 循环 + `wait_for_task()` 阻塞 | 10 文件耗时 25min，串行退化 |
| 后端批量能力 | `POST /api/v1/tasks` 支持 `items[]` 批量 | CLI 没有调用这个能力 |
| 调度器 | `schedule_batch()` LPT + EFT 已实现 | CLI 逐任务创建，无法触发批量调度 |
| `upload` 命令 | 支持 `--create-task` 但也是逐个创建 | 批量上传后仍然一个个建任务 |

**源码证据**（`transcribe.py` 第 44-89 行）：
```python
for fp in existing:           # 串行循环
    file_data = c.client.upload_file(fp)      # 上传
    tasks = c.client.create_tasks(items)       # 建1个任务
    task = wait_for_task(c.client, task_id)    # 阻塞等待完成
    content = c.client.get_result(task_id)     # 下载结果
```

**两位用户都最终绕过 CLI、直接用 API 才实现了并行。**

### 2.2 批次（task_group_id）概念未产品化

| 维度 | 现状 | 影响 |
|------|------|------|
| 后端 | `create_tasks()` 多任务时生成 `task_group_id` | 字段存在但无后续接口 |
| CLI | 无 `--group` 相关参数 | 用户拿到一批 task_id 后无法批次管理 |
| API | 无 task-groups 端点 | 无法按批次查询/下载/删除 |
| 用户感受 | 批量提交后"散装"管理 | 用户 A 需逐个轮询，用户 B 自写轮询循环 |

### 2.3 服务管理 CLI 缺失

| 后端 API 已有 | CLI 现状 |
|--------------|---------|
| `POST /servers/{id}/probe` | 无 `server probe` 命令 |
| `POST /servers/benchmark` | 无 `server benchmark` 命令 |
| `PATCH /servers/{id}` | 无 `server update` 命令 |

用户 A 和用户 B 注册服务器时都用了 `curl` 而非 CLI，反映 CLI 服务管理能力不足。

### 2.4 后台稳定性问题（用户 B 日志证实）

当天后端日志出现持续报错：`no such column: callback_outbox.outbox_id`。

- 根因：`callback_outbox` ORM 模型（`outbox_id`/`sent_at`）与数据库迁移创建的表结构（`id`/`signature`/`updated_at`）不一致
- 影响：`task_runner` 主循环异常，回调重试不可用
- 用户感受：看到后台报错不知道是否影响转写

### 2.5 文档与引导不足

| 缺陷 | 表现 |
|------|------|
| CLI 适用场景未说明 | 用户 B 用 CLI 跑批量，浪费 25 分钟才发现是串行 |
| 批量操作推荐路径缺失 | 两位用户都是靠自己摸索出 API 批量方案 |
| 错误分级不清 | 用户无法判断日志报错是阻断性还是可忽略 |
| 依赖说明缺失 | `ffprobe` 缺失时影响范围不明确 |
| 环境诊断工具缺失 | 无 `system doctor` 类命令 |

## 4. 功能完善行动方案

### 4.1 第一优先级（P0）：消除稳定性阻断 + 打通批量主流程

#### P0-1 修复 callback_outbox schema 漂移

**目标**：消除 `no such column: callback_outbox.outbox_id` 持续报错。

**开发任务**：

| # | 任务 | 说明 |
|---|------|------|
| 1 | 新增 Alembic 迁移 | 对齐 `callback_outbox` 表结构到当前 ORM 模型（`outbox_id` 主键、`sent_at` 字段） |
| 2 | 兼容老库升级 | 迁移脚本需检测旧字段存在性，自动 rename/add/drop，不要求用户手工删库 |
| 3 | 启动自检 | 应用启动时校验关键表结构与 ORM 模型一致性，不一致时阻断启动并给出明确提示 |
| 4 | 健康检查增强 | `GET /health` 增加 `schema_ok` 字段 |

**测试要求**：
- 单元测试：从旧 schema 升级到新 schema 的迁移回归测试
- 集成测试：`_retry_pending_callbacks` 在新 schema 下正常扫描和投递

#### P0-2 CLI 批量并行转写（`transcribe` 命令增强）

**目标**：让 `transcribe` 命令成为可直接用于批量生产的入口。

**设计方案**：

```
# 新增参数
--batch           批量模式（多文件时自动启用）
--wait all|none   等待策略：all=等整批完成（默认），none=提交即返回
--download        整批完成后自动下载结果
--json-summary    输出批次 JSON 摘要
```

**行为变更**：

| 场景 | 当前行为 | 新行为 |
|------|---------|-------|
| 单文件 | 上传→建任务→等待→下载 | **不变**，保持兼容 |
| 多文件（默认） | for 循环串行阻塞 | 全部上传→一次性 `create_tasks(items[])`→整批视角轮询 |
| 多文件 + `--wait none` | 不存在 | 全部上传→批量创建→输出 task_group_id 后立即返回 |
| 多文件 + `--download` | 不存在 | 整批完成后批量下载到 `--output-dir` |

**核心代码改造思路**（`transcribe.py`）：

```python
# 阶段一：全部上传
file_ids = []
for fp in existing:
    file_data = c.client.upload_file(fp)
    file_ids.append((fp, file_data["file_id"]))

# 阶段二：一次性创建任务
items = [{"file_id": fid, "language": language, "options": options or None}
         for _, fid in file_ids]
tasks = c.client.create_tasks(items, callback=cb)
task_group_id = tasks[0].get("task_group_id")

# 阶段三：整批视角等待
if wait_mode == "all":
    completed = wait_for_batch(c.client, [t["task_id"] for t in tasks],
                                poll_interval=poll_interval, timeout=wait_timeout)

# 阶段四：批量下载
if download:
    for task in completed:
        content = c.client.get_result(task["task_id"], fmt=fmt)
        dest = output_dir / f"{file_name_map[task['task_id']]}_result{suffix}"
        dest.write_text(content, encoding="utf-8")
```

**测试要求**：
- 集成测试：`transcribe 10个文件` 默认走批量路径，总耗时与 API 方案接近
- 回归测试：`transcribe 单文件` 行为不变
- 模拟测试：`--wait none` 提交后立即返回，输出 task_group_id

#### P0-3 将 task_group_id 产品化为批次资源

**目标**：用户可以用一个 `task_group_id` 完成查询、等待、下载和删除。

**API 新增**：

| 端点 | 方法 | 功能 |
|------|------|------|
| `/api/v1/task-groups/{group_id}` | GET | 批次概览（总数/完成数/失败数/进度） |
| `/api/v1/task-groups/{group_id}/tasks` | GET | 批次内任务列表 |
| `/api/v1/task-groups/{group_id}/results` | GET | 批次结果打包下载 |
| `/api/v1/task-groups/{group_id}` | DELETE | 删除整批任务及结果 |

**CLI 新增**：

| 命令 | 功能 |
|------|------|
| `task list --group <id>` | 按批次查看任务列表 |
| `task wait --group <id>` | 等待整批完成（整体进度展示） |
| `task result --group <id> --format txt --output-dir <dir>` | 批量下载结果 |
| `task delete --group <id>` | 删除整批任务 |

**现有 `task list` 改造**：在 `list_tasks` 命令中增加 `--group` 参数：

```python
@app.command(name="list")
def list_tasks(
    ctx: typer.Context,
    status: Optional[str] = typer.Option(None, "--status"),
    group: Optional[str] = typer.Option(None, "--group", help="按批次 ID 筛选"),
    ...
)
```

**现有 `task wait` 改造**：支持 `--group` 替代逐个传 task_id：

```python
@app.command(name="wait")
def wait(
    ctx: typer.Context,
    task_ids: Optional[list[str]] = typer.Argument(None),
    group: Optional[str] = typer.Option(None, "--group", help="等待整批完成"),
    ...
)
```

**测试要求**：
- API 测试：按 `task_group_id` CRUD 全流程
- CLI 测试：`task list --group`、`task wait --group`、`task result --group` 端到端
- 边界测试：单任务无 group_id 时的兼容性

#### P0-4 系统诊断命令

**目标**：让用户明确区分可继续运行的告警与必须先修复的阻断问题。

**CLI 新增**：`system doctor` 命令

```
$ asr-cli system doctor

[检查项]                    [状态]     [说明]
数据库迁移                  ✅ 正常    schema 版本: 003
callback_outbox 表结构      ✅ 正常    字段匹配
ffmpeg/ffprobe              ⚠️ 告警    ffprobe 未安装，时长估算将使用文件大小推算
上传目录可写性              ✅ 正常    ./uploads/ 可写
ASR 服务连通性              ✅ 正常    3/3 在线
ASR 服务 RTF 基准           ⚠️ 告警    funasr-96 RTF=0.737（较慢）
```

**API 新增**：`GET /api/v1/diagnostics`

```json
{
  "checks": [
    {"name": "database_schema", "level": "ok", "detail": "version 003"},
    {"name": "ffprobe", "level": "warning", "detail": "not found, duration estimation degraded"},
    {"name": "upload_dir", "level": "ok", "detail": "./uploads/ writable"},
    {"name": "asr_servers", "level": "ok", "detail": "3/3 online"}
  ],
  "has_blocking_errors": false
}
```

**错误分级**：

| 级别 | 含义 | 示例 |
|------|------|------|
| `error`（阻断） | 系统不可正常运行 | schema 漂移、数据库连接失败 |
| `warning`（降级） | 功能受限但可运行 | ffprobe 缺失、某台服务器离线 |
| `ok` | 正常 | 所有检查通过 |

**测试要求**：
- 单元测试：模拟各种异常场景（缺 ffprobe、schema 不一致、服务器不可达），验证输出正确等级
- CLI 测试：`system doctor` 命令输出格式正确

## 8. 最终结论

本次 Beta 测试的两位独立用户，在不同工具（Claude Code / OpenClaw）、不同操作路径下，**不约而同地撞上了同一个核心问题**：

> **系统的并行转写能力已经存在于后端，但 CLI 作为主要用户入口没有把这个能力暴露出来。**

具体数据：
- 用户 B 串行 CLI 耗时 25 分钟 → API 并行 5.6 分钟（**4.5 倍差距**）
- 用户 A 8 集视频 API 批量仅需 237 秒（3.95 分钟）

因此，下一阶段的核心任务是：

1. **先修后台**（P0-1）：消除 schema 漂移，让系统可信
2. **打通 CLI 批量**（P0-2）：让 `transcribe` 命令自动走批量路径
3. **产品化批次概念**（P0-3）：让用户可以用一个 ID 管理一整批任务
4. **增加诊断工具**（P0-4）：让用户能区分阻断错误和降级告警
5. **补齐文档**（P1-3）：明确告诉用户什么场景用什么工具

只有完成这个闭环，Beta 用户才会感受到"多服务器并行转写"是产品能力，而不是只有会写 API 脚本的人才能用出来的隐藏功能。
