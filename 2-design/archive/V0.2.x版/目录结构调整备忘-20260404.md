# 目录结构调整备忘

## 1. 调整目标

本次调整的目标是把“源码目录”“运行时目录”“测试素材与测试工件目录”彻底分离，避免以下历史问题反复出现：

1. 后端因为启动工作目录不同，把 SQLite、上传文件、结果文件或临时文件写到错误位置。
2. `3-dev/src/` 下混入持久化数据、临时数据或日志，导致源码树污染。
3. 浏览器 E2E、CLI 和 benchmark 的素材与输出散落在独立目录，和 `4-tests/` 体系脱节。
4. `.runtime/` 与其他运行时目录并存，职责重复。

## 2. 新目录约定

### 2.1 运行时目录

仓库根目录新增并正式采用：

```text
runtime/
├── storage/   # SQLite、uploads、results、temp、backups
└── logs/      # backend/frontend 日志、PID、迁移日志
```

结论：`.runtime/` 可以并入 `runtime/`，本次已经按这个方向完成代码调整。

### 2.2 测试素材与工件目录

原 `7-data/` 的职责迁移到：

```text
4-tests/batch-testing/
├── assets/
└── outputs/
    ├── cli/
    ├── e2e/
    └── benchmark/
```

这样测试素材、测试输出和 `4-tests/scripts/` 中的自动化脚本落到同一体系，更符合仓库结构语义。

## 3. 代码调整摘要

### 3.1 后端默认路径改为仓库根目录 runtime

关键文件：[3-dev/src/backend/app/config.py](../3-dev/src/backend/app/config.py)

本次把原来依赖 `./data` 的默认值，改为基于仓库根目录推导的固定路径：

1. 新增 `PROJECT_ROOT`、`DEFAULT_RUNTIME_ROOT`、`DEFAULT_STORAGE_ROOT`、`DEFAULT_LOGS_DIR`。
2. 新增 `_resolve_project_path()`，把相对路径统一解析到仓库根目录，而不是进程当前工作目录。
3. 新增 `_normalize_sqlite_url()`，把相对 SQLite URL 归一化为绝对路径。
4. 在 `Settings.model_post_init()` 中统一推导：
   - `runtime_root`
   - `storage_root`
   - `logs_dir`
   - `upload_dir`
   - `result_dir`
   - `temp_dir`
   - `database_url`

效果：即使从不同目录启动后端，默认数据库和文件存储仍然稳定落到 `runtime/storage/`，不会再写到 `3-dev/src/` 内部，也不会在仓库根目录误生成历史 `data/`。

## 4. 当前效果

本次调整完成后，仓库在目录职责上变为：

1. `3-dev/src/` 只承载源码、前后端工程与启动脚本，不再承载默认运行时数据。
2. `runtime/` 作为正式运行目录，适合本地开发和后续部署场景复用。
3. `4-tests/batch-testing/` 作为测试素材与测试工件目录，和 `4-tests/scripts/` 对齐。
4. 历史的 `.runtime/`、`7-data/`、`3-dev/src/backend/data/` 不再是活动代码依赖的默认路径。

## 5. 验证结果

本次调整后完成了以下验证：

1. 对活动代码、脚本、技能和说明文档做了关键路径全文检索，确认没有继续使用旧的默认 `./data`、`7-data`、`.runtime`、`/app/data` 作为活动配置入口。
2. 对关键修改文件执行错误检查，核心文件未发现新的语法或静态错误。
3. reset_db 的单元测试已同步到 `runtime/storage` 结构，可用于持续回归。

建议后续持续遵守两条约束：

1. 任何新的运行时持久化、缓存或日志，都优先放入 `runtime/`，不要回写到 `3-dev/src/`。
2. 任何新的测试素材、批量输出或基准结果，都优先放入 `4-tests/batch-testing/`，不要再新增平行的顶层数据目录。
