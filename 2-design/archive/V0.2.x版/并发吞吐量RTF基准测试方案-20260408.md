# 并发吞吐量 RTF 基准测试方案

> 日期: 2026-04-08  
> 最后更新: 2026-04-13  
> 状态: V2.1 已实施（口径修正 + 屏障修正 + 超时/早期异常改进）

## 1. 问题背景

当前 benchmark 仅测量**单线程 RTF**——依次发送一个音频文件到服务器，测量 `处理耗时 / 音频时长`。  
这导致线程数不同的服务器（如 4 线程 vs 8 线程 vs 16 线程）测出的 RTF 几乎相同，  
调度器无法区分它们的真实吞吐能力，任务分配趋于均等分配。

**根本原因**: 单文件 benchmark 只使用了服务器的 1 个 worker 线程，无法反映其并发处理能力。

## 2. 方案概述

引入**双指标 benchmark 体系**：

| 指标 | 含义 | 用途 |
|------|------|------|
| `single_rtf` | 单线程 RTF = 单文件处理耗时 / 音频时长 | 单任务 ETA 估算 |
| `throughput_rtf` | 吞吐量 RTF = 并发满载时的 wall_clock / (N × 音频时长) | 调度器配额分配 |

### 吞吐量 RTF 的含义

`throughput_rtf` 反映服务器在满载并发下，**每秒能处理多少秒音频**的倒数。  
例如 `throughput_rtf = 0.003` 意味着服务器每秒可处理约 333 秒的音频。

### 调度器配额公式

```
speed(server) = 1 / throughput_rtf
quota(server) = total_tasks × speed(server) / Σ speeds
```

这样 8 线程服务器自然获得约 2× 于 4 线程服务器的任务量。

## 3. 梯度并发测试流程

### 样本分工

| 阶段 | 样本文件 | 选择原因 |
|------|---------|---------|
| Phase 1: single_rtf | `tv-report-1.wav` (长 WAV) | 较长音频 → 单线程测量更准确 |
| Phase 2: throughput_rtf | `test.mp4` (短 MP4) ×N | 短文件 ×N 并发 → 快速完成梯度测试 |

### 测试流程

```
对每台 ONLINE 服务器:
  1. 单线程测试 → single_rtf
     - 发送 tv-report-1.wav (一次)
     - single_rtf = elapsed / audio_duration
     
  2. 梯度并发测试 → throughput_rtf
     对 N ∈ [1, 2, 4, 8] (始终跑完整梯度，不受 max_concurrency 封顶):
       - 同时建立 N 个 WebSocket 连接
       - 每个连接发送 test.mp4 的一份拷贝
       - 记录从第一个开始发送到最后一个收到结果的 wall_clock 时间 T
       - per_file_rtf(N) = T / audio_duration
       - throughput_rtf(N) = T / (N × audio_duration)
       - 记录各级别的结果用于分析
     
  3. 退化检测取最终值（详见 §3.1）
     - 从 N=1 逐级验证，找到最高非退化并发级别
     - throughput_rtf = throughput_rtf(recommended_N)
     - recommended_N 同时写入 max_concurrency
```

### 3.1 退化检测算法（2026-04-09 增强）

**原方案**取 `throughput_rtf(max_tested_N)`，即无条件使用最高级别的结果。
**增强方案**使用退化检测逐级验证，避免以下问题：

- 服务器只有 2 个 worker 但配置了 `max_concurrency=4`，N=4 时吞吐不增
- 某次 benchmark 因瞬时负载得到较低推荐值，后续 benchmark 受 max_concurrency 封顶无法恢复

**算法流程**：

```
best = gradient[0]  # 从 N=1 开始

for i in 1..len(gradient):
  current = gradient[i]
  prev = gradient[i-1]
  
  # 条件 1: 吞吐量必须有 ≥10% 的改善（V1 wall-clock 口径）
  improvement = 1 - current.throughput_rtf / prev.throughput_rtf
  if improvement < 0.10:
    break  # 退化点
  
  # 条件 2: 单文件 RTF 不能超过 single_rtf 的 2 倍（V1 wall-clock 口径）
  if current.per_file_rtf > single_rtf × 2.0:
    break  # 退化点
  
  best = current

recommended_concurrency = best.concurrency
throughput_rtf = best.throughput_rtf
```

**口径约束**（2026-04-13 修正）：退化检测统一使用 V1 wall-clock 指标（`throughput_rtf`、`per_file_rtf`），与 `single_rtf`（`total_ms / duration`）同基准。`server_throughput_rtf` / `server_per_file_rtf` 仅作为观测指标记录在 `ConcurrencyGradient` 上，不参与推荐并发的决策。原因：`concurrent_post_upload_ms`（尾部时间）不包含上传期间的服务端流式处理，用它约束端到端口径的 `single_rtf` 基线会系统性放宽阈值。

**关键设计决策**：

| 决策 | 说明 |
|------|------|
| 梯度不受 max_concurrency 封顶 | 防止螺旋下降：被降低的 max_concurrency 限制后续探测范围 |
| `THROUGHPUT_MIN_IMPROVEMENT = 0.10` | 吞吐量需有 ≥10% 改善才认为当前级别有效 |
| `PER_FILE_MAX_DEGRADATION = 2.0` | 单文件延迟不超过 single_rtf 的 2 倍 |
| recommended_concurrency → max_concurrency | benchmark 自动校准调度槽位数（仅在梯度完整时允许降级，见 §3.2） |
| server_* 仅做观测 | 口径不同于端到端基线，不用于驱动推荐并发 |

**退化检测示例**（2 worker 节点冒充 4 并发）：

| N | per_file_rtf | throughput_rtf | 改善幅度 | 判定 |
|---|-------------|---------------|---------|------|
| 1 | 0.024 | 0.024 | — | ✅ 基准 |
| 2 | 0.025 | 0.0125 | 47.9% | ✅ 接受 |
| 4 | 0.050 | 0.0125 | 0% | ❌ 退化 |

→ `recommended_concurrency = 2`，`throughput_rtf = 0.0125`

### 3.2 不完整梯度保护（2026-04-09）

如果梯度测试因超时或网络瞬时错误提前终止（例如 N=1 成功但 N=2 超时），`gradient_complete` 标记为 `false`。此时 `_apply_benchmark_result()` **只允许 `max_concurrency` 上调或维持**，拒绝基于不完整证据的降级。

原因：一次瞬时抖动不代表服务器并发能力下降，不应把原本 4/8 槽的节点永久降到 1 槽。

| 场景 | gradient_complete | recommended < current | 结果 |
|------|:-:|:-:|------|
| 完整梯度 + 退化检测降级 | true | 是 | 正常写入（可信降级） |
| 部分梯度 + 建议升级 | false | 否 | 正常写入（向上安全） |
| 部分梯度 + 建议降级 | false | 是 | **跳过写入**，保持原 max_concurrency |

## 7. V2 优化：时间分解 + 同步起跑（2026-04-09）

### 7.1 问题：V1 的计时缺陷

V1 benchmark 存在两个核心缺陷：

**缺陷 1：时间不分段**  
`_benchmark_single_sample` 把"建连 → 上传音频 → 等待处理 → 收到结果"整体计为一个 elapsed 值。
远端测试（WAN/跨地域）时，网络传输时间（上传 + 下行）可能占总耗时的 50%+，
导致不同并发级别测出的 RTF 差异被网络噪声淹没，无法区分服务端真实处理能力差异。

**缺陷 2：并发上传无同步**  
`_benchmark_concurrent` 对 N 路连接使用 `asyncio.gather`，
每路独立建连 → 独立上传 → 独立等待。先完成上传的连接会让服务器提前开始处理，
后续连接的音频到达时服务器可能已经空闲，无法真正测量"N 路同时到达"时的并发处理能力。

### 7.2 V2 设计目标

1. **剥离客户端时间**：将 wall_clock 拆分为 `connect_ms`、`upload_ms`、`post_upload_wait_ms`，
   用 `post_upload_wait_ms` 近似服务端处理时间
2. **同步起跑**：N 路连接预先建好，用 `asyncio.Event` 同步触发上传，
   确保服务器在短时间窗口内同时收到 N 路音频
3. **Ping RTT 基线**：在每个梯度级别前测一次 WebSocket ping/pong RTT，
   用于判断网络环境（LAN < 5ms / WAN > 50ms）并辅助校正

### 7.3 时间分解模型

每一路 WebSocket 连接采集 5 个时间点，推导 4 个阶段耗时：

```
Timeline per connection (单线程):

  t0              t1                  t2              t3           t4
  |── connect ──|── upload chunks ──|── wait... ──|── ... ──|
  ws_connect     start_msg_sent     upload_done    first_resp  final_resp
                                    (EOF sent)

Timeline per connection (并发, N>1):

  t0              t1                       t_upload_start       t2       t3      t4
  |── connect ──|── barrier wait... ──|── upload chunks ──|── wait ──|── ... ──|
  ws_connect     start_msg_sent        fire_event 后       upload_done         final_resp
                 ready_count++

阶段耗时:
  connect_ms           = (t1 - t0) × 1000    建连 + TLS 握手 + 发送 start_msg
  upload_ms (单线程)   = (t2 - t1) × 1000    发送全部 chunks + EOF
  upload_ms (并发)     = (t2 - t_upload_start) × 1000   纯上传, 不含 barrier wait
  first_response_ms    = (t3 - t2) × 1000    EOF → 首条结果
  post_upload_wait_ms  = (t4 - t2) × 1000    EOF → 最终结果 ≈ 服务端处理时间
  total_ms             = (t4 - t0) × 1000    端到端总耗时
```

### 7.4 并发测试执行流程

```
Phase 0: Ping RTT 测量（在梯度级别 timeout 内）
  ┌──────────────────────────────────┐
  │ _measure_ping_rtt()              │
  │ 与后续 benchmark 共享超时预算    │
  │ 失败不中断 (观测项)              │
  └──────────────────────────────────┘
                    │
Phase 1: 预连接 + 早期异常检测
  ┌──────────────────────────────────┐
  │ 为 N 路各自:                     │
  │  1. 建立 WebSocket 连接          │
  │  2. 发送 start_msg               │
  │  3. 就绪后 ready_count += 1      │
  │                                  │
  │ 等待方式: asyncio.wait(          │
  │   [ready_waiter, *worker_tasks], │
  │   FIRST_COMPLETED, timeout=30s)  │
  │                                  │
  │ worker 在 connect/send 阶段抛错  │
  │ → 立即报出真实根因, 不等 30s     │
  │                                  │
  │ ready_count == N → ready!        │
  └──────────────────────────────────┘
                    │
Phase 2: 同步上传（Event 发令枪）
  ┌──────────────────────────────────┐
  │ fire_event.set()                 │
  │                                  │
  │ conn_i: t_upload_start = now()   │
  │   ← barrier 后才记录上传起始     │
  │ conn_1 ─→ send chunks + EOF     │
  │ conn_2 ─→ send chunks + EOF     │
  │ ...                              │
  │ conn_N ─→ send chunks + EOF     │
  │                                  │
  │ upload_ms = t2 - t_upload_start  │
  │   (纯上传, 不含 barrier wait)    │
  │ 记录每路 upload_done_abs[i]      │
  └──────────────────────────────────┘
                    │
Phase 3: 等待结果
  ┌──────────────────────────────────┐
  │ conn_1 ─→ recv until final      │
  │ conn_2 ─→ recv until final      │
  │ ...                              │
  │ conn_N ─→ recv until final      │
  │                                  │
  │ 记录每路 final_resp_abs[i]       │
  └──────────────────────────────────┘
                    │
Phase 4: 聚合指标
  ┌──────────────────────────────────┐
  │ upload_spread_ms =               │
  │   max(upload_done) -             │
  │   min(upload_done)               │
  │                                  │
  │ concurrent_post_upload_ms =      │
  │   max(final_response) -          │
  │   max(upload_done)               │
  │                                  │
  │ → server_per_file_rtf  (观测用)  │
  │ → server_throughput_rtf(观测用)  │
  └──────────────────────────────────┘
```

### 7.6 RTF 计算变更

**保留（向后兼容）：**
- `per_file_rtf = wall_clock / audio_duration`
- `throughput_rtf = wall_clock / (N × audio_duration)`

**新增（观测指标，不驱动决策）：**
- `server_per_file_rtf = concurrent_post_upload_ms / 1000 / audio_duration`
- `server_throughput_rtf = concurrent_post_upload_ms / 1000 / (N × audio_duration)`

**退化检测统一使用 V1 wall-clock 指标**（`throughput_rtf`、`per_file_rtf`），与 `single_rtf` 同口径。`server_*` 指标仅记录在 `ConcurrencyGradient` 上供分析，不参与推荐并发的计算。原因：`concurrent_post_upload_ms` 是"最后上传完到最后结果回"的尾部时间，不包含上传期间的服务端流式处理，用它约束端到端口径的 `single_rtf` 基线会系统性放宽阈值。

### 7.7 Ping RTT 基线

在每个梯度级别前，用 WebSocket `ws.ping()` / `ws.pong` 测一次 RTT。RTT 测量在该级别的 `asyncio.timeout(timeout)` 内执行，与后续 benchmark 共享超时预算，不会额外阻塞：

- LAN 场景: RTT < 5ms → `post_upload_wait_ms` 就约等于服务端处理时间
- WAN 场景: RTT > 50ms → 报告中标注网络延迟占比
- 设备内部: RTT < 1ms → 几乎纯服务端处理时间

### 7.8 上传离散度（upload_spread）处理

asyncio 事件循环是单线程的，N 路 `await ws.send(chunk)` 交替执行。
当 N=8 且带宽有限时，upload_spread 可能较大。

处理策略：
- **upload_spread < 总上传时间的 20%**: 视为"近似同时到达"，指标有效
- **upload_spread > 总上传时间的 50%**: 在 benchmark_notes 中标注
  "上传离散度高，server_throughput_rtf 可能包含部分并发上传重叠"
- 无论如何，`server_throughput_rtf` 始终比 V1 的 `throughput_rtf` 更准确，
  因为至少剥离了建连时间和大部分上传时间
