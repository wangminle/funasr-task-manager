> 日期：2026-04-07
> 背景：0403 / 0404 两轮 CLI 批量调度测试对比分析
> 目标：在严格并发约束下最小化批次完工时间（makespan）

> 2026-04-08 复核：本文中的 Phase 1 / Phase 2 / Phase 3 方案已经在当前代码中落地，相关实现位于 `scheduler.py` 与 `task_runner.py`。本文保留 0407 的分析过程，但已把实施状态更新为与仓库现状一致。

关键发现：**短任务受害最严重**（排队增量 +79s），长任务反而受益（并发竞争减少后执行更快）。

### 1.3 根因分析：三层开销叠加

**根因 1：0404 基线版本的调度轮次往返延迟（Dispatch Latency）**

0404 基线补位机制链路：任务完成 → `_request_dispatch()` → `_dispatch_queued_tasks()` → 查 DB + 构建 profiles → `schedule_batch()` → `select_dispatchable_now()` → commit → `asyncio.create_task()`。每轮 DB 查询 + 算法 + commit 约造成 0.5-2s 空槽闲置。25 个任务分 5-6 轮下发，累积空槽时间可达 5-10s。

**根因 2：0404 基线版本中 `_allocate_quotas` 的 `enforce_min` 规则过度均衡化**

```python
# 0404 基线代码
enforce_min = effective_count >= len(servers_with_slots)
quotas[sid] = max(1, round(raw)) if enforce_min else round(raw)
```

当补位轮次只剩几个任务时，3 台机器都有空槽，`enforce_min=True` 导致每台至少分 1 个——即使慢节点预估完成时间远长于快节点。例：剩 3 个任务 → 配额 1:1:1，理想应 3:0:0（全给快节点）。

**根因 3：旧版"违规并发"反映出 max_concurrency 可能过于保守**

旧版 `asr-local-01` 同时跑 16 个任务（远超 max_concurrency=4），总耗时更短。说明对该 ASR 服务器，真实并发能力可能大于 4。

## 二、优化目标与约束

### 2.1 目标函数（按优先级排序）

1. **主目标：最小化整批 makespan**
2. **次目标：最大化快节点利用率**（快节点空闲时间占比 < 20%）
3. **护栏：避免慢节点完全饥饿**（任何节点不应连续 3 个 dispatch 周期无任务）

### 2.2 硬约束

- 单节点并发 ≤ `max_concurrency`（不可违反）
- 全局并发 ≤ 所有节点 `max_concurrency` 之和
- 只重排 QUEUED 状态任务，不回退 DISPATCHED/TRANSCRIBING 任务

### 2.3 明确不追求

- 任务数在节点间均衡分配
- 各节点负载完全相等

## 三、方案设计

### 3.1 总体架构：三阶段调度

截至 2026-04-08，当前代码已经按下述三阶段思路实现。

```
Phase 1: 全局预规划（LPT + EFT）
  → 一次性规划全部任务，为每个虚拟槽位生成有序队列
  → 长任务排前面，短任务排后面（LPT）
  → 配额严格按速度比例，不强制每机至少 1

Phase 2: 按槽位队列下发
  → 每台服务器首波下发 max_concurrency 个任务
  → 槽位空出来后，立即从本服务器队列取下一个（无需重新 schedule_batch）

Phase 3: 工作窃取
  → 某台服务器队列清空 + 有空闲槽位
  → 从其他服务器队列尾部"偷"短任务
  → 只在"偷过来更快完成"时才偷
```

### 3.2 场景对比

| 场景 | 旧版(过度派发) | 0404 基线版(补位重配额) | 当前实现(预规划+窃取) |
|------|--------------|------------------|-------------------|
| 首波下发 | 25个一次性全启 | 12个(4+4+4) | 12个(按速度比例分配) |
| 快节点完成后 | 已经全部在跑 | 重新 schedule_batch，被 enforce_min 均衡化 | 直接从本队列取下一个 |
| 快节点队列清空 | 不适用 | 重新配额，慢节点也分到任务 | 从慢节点队列尾部偷短任务 |
| 短任务等待 | 几乎不等 | 平均等 79s | 预期接近不等(在队列尾部，可被窃取) |

### 3.3 Phase 1 详细设计：去除 enforce_min，纯速度比例配额

**改动范围**：`scheduler.py` 的 `_allocate_quotas` 方法

**核心改动**：

```python
# 旧代码
enforce_min = effective_count >= len(servers_with_slots)
quotas[sid] = max(1, round(raw)) if enforce_min else round(raw)
# 舍入误差补偿中 min_q = 1 if enforce_min else 0

# 新代码
quotas[sid] = round(raw)  # 纯比例，慢节点可能为 0
# 舍入误差补偿中 min_q = 0（始终允许减到 0）
```

**补偿机制**：舍入误差的多余任务优先给最快节点。不足时从最慢节点减去（允许减到 0）。

**用实际数据验证**（RTF = 0.12 / 0.35 / 0.42，即速度比 8.33 : 2.86 : 2.38）：

场景 A：首波 12 槽位，25 个任务
- 旧配额：4:4:4（受 max_concurrency 限制，两版行为一致）
- 新配额：4:4:4（同样受限，无差异）

场景 B：补位轮次，3 个剩余任务，3 台都有空槽
- 旧配额（enforce_min）：1:1:1 → 慢节点不必要地分到任务
- 新配额（纯比例）：2:1:0 或 3:0:0 → 快节点拿到更多

场景 C：补位轮次，1 个剩余任务
- 旧版：1:0:0（两版行为一致，只有 1 个任务无法 enforce_min）
- 新版：1:0:0

**预期效果**：补位轮次中快节点配额增加 → 快节点利用率提升 → makespan 缩短。

### 3.4 Phase 2 详细设计：槽位队列预下发（已实现）

**新增数据结构**：

```python
@dataclass
class SlotQueue:
    server_id: str
    slot_index: int
    tasks: list[ScheduleDecision]  # 按执行顺序排列
```

**行为**：
1. 首次 `schedule_batch` 生成全量规划，按 `(server_id, slot_index, estimated_start)` 组织为 SlotQueue
2. `_dispatch_queued_tasks` 不再每轮重新调用 `schedule_batch`，而是从各 SlotQueue 头部取出已可执行的任务直接下发
3. 任务完成时仅触发该槽位的下一个任务下发，消除重新配额的开销

**触发重新规划的条件**（异常场景）：
- 服务器离线/熔断
- RTF 偏差超过阈值（如 P90 偏差 > 50%）
- 全量队列清空（批次完成）

### 3.5 Phase 3 详细设计：工作窃取（已实现）

**触发条件**（全部满足才窃取）：
1. 窃取方自己的 SlotQueue 已清空
2. 窃取方有空闲槽位（running_tasks < max_concurrency）
3. 被窃取的任务尚未开始执行（仍在队列中，状态 QUEUED）
4. `finish_if_stolen < finish_if_stay`——偷过来确实更快

**窃取策略**：
- 从其他服务器队列**尾部**开始扫描（即最短的任务，因为 LPT 排序）
- 选让全局 makespan 缩短最多的候选
- 短任务优先被窃取（更灵活，估计误差更小）

**不触发窃取**：
- 所有队列都空（批次已完成）
- 空闲服务器上的预估时间反而更长（慢节点不抢快节点的活）

```python
async def _try_work_stealing(self, idle_server_id: str):
```

## 七、风险与缓解

| 风险 | 影响 | 缓解措施 |
|------|------|---------|
| RTF 基线不准导致快节点过载 | 快节点实际变慢 | 运行时 RTF 校准 + P90 滚动统计 |
| 慢节点长期空闲 | 资源浪费 | 护栏指标监控 + EFT 自然分流 |
| 工作窃取引入竞态 | 任务重复执行 | 仅窃取 QUEUED 状态 + DB 乐观锁 |
| 全量预规划被异常打断 | 部分任务卡死 | 异常时回退到逐轮 schedule_batch |
