## 3. 核心结论摘要

本次 Beta 暴露出来的问题，不是单一问题，而是两层问题叠加：

### 3.1 第一层：产品能力暴露不足

后端实际上已经支持批量建任务、并行调度、多服务器分配，但 CLI 仍以“单文件上传 -> 单任务创建 -> 阻塞等待 -> 下载结果”的串行路径为主，导致用户直观感受是：

- 我明明配了 3 台服务器，为什么还是一个一个跑。
- 浏览器/API 能并行，为什么 CLI 不行。
- 系统看起来支持批量，为什么实际操作时没有“批量任务”的清晰概念。

### 3.2 第二层：后台存在真实稳定性缺陷

当天日志显示，后台在 server probe 成功后，`task_runner` 主循环持续报错：

- `no such column: callback_outbox.outbox_id`

这不是体验层问题，而是数据库 schema 与当前模型定义不一致导致的后台循环异常。即使该问题未必直接导致转写失败，也会带来以下后果：

- 用户看到后台持续报错，误以为转写系统整体不稳定。
- 回调补偿重试逻辑不可用。
- 运维和排障成本明显上升。
- 一部分“卡住”现象会被误判成并行调度问题。

因此，下一阶段不能只做“CLI 并行增强”，必须同时修补后台稳定性基础设施。

## 4. Beta 用户的主要困惑

### 4.1 困惑一：为什么配置了多台服务器，却没有看到并行转写

#### 现象

- 第一轮 Beta 使用 CLI 转写 10 个视频，总耗时约 25 分钟。
- 第二轮改为 API 批量创建任务后，总耗时降至约 5.6 分钟。

#### 真实原因

- `transcribe.py` 当前实现对每个文件执行一次完整阻塞流程。
- 代码中存在明确的 `for fp in existing` 串行循环，并在每个任务创建后立即 `wait_for_task(...)`。

#### 困惑一的用户侧感知

- 用户会自然认为“多服务器能力不生效”。
- 用户会怀疑调度器没有工作，或者服务器注册没有成功。

#### 实际判断

不是调度器缺失，而是 CLI 没有把后端的批量调度能力暴露出来。

### 4.2 困惑二：为什么 API/前端可以批量，CLI 却像单机脚本

#### 困惑二的现象

- 后端 `POST /api/v1/tasks` 已支持 `items` 批量创建。
- 前端上传页也是“多文件上传后一次性 createTasks(items)”。
- CLI 主命令仍围绕单任务阻塞路径设计。

#### 困惑二的用户侧感知

- 产品不同入口表现不一致。
- 用户不知道应该把 CLI 当作“单任务便捷工具”还是“批量生产工具”。

#### 本质问题

产品语义没有统一：系统内部已经有批量概念，但外部没有形成统一的“批次 / 任务组”工作流。

### 4.3 困惑三：后台日志持续报错，到底影响不影响转写

#### 当天日志证据

在 `08:33:43` 到 `08:33:48` 这段日志中，先出现 3 台服务的探活成功：

- `server_probe_done ... reachable=True uri=wss://100.116.250.20:10095`
- `server_probe_done ... reachable=True uri=wss://100.116.250.20:10096`
- `server_probe_done ... reachable=True uri=wss://100.116.250.20:10097`

随后立即出现 `task_runner_loop_error`：

- `no such column: callback_outbox.outbox_id`

并且该错误重复出现。

#### 根因核对

当前模型 `CallbackOutbox` 使用：

- `outbox_id` 作为主键
- `sent_at` 字段

但初始迁移仍创建的是旧结构：

- `id` 自增主键
- `signature`
- `updated_at`

这说明数据库迁移与当前 ORM 模型发生了漂移。

#### 困惑三的用户侧感知

- 不知道错误是否影响转写结果。
- 不知道该先处理环境问题、数据库问题，还是继续提交任务。

#### 产品缺口

- 缺少启动自检和迁移一致性告警。
- 缺少“阻断性错误”和“非阻断性告警”的分级呈现。

### 4.4 困惑四：批量任务完成后，如何统一等待、下载和回看

#### 现状

- 后端返回 `task_group_id`，但该概念没有被产品化成一等资源。
- CLI 没有 `--group` 维度的 wait、result、list、delete。
- 用户拿到一批 task_id 后，仍需要逐个处理结果。

#### 困惑四的用户侧感知

- 即使提交成功，也不知道如何“像处理一个批次一样”管理这一批任务。

### 4.5 困惑五：日志中的依赖告警是否必须处理

历史记录中已多次出现 `metadata_extraction_failed / ffprobe not found` 类告警。虽然当天这份 8 分钟日志中没有截到该条，但已有 Beta 报告和历史测试记录反复提到该问题。

#### 困惑五的用户侧感知

- 用户无法判断这是可忽略告警，还是会影响识别准确率与调度估算。

#### 困惑五对应的产品缺口

- 缺少环境诊断命令。
- 缺少依赖缺失时的清晰降级说明。

## 5. 当前系统能力与缺口判断

### 5.1 已经存在但未充分暴露的能力

| 能力 | 当前状态 | 问题 |
| ---- | -------- | ---- |
| 批量任务创建 | 后端已支持 | CLI 未形成对应主流程 |
| `task_group_id` | 后端已有返回字段 | 未提供面向批次的查询/下载/删除能力 |
| 多服务器调度 | 后端已支持 | 用户入口未形成直觉化并行操作 |
| 服务探测/测速/更新 | 后端 API 已支持 | CLI 未补齐 |
| 前端批量创建 | 已支持 | 但批次视图和批量结果管理不足 |

### 5.2 当前真实缺失的能力

| 能力 | 当前状态 | 影响 |
| ---- | -------- | ---- |
| 批量 CLI 工作流 | 缺失 | Beta 用户最核心痛点 |
| 任务组一等资源 | 缺失 | 批量任务不可管理 |
| 数据库 schema 漂移自检 | 缺失 | 后台循环异常难定位 |
| 环境诊断命令 | 缺失 | `ffprobe` 等依赖问题难解释 |
| 错误分级与可读提示 | 缺失 | 用户无法判断是否可继续操作 |

## 6. 功能补充需求

### 6.1 P0：必须优先完成

### P0-1 修复 callback_outbox schema 漂移并增加启动自检

#### P0-1 目标

消除当天日志中的 `callback_outbox.outbox_id` 异常，确保后台任务循环和回调补偿机制可用。

#### 开发要求

1. 新增数据库迁移，对齐 `callback_outbox` 当前模型结构。
2. 对历史库做兼容迁移，而不是要求用户手工删库重建。
3. 应用启动时增加 schema 版本与关键表结构自检。
4. 若发现表结构不匹配，健康检查和 CLI 均要给出明确阻断提示。

#### P0-1 验收标准

1. 当天日志中的 `no such column: callback_outbox.outbox_id` 不再出现。
2. `_retry_pending_callbacks` 可正常扫描 PENDING 记录。
3. 老库升级后无需人工 SQL 修表即可启动。
4. 增加针对迁移兼容的自动化测试。

### P0-2 增加 CLI 批量并行转写主流程

#### P0-2 目标

让 CLI 成为可直接用于批量生产的入口，而不是只能做单任务便捷调用。

#### 建议设计

在 `transcribe` 命令上补充以下能力：

- `--batch`：声明使用批量模式。
- `--wait [all|none]`：控制是提交后立即返回，还是等待整批完成。
- `--download [all|success]`：整批完成后批量下载结果。
- `--output-dir`：整批结果统一输出目录。
- `--json-summary`：输出批次摘要，包括 `task_group_id`。

#### 行为要求

1. 多文件时先完成上传，再一次性创建任务。
2. 批量创建后返回 `task_group_id` 和全部 `task_id`。
3. 等待模式应按整批视角展示进度，而不是逐个阻塞。
4. 保留当前单文件串行体验，避免破坏现有脚本。

#### P0-2 验收标准

1. CLI 处理 10 个文件时，默认批量模式可触发后端并行调度。
2. 与当前 API 批量方案相比，总耗时接近，不再退化到串行模式。
3. 单文件场景兼容现有参数和行为。

### P0-3 将 task_group_id 产品化为“任务批次”

#### P0-3 目标

让用户可以把一次批量提交视为一个完整对象进行管理。

#### API 补充建议

新增以下接口：

1. `GET /api/v1/task-groups/{task_group_id}`
2. `GET /api/v1/task-groups/{task_group_id}/tasks`
3. `GET /api/v1/task-groups/{task_group_id}/results`
4. `DELETE /api/v1/task-groups/{task_group_id}`

#### CLI 补充建议

新增以下命令或参数：

1. `task list --group <id>`
2. `task wait --group <id>`
3. `task result --group <id> --format txt --output-dir <dir>`
4. `task delete --group <id>`

#### 前端补充建议

1. 上传成功后展示批次编号。
2. 任务列表支持按批次过滤。
3. 增加批次详情页或批次抽屉视图。
4. 支持整批下载结果。

#### P0-3 验收标准

1. 一次批量提交后，用户可只凭 `task_group_id` 完成查询、等待、下载和删除。
2. 前端、CLI、API 对“批次”语义保持一致。

### P0-4 增加系统诊断与错误分级输出

#### P0-4 目标

让用户明确区分“系统可继续运行的告警”和“必须先修复的阻断问题”。

#### 功能要求

1. 增加 `system doctor` 或等效 CLI 诊断命令。
2. 检查数据库迁移状态、ffmpeg/ffprobe、上传目录可写性、服务连通性。
3. 健康检查接口增加 `verbose` 模式或单独诊断接口。
4. 将错误分为：阻断错误、功能降级告警、信息提示。

#### P0-4 验收标准

1. 数据库 schema 漂移被标记为阻断错误。
2. `ffprobe` 缺失被标记为降级告警，并提示影响范围。
3. CLI 和前端都能看到同一套诊断结论。

### 6.2 P1：建议本轮一并完成

### P1-1 补齐 CLI 的服务管理能力

当前后端已有：

- 服务探测
- 服务测速/benchmark
- 服务更新

CLI 仍缺少对应入口。建议补齐：

1. `server probe <server_id>`
2. `server benchmark`
3. `server update <server_id> ...`

目标是让 CLI 与前端、后端 API 基本对齐。

### P1-2 补齐批量结果交付体验

建议支持：

1. 批量结果打包下载为 zip。
2. 同时导出 txt/json/srt。
3. 输出批次级 summary 文件，记录原文件名、task_id、状态、输出路径。

### P1-3 README 与运维文档重写批量工作流说明

需要明确写清：

1. CLI 单任务模式适合什么场景。
2. 批量模式应如何使用。
3. API 与前端的推荐路径。
4. 依赖缺失与迁移错误的诊断方式。

### 6.3 P2：可以放在后续优化

### P2-1 上传阶段并行化

当前前端和大多数脚本仍偏向串行上传。后续可增加受控并发上传，进一步缩短大批量准备时间。

### P2-2 批次级统计与复盘

建议补充：

1. 批次平均耗时
2. 各服务器分配占比
3. 成功率与失败原因分布
4. 批次级导出报告

## 8. 开发顺序建议

### 第一阶段：先消除稳定性阻断

1. 修复 callback_outbox 迁移与模型漂移。
2. 增加启动自检和健康检查阻断提示。

### 第二阶段：打通批量生产主流程

1. CLI 批量模式。
2. 批次一等资源。
3. 批量等待与批量下载。

### 第三阶段：对齐多入口体验

1. 补齐 CLI 服务管理。

3. README、运维文档、故障诊断说明。

## 9. 测试与验收要求

### 9.1 自动化测试

必须新增或补齐以下测试：

1. 数据库迁移兼容测试：老 schema 升级到新 schema。
2. callback_outbox 重试测试：PENDING 记录可被正常扫描和投递。
3. CLI 批量转写集成测试：多文件一次创建、多任务完成、多结果下载。
4. 批次 API 测试：按 `task_group_id` 查询、下载、删除。
5. 诊断命令测试：缺依赖、schema 错误、服务不可达时输出正确等级。

### 9.2 人工验收场景

1. 注册 3 台服务器，提交 10 个文件，CLI 批量模式总耗时明显优于串行模式。
2. 提交后仅记录一个批次号，用户可通过该批次号完成后续所有操作。
3. 使用旧数据库启动时，系统能明确提示是否需要迁移，而不是直接在后台循环报栈。
4. 缺少 ffprobe 时，用户能在诊断页或 CLI 中看到明确说明。

---

## 10. 非目标说明

本轮需求不把以下事项作为核心目标：

1. 重写调度算法。当前证据表明，调度器并不是本次 Beta 的主问题。
2. 彻底改造前端上传交互视觉。当前重点是批次语义和诊断可见性。
3. 引入新的分布式基础设施。当前优先级仍是把已有能力稳定、完整地暴露出来。

---

## 11. 最终结论

本次 Beta 的结论可以归纳为一句话：

**系统的并行能力基本存在，但没有以正确的产品形态交付给用户；与此同时，后台还存在一个会持续制造误判的 schema 漂移稳定性问题。**

因此，下一阶段的正确方向不是单纯“做一点 CLI 优化”，而是完成以下闭环：

1. 先修复后台迁移与自检，保证系统可信。
2. 再把批量任务正式产品化为“批次”工作流。
3. 最后让 CLI、API、前端三种入口在能力与语义上保持一致。

只有这样，Beta 用户才会真正感受到“多服务器并行转写”是产品能力，而不是只能靠手写 API 脚本才能用出来的隐藏能力。
