# 服务器本地文件批量转写 Skill 设计方案

> 日期：2026-05-03
> 目标 Skill：`funasr-task-manager-local-batch-transcribe`
> 适用平台：Codex、Claude Code、OpenClaw、Hermes，以及其他支持本地文件读写和命令执行的智能体运行时

---

## 一、背景与问题

此前服务器本地文件的批量转写主要依赖人工给出连续指令，例如“扫描这个目录”“把这些文件逐个转写”“看一下进度”“失败的再跑一次”。虽然项目已有后端批量任务能力和 CLI 多文件转写能力，但缺少一个面向智能体的固定执行规程，导致以下问题：

| 问题 | 影响 |
|------|------|
| 待处理目录没有统一入口 | 用户每次都要说明文件在哪里，智能体也容易遗漏文件 |
| 没有扫描中间目录 | 用户直接把源文件、转写结果、临时清单混放，后续重试和追踪困难 |
| 缺少自动排队与跳过规则 | 已完成文件可能被重复提交，失败文件不易定位 |
| 进度反馈依赖人工询问 | 批量任务长时间运行时，用户无法及时知道当前处理情况 |
| 各智能体平台实现不一致 | Codex、Claude Code、OpenClaw、Hermes 各自临场发挥，可靠性不稳定 |

本设计新增一个平台无关的本地批量转写 Skill，把“发现文件 → 预检查 → 批量提交 → 监控进度 → 结果归档 → 用户反馈 → 失败重试”标准化。

### 本 Skill 的核心价值

1. **从“人驱动”变成“规程驱动”** — 现在每一步都要人发指令，Skill 把执行链路固化为 Phase 0-7，智能体拿到触发词就能走完整个流程，减少遗漏和临场发挥。
2. **主动反馈而非被动等待** — 最低每 30 秒或每 5 个文件完成就反馈一次，这是目前最大的痛点。批量转写跑几十分钟，用户不问就没信息，体验很差。
3. **跨平台一致性** — 把平台差异抽到适配层，核心规程只写一次，Codex、Claude Code、OpenClaw、Hermes 都能执行。

### 关键挑战与风险

| 挑战 | 分析 | 应对策略 |
|------|------|----------|
| 进度监控的平台差异 | 不同智能体平台的“等待+轮询”能力差异很大——Claude Code 可以用循环自 pacing，Codex 的循环执行能力有限。“至少每 30 秒反馈”在有些平台上可能需要拆成多次用户交互才能实现 | Phase 5 的反馈模板需要在平台适配层 (`platform-adapters.md`) 里具体说明每个平台怎么做到 |
| manifest 管理成本 | 每次扫描建清单、指纹去重、失败重试建新批次——这些逻辑虽然不复杂，但写进 SKILL.md 后，每个平台都得忠实地实现，否则跨平台一致性就名存实亡 | 先在 Claude Code 上跑通一个完整流程（作为参考实现），再逐步推广到其他平台 |
| 大批量场景的可靠性 | 50+ 文件的分块提交、多 group 聚合监控、断点续传，是 channel-intake 不具备的能力，也是本 Skill 的核心差异化价值 | 验收标准应增加 100+ 文件的大批量场景 |

### 与 channel-intake 的定位差异

本 Skill 与 `funasr-task-manager-channel-intake` 是两套独立的 Skill，分别服务不同场景：

| 维度 | channel-intake（聊天渠道入口） | local-batch-transcribe（服务器本地入口） |
|------|-------------------------------|---------------------------------------|
| 触发时机 | 实时、事件驱动（用户发文件到对话框） | 按需或定时扫描（用户指定目录/inbox） |
| 文件规模 | 1~5 个（用户当场发的） | 10~500+ 个 |
| 反馈节奏 | 逐文件即时反馈 | 批次级进度报告 |
| 失败容忍 | 低（单文件失败要立刻告知） | 高（先跑完全批再汇总失败） |
| 用户预期 | 几分钟内拿到结果 | 可以等几十分钟到几小时 |
| 状态管理 | 无 manifest，即时处理 | manifest 驱动全流程，支持断点续传 |
| 结果输出 | 回渠道消息/文件附件 | 写本地文件系统 |
| 链路特征 | **同步短链路** | **异步长链路** |

两者共享的基础设施：`media-preflight`（预检）→ 后端 API（上传+创建任务+调度）→ `result-delivery`（结果格式化）。

`local-batch-transcribe` 在此之上新增：**manifest 管理** + **分块提交策略** + **异步进度监控循环** + **断点续传**。

## 二、设计目标

1. 支持用户指定任意服务器本地目录作为待处理目录。
2. 支持智能体固定扫描项目初始化时创建的专用中间工作目录。
3. 对所有待处理文件建立清单、状态、结果、失败原因和重试记录。
4. 批量提交给现有 `funasr-task-manager` 后端，复用已有调度、VAD 分段并行和结果合并能力。
5. 智能体必须主动反馈进度，不允许长时间静默执行。
6. Skill 文档必须能被 Codex、Claude Code、OpenClaw、Hermes 共同使用。
7. 默认中文交互，输出内容面向非工程用户也能理解。

非目标：

- 不重新实现 FunASR 转写引擎。
- 不替代后端任务调度器。
- 不要求用户必须通过聊天渠道上传文件。
- 不把平台私有能力写死到核心流程中。

## 三、初始化阶段新增目录

`funasr-task-manager-init` 应在环境初始化成功后创建专用的服务器本地批量转写中间工作目录。

默认目录：

```text
runtime/agent-local-batch/
├── inbox/          # 智能体固定扫描入口，用户可把待转写文件放入这里
├── processing/     # 当前批次的工作副本、锁文件和临时文件
├── outputs/        # 转写结果归档
├── manifests/      # 批次清单、状态快照、失败清单
├── logs/           # 智能体运行日志和进度事件
└── archive/        # 已完成批次的归档索引，不存放大文件副本
```

初始化规则：

| 项 | 规则 |
|----|------|
| 创建时机 | `funasr-task-manager-init` Phase 5 健康检查通过后 |
| 创建方式 | 幂等创建，目录已存在则保留 |
| 是否提交 Git | 目录结构可通过 `.gitkeep` 保留，运行时产物必须被 `.gitignore` 忽略 |
| 大文件策略 | `inbox/` 允许放真实音视频文件，但不要纳入 Git |
| 权限策略 | 仅要求当前智能体运行用户可读写，不要求 root |

推荐补充 `.gitignore`：

```gitignore
runtime/agent-local-batch/inbox/*
runtime/agent-local-batch/processing/*
runtime/agent-local-batch/outputs/*
runtime/agent-local-batch/manifests/*
runtime/agent-local-batch/logs/*
runtime/agent-local-batch/archive/*
!runtime/agent-local-batch/**/.gitkeep
```

## 四、待处理目录来源

Skill 接受两类输入来源。

### 4.1 用户显式指定目录

用户可以说：

- “转写 `/data/meeting-audio/` 下面所有音频”
- “扫描 `D:\records\2026-05`”
- “只跑这个目录里的 mp3 和 mp4”
- “把失败的再跑一遍”

智能体必须先确认目录存在、可读，并输出将要扫描的根目录。

### 4.2 固定扫描目录

当用户没有指定目录，但表达了“服务器本地批量转写”“看看有没有待处理文件”“自动处理待转写目录”时，智能体默认扫描：

```text
runtime/agent-local-batch/inbox/
```

这个目录是面向人和智能体的约定入口。用户只要把音视频文件放入该目录，智能体即可自动发现。

## 五、Skill 触发条件

### 自动触发

- 用户要求批量转写服务器本地文件。
- 用户要求扫描本地目录、待处理目录、inbox 或 workdir。
- 用户要求继续上次批量转写。
- 用户要求重试失败项。

### 关键词

`本地批量转写` / `服务器本地文件` / `扫描目录` / `待处理目录` / `inbox` / `批量识别` / `继续跑` / `重试失败` / `local batch transcribe`

## 六、核心执行流程

### Phase 0：运行上下文检查

1. 确认当前目录是 `funasr-task-manager` 仓库。
2. 检查后端健康状态：`GET http://127.0.0.1:15797/health`。
3. 后端不可达时，先进入 `funasr-task-manager-init` 启动流程。
4. 确认 `runtime/agent-local-batch/` 目录存在；不存在则创建并提示“检测到初始化目录缺失，已补建”。

### Phase 1：确定扫描来源

优先级：

1. 用户本轮明确指定的目录或文件。
2. 用户指定的上次批次 ID。
3. 默认固定扫描目录 `runtime/agent-local-batch/inbox/`。

如果默认扫描目录为空，智能体应直接反馈：

```text
未发现待转写文件。请把音视频文件放入 runtime/agent-local-batch/inbox/，或告诉我具体目录。
```

### Phase 2：扫描与建清单

扫描规则：

| 项 | 规则 |
|----|------|
| 递归 | 默认递归扫描子目录 |
| 支持格式 | `.wav` `.mp3` `.mp4` `.flac` `.ogg` `.webm` `.m4a` `.aac` `.wma` `.mkv` `.avi` `.mov` `.pcm` |
| 跳过隐藏文件 | 默认跳过以 `.` 开头的文件和系统临时文件 |
| 跳过未完成文件 | 文件大小或 mtime 在连续两次扫描中变化时，暂不提交 |
| 去重依据 | 绝对路径 + 文件大小 + mtime + 可选 SHA256 |
| 已完成跳过 | manifest 中标记 `SUCCEEDED` 且文件指纹未变化时跳过 |

每次运行创建一个批次清单：

```text
runtime/agent-local-batch/manifests/batch-YYYYMMDD-HHMMSS.json
```

清单字段建议：


#### Manifest item 状态机

与后端任务状态流转对齐：

```
后端状态：PENDING → PREPROCESSING → QUEUED → DISPATCHED → TRANSCRIBING → SUCCEEDED/FAILED
```

Manifest item 的状态定义：

| 状态 | 含义 | 来源 |
|------|------|------|
| `LOCAL_PENDING` | 扫描到文件，未提交 | 扫描阶段设置 |
| `SKIPPED` | 文件已完成或不符合格式，跳过 | 去重/预检查阶段设置 |
| `FAILED_PRECHECK` | 预检查失败（不可重试） | 预检查阶段设置 |
| `UPLOADING` | 正在上传 | 提交阶段设置 |
| `FAILED_UPLOAD` | 上传失败（可重试 1 次） | 提交阶段设置 |
| `SUBMITTED` | 已提交到后端（有 task_id） | 提交成功后设置 |
| `BACKEND_PROCESSING` | 后端处理中 | 监控阶段从 API 同步 |
| `SUCCEEDED` | 转写成功，结果已下载 | 结果下载成功后设置 |
| `FAILED_TRANSCRIBE` | 后端转写失败 | 监控阶段从 API 同步 |
| `FAILED_DOWNLOAD` | 结果下载失败 | 结果收集阶段设置 |
| `TIMEOUT` | 轮询超时 | 监控超时后设置 |

`backend_status` 字段额外记录后端原始状态（PREPROCESSING/QUEUED/DISPATCHED/TRANSCRIBING 等），方便排查。

#### Manifest 批次级状态

| 批次 status | 含义 |
|-------------|------|
| `SCANNED` | 扫描完成，未开始提交 |
| `SUBMITTING` | 正在分 chunk 提交 |
| `MONITORING` | 全部提交完成，监控中 |
| `COMPLETED` | 全部处理完成（含部分失败） |
| `INTERRUPTED` | Agent session 中断，待恢复 |
| `FAILED` | 整批失败（如后端不可达） |

### Phase 3：媒体预检查

对候选文件调用 `funasr-task-manager-media-preflight` 逻辑：

1. 检查文件存在、大小、格式。
2. 使用 `ffprobe` 获取时长、编码、采样率、声道。
3. 标记是否需要转码。
4. 估算总耗时（基于 RTF baseline 和在线服务器数）。

**预估总耗时的计算模型**：

复用 `funasr-task-manager-server-benchmark` Skill 产出的 RTF 数据：

```
estimated_total_seconds = sum(file.duration_sec for file in batch_items) * rtf_baseline
estimated_wall_clock = estimated_total_seconds / num_online_servers / avg_concurrency
```

如果无 benchmark 数据可用，使用默认 RTF = 0.1（即 1 小时音频约 6 分钟处理完）。

扫描完成后的反馈中应包含预估：

```text
扫描完成：42 个文件，总音频时长 12.5 小时。
基于当前 3 台在线服务器（RTF ≈ 0.08），预计处理时间约 20 分钟。
```

预检查失败的文件不提交，状态设为 `SKIPPED` 或 `FAILED_PRECHECK`，并写入错误原因。

### Phase 4：批量提交

#### 设计原则：以 CLI 为执行骨架

Skill 的"提交"步骤应优先调用已验证的 CLI 命令，而不是让 Agent 自己拼 API 请求。CLI 已经处理好了上传失败重试、文件名映射、task_group_id 追踪、进度输出等细节。采用**两阶段调用模式**：

```
阶段 A（提交）：python -m cli transcribe <files...> --batch --no-wait --json-summary
  → 拿到 task_group_id 和 task_ids → 写入 manifest

阶段 B（监控）：python -m cli task wait --group <group_id> --timeout 3600
  或者：直接 GET /api/v1/task-groups/{id} 轮询 → 解析进度并向用户反馈
```

#### 具体命令模式

**场景 1：小批量一步到位（≤50 文件）**

```bash
cd 3-dev/src/backend
python -m cli transcribe \
  /data/audio1.mp3 /data/audio2.wav /data/audio3.mp4 \
  --batch \
  --language auto \
  --format txt \
  --output-dir ../../../runtime/agent-local-batch/outputs/<batch_id> \
  --poll-interval 10 \
  --timeout 3600
```

Agent 解析 CLI stdout 的实时输出作为进度反馈来源。

**场景 2：大批量分离提交+监控（>50 文件）**

```bash
# Step 1: 提交第 N 个 chunk（不等待）
python -m cli transcribe \
  /data/chunk_n/*.mp3 \
  --batch --no-wait --json-summary \
  --language auto \
  --output-dir ../../../runtime/agent-local-batch/outputs/<batch_id>
# → 输出 JSON: {"task_group_id": "...", "task_ids": [...], "task_count": N}

# Step 2: 监控（对每个 chunk 的 group 轮询）
python -m cli task wait --group <task_group_id> --timeout 3600

# Step 3: 下载结果（如 Step 1 未带 --download）
python -m cli task results --group <task_group_id> \
  --format txt \
  --output-dir ../../../runtime/agent-local-batch/outputs/<batch_id>
```

#### 分块提交策略

| 文件数 | 策略 |
|--------|------|
| 1-50 | 单批提交，CLI 阻塞等待 |
| 51-300 | 每 50 个文件一批，分别用 `--no-wait --json-summary` 提交，然后统一监控 |
| 300+ | 同上，但提交前必须向用户报告总量和预计分批数量，等待确认 |

**分块提交与后端 task_group 的映射关系**：每个 chunk 调用一次 `POST /api/v1/tasks` 会创建一个独立的 `task_group`。manifest 应记录 chunk → group 的映射：

```json
{
  "chunks": [
    {"chunk_index": 0, "task_group_id": "grp_001", "file_count": 50, "status": "SUBMITTED"},
    {"chunk_index": 1, "task_group_id": "grp_002", "file_count": 50, "status": "SUBMITTED"},
    {"chunk_index": 2, "task_group_id": "grp_003", "file_count": 12, "status": "SUBMITTED"}
  ],
  "total_files": 112,
  "total_groups": 3
}
```

进度汇总时聚合所有 group 的完成情况。

#### 提交后必须记录

- 后端 `task_group_id`（每个 chunk 一个）
- 每个文件对应的 `task_id`
- 输出目录
- 提交时间
- chunk 索引（大批量时）

#### HTTP API 备选

如果平台更适合直接调用 HTTP API（如无法执行 CLI），也允许使用：

1. `POST /api/v1/files/upload` — 上传文件，拿到 `file_id`
2. `POST /api/v1/tasks` — 创建转写任务（body: `{"items": [...], "segment_level": "10m"}`）
3. `GET /api/v1/task-groups/{id}` — 查询批次状态
4. `GET /api/v1/task-groups/{id}/results` — 批量下载结果

但核心 manifest 格式保持一致，CLI 优先。

### Phase 5：进度监控与反馈

智能体必须主动反馈，不允许等用户追问。

#### 三层监控架构

根据批量规模和平台能力，选择合适的监控层：

```
┌────────────────────────────────────────────────────────────┐
│  Layer 1：CLI 阻塞等待（适合 ≤50 文件的快速批次）          │
│  python -m cli transcribe ... --batch --poll-interval 10   │
│  → CLI 内部轮询，Agent 解析 stdout 实时输出               │
│  → 适用于所有支持 shell 命令执行的平台                    │
└────────────────────────────────────────────────────────────┘

┌────────────────────────────────────────────────────────────┐
│  Layer 2：提交分离式（适合 >50 文件的大批量）              │
│  Step 1: --no-wait --json-summary → 拿 group_id           │
│  Step 2: Agent 循环调用 GET /api/v1/task-groups/{id}      │
│          每 15-30s 一次，解析 progress 字段               │
│  → Agent 主动向用户汇报                                   │
│  → 适用于需要跨 chunk 聚合进度的场景                      │
└────────────────────────────────────────────────────────────┘

┌────────────────────────────────────────────────────────────┐
│  Layer 3：SSE 实时监听（适合需要秒级反馈的单任务场景）     │
│  GET /api/v1/tasks/{id}/progress（SSE）                    │
│  → 仅对单任务有效，批量场景不推荐                         │
│  → 适用于用户特别关注某个大文件的处理进展                 │
└────────────────────────────────────────────────────────────┘
```

#### 通用进度反馈协议（平台无关）

Skill 规定 Agent 必须执行的"轮询-汇报"循环伪代码：

```
while batch_not_complete:
    status = query_progress()    ← API 或 CLI
    if status_changed or elapsed > 30s:
        report_to_user(status)   ← 平台适配层
        update_manifest(status)  ← 写入本地 manifest
    sleep(poll_interval)         ← 10~30s
```

`query_progress()` 的实现方式：
- **选项 A**：`GET /api/v1/task-groups/{id}`（推荐，REST 无状态）
- **选项 B**：`python -m cli task list --group {id} --json`（适合 CLI 优先平台）
- **选项 C**：读取 `task wait` 命令的 stdout 增量输出（适合阻塞模式）

#### 平台适配指引

> 如果平台支持后台命令执行并捕获 stdout（如 Claude Code 的 shell tool、Codex 的 sandbox），优先使用 CLI 阻塞模式 (Layer 1)，由 Agent 定期读取 stdout 新增内容作为进度。
>
> 如果平台只能逐条执行命令（如某些受限 Agent），则使用 API 轮询模式 (Layer 2)：每次执行一条 curl/httpie 命令查询状态，解析后向用户汇报。
>
> 详细的各平台实现方案见 `references/platform-adapters.md`。

#### 最低反馈频率

| 阶段 | 反馈要求 |
|------|----------|
| 扫描完成 | 立即反馈总文件数、待提交数、跳过数 |
| 提交完成 | 立即反馈批次 ID、任务数、预计耗时 |
| 转写中 | 至少每 30 秒反馈一次，或每完成 5 个文件反馈一次，取更早者 |
| 异常 | 立即反馈错误原因和下一步选择 |
| 完成 | 立即反馈成功数、失败数、结果目录、失败清单 |

#### 进度反馈模板

```text
批量转写进度：18/42 已完成，2 个失败，22 个处理中。

最近完成：会议录音-03.mp3
```

长时间无状态变化时：

```text
转写仍在运行，目前没有新的完成项。后端状态正常，当前仍有 6 个任务处于 TRANSCRIBING。
```

#### 强制反馈检查清单

> **强制规则**：Agent 在执行批量转写监控时，必须逐条确认以下反馈已发送。学习 `channel-intake` 和 `result-delivery` 的"执行检查清单"机制，防止 Agent 静默跑完整个批次。

| # | 检查项 | 时机 | 示例 |
|---|--------|------|------|
| 1 | 扫描结果通知 | Phase 2 完成 | "共发现 42 个文件，36 个待转写" |
| 2 | 预估耗时通知 | Phase 3 完成 | "基于当前 RTF，预计 20 分钟完成" |
| 3 | 提交确认 | Phase 4 每 chunk 完成 | "已提交第 1/3 批（50 个文件），批次 ID: xxx" |

| 6 | 完成汇总通知 | Phase 6 完成 | "批量转写完成：34 成功，2 失败" |

### Phase 6：结果归档

结果目录：

```text
runtime/agent-local-batch/outputs/<batch_id>/
├── txt/
├── json/
├── srt/
└── summary-<batch_id>.md
```

默认至少保存 `txt` 结果。用户要求字幕或结构化结果时再保存 `srt` 或 `json`。

批次汇总文件命名：

```text

```

因为它是生成的 Markdown 报告，必须满足项目规范，文件名以 `-YYYYMMDD.md` 结尾。

### Phase 7：失败处理与重试

失败分类：

| 类型 | 状态 | 处理 |
|------|------|------|
| 格式不支持 | `FAILED_PRECHECK` | 不自动重试，提示转换格式 |
| 文件读取失败 | `FAILED_IO` | 检查权限或路径 |
| 上传失败 | `FAILED_UPLOAD` | 可自动重试 1 次 |
| 后端任务失败 | `FAILED_TRANSCRIBE` | 可按用户确认重试 |
| 下载结果失败 | `FAILED_DOWNLOAD` | 可自动重试下载 |
| 超时 | `TIMEOUT` | 查询后端任务状态后决定继续等待或重试 |

用户说“重试失败项”时，智能体读取最近一个 manifest，筛选可重试状态，并创建新批次：

```text
retry-of-local-20260503-142015-YYYYMMDD-HHMMSS
```

不得覆盖原 manifest。

#### 断点续传与中断恢复

Agent session 可能因超时、网络中断或平台限制而中断。manifest 必须支持从中断点恢复：

**中断时的 manifest 状态**：

```json
{
  "batch_id": "local-20260503-142015",
  "status": "INTERRUPTED",
  "resume_from": {
    "last_submitted_index": 35,
    "last_monitored_groups": ["grp_001", "grp_002"],
    "interrupted_at": "2026-05-03T15:30:00+08:00",
    "reason": "agent_session_timeout"
  }
}
```

**恢复行为**：当新 session 的 Agent 启动时扫描 `manifests/` 目录，发现 `INTERRUPTED` 或 `SUBMITTING`/`MONITORING` 状态的批次，应主动提示：

```text
发现一个未完成的批量转写批次 (local-20260503-142015)：
  已提交 35/112 个文件，其中 28 已完成，7 仍在运行。
  是否继续？
```

**恢复策略**：

| 中断点 | 恢复动作 |
|--------|----------|
| 提交阶段中断（部分 chunk 未提交） | 从 `last_submitted_index` 继续提交剩余文件 |
| 监控阶段中断（任务仍在后端运行） | 查询所有已提交 group 的最新状态，继续监控 |
| 下载阶段中断（部分结果未下载） | 检查 `output_path` 为空的 SUCCEEDED 任务，补下载 |

### 与 channel-intake 的协作契约

`local-batch-transcribe` 与 `channel-intake` 共享同一个后端调度器和 FunASR 服务器资源。当两者同时活跃时，需要一套**优雅让步**机制，而非简单的中断或抢占。

#### 核心原则：Batch 让步、Intake 优先、不中断已运行任务

```
┌─────────────────────────────────────────────────────────────────────┐
│  正常状态：batch 独占调度                                              │
│  ┌─────────────────────────────────────────────────────────────┐    │
│  │  batch queue: [task_18] [task_19] [task_20] [task_21] ...   │    │
│  │               ^^^^^^^^  ^^^^^^^^  ~~~~~~~~  ~~~~~~~~        │    │
│  │               运行中     运行中    等待调度   等待调度        │    │
│  └─────────────────────────────────────────────────────────────┘    │
│                                                                      │
│  intake 任务到达 → 触发让步                                           │
│                                                                      │
│  ┌─────────────────────────────────────────────────────────────┐    │
│  │  batch queue: [task_18] [task_19] │ [task_20] [task_21]     │    │
│  │               继续运行   继续运行  │  暂停调度  暂停调度      │    │
│  │                                   │                         │    │
│  │  ─── 让步点 ──────────────────────┘                         │    │
│  │                                                              │    │
│  │  intake queue: [intake_A] [intake_B]  ← 插队处理             │    │
│  │                ^^^^^^^^^^  ^^^^^^^^^^                         │    │
│  │                优先启动     优先启动                           │    │
│  └─────────────────────────────────────────────────────────────┘    │
│                                                                      │
│  intake 全部完成 → batch 恢复调度                                     │
│                                                                      │
│  ┌─────────────────────────────────────────────────────────────┐    │
│  │  batch queue: [task_20] [task_21] [task_22] ...             │    │
│  │               恢复运行   恢复运行   等待调度                  │    │
│  └─────────────────────────────────────────────────────────────┘    │
└─────────────────────────────────────────────────────────────────────┘
```

#### 让步行为详细规则

| 阶段 | batch 行为 | intake 行为 | 用户通知 |
|------|-----------|------------|---------|
| intake 任务到达时 | 已运行的 batch 任务继续，**暂停新的 batch 任务调度启动** | 正常提交并获得优先调度 | 通知 batch 用户："收到实时转写请求，batch 队列暂停新任务启动，当前运行中的任务不受影响，预计 {N} 分钟后恢复" |
| intake 处理期间 | batch 队列中的等待任务**只更新 manifest 状态，不启动新的转写** | 正常执行全流程 | 无需重复通知（除非 intake 处理时间超出预期） |
| intake 全部完成后 | batch 恢复正常调度，继续启动等待中的任务 | 交接完成 | 通知 batch 用户："实时任务已完成，batch 队列恢复，继续处理剩余 {M} 个文件" |

#### 不中断原则

1. **已经在 FunASR 节点上运行的 batch 任务绝不中断** — 中断会浪费已计算的中间结果。
2. **只暂停"从 QUEUED 到 DISPATCHED"的调度决策** — 即 batch 任务在后端队列中等待，但调度器暂时不把它分配给服务器节点。
3. **intake 任务到达时，让当前 batch 任务完成到一个"让步点"** — 让步点是指当前正在运行的 batch 任务全部完成（或达到一个 chunk 边界），此后的资源优先给 intake。

#### 用户通知模板

**batch 暂停时（通知 batch 用户）**：

```text
⏸️ 收到实时转写请求（{intake_file_count} 个文件），batch 队列暂停新任务启动。
当前正在运行的 {running_count} 个 batch 任务不受影响，预计 {eta} 后完成。
完成后将优先处理实时请求，然后恢复 batch 队列。
```

**batch 恢复时（通知 batch 用户）**：

```text
▶️ 实时转写已完成，batch 队列恢复。
继续处理剩余 {remaining_count} 个文件，当前进度 {completed}/{total}。
```

**intake 用户的无感体验**：

intake 用户无需知道 batch 的存在。对 intake 用户而言，系统响应速度应与没有 batch 任务时一致。如果因 batch 任务占用节点导致 intake 需要短暂排队，应在 intake 的进度通知中体现：

```text
⏳ 转写排队中，预计 {wait_sec}s 后开始（当前有 {running_batch} 个后台任务即将完成释放资源）。
```

#### 实现机制

本协作契约在 **Agent 调度层** 实现，而非修改后端调度器代码：

1. **检测 intake 到达**：batch Skill 在 Phase 5 轮询循环中，每次查询进度时同时检查 `GET /api/v1/stats` 中的 `queue_depth` 和任务来源标记。如果检测到有非本 batch 的新任务进入（特别是 intake 标记的任务），进入让步模式。
2. **暂停 batch 调度**：batch Skill 停止提交新的 chunk，或者对已提交但未启动的任务不再催促调度。已 DISPATCHED/TRANSCRIBING 的任务自然运行完毕。
3. **恢复条件**：当 intake 任务全部完成（`queue_depth` 恢复或 intake 相关 task 全部 terminal），batch Skill 恢复正常的 chunk 提交和调度。

#### 边界场景处理

| 场景 | 处理方式 |
|------|---------|
| batch 正在上传文件时 intake 到达 | 完成当前 chunk 的上传，但不提交下一个 chunk |
| intake 处理时间超过 30 分钟 | 向 batch 用户发送更新："实时任务仍在处理中（{intake_elapsed}），预计 {eta} 恢复" |
| 多个 intake 请求连续到达 | 全部处理完毕后才恢复 batch |
| batch 用户主动要求恢复 | 尊重用户指令，恢复 batch 并行（此时 intake 和 batch 共享资源） |
| intake 和 batch 是同一用户 | 同样执行让步机制，但通知合并："你的实时请求已插队，batch 暂停" |

## 十、安全与边界规则

1. 默认不删除用户源文件。
2. 默认不移动用户源文件；如需归档，只记录 manifest，不搬动原始数据。
3. 不把大音视频文件复制到 Git 管理目录之外的未知位置。
4. 不把转写全文直接刷屏输出；长结果保存为文件并给出路径。
5. 不在项目根目录生成散落的 `transcript_*.txt`、`result_*.json`。
6. 失败重试必须创建新批次，不覆盖历史批次。
7. manifest 和日志只记录必要路径、状态、错误信息；如涉及敏感文件名，按用户要求脱敏。

```
└───────────────────────────────┬─────────────────────────────────┘
                                │
                                ▼
```

## 十三、验收标准

| 场景 | 期望结果 |
|------|----------|
| 默认扫描空 inbox | 明确提示未发现待处理文件，不报错 |
| inbox 中有 3 个音频 | 自动扫描、预检查、批量提交、反馈进度、保存结果 |
| 用户指定外部目录 | 使用用户目录作为 source，结果仍写入 `runtime/agent-local-batch/outputs/` |
| 已完成文件再次扫描 | 指纹未变化时跳过，不重复转写 |
| 批量任务中部分失败 | 完成后给出失败清单和可重试建议 |
| 用户说“重试失败项” | 读取最近 manifest，只提交失败项，创建 retry 批次 |
| Codex / Claude Code / OpenClaw / Hermes | 均能按同一核心流程执行，平台差异只影响安装路径和命令执行细节 |
| 100+ 文件大批量场景 | 自动分 chunk 提交，聚合多 group 进度，定期反馈，全部完成后汇总 |
| Agent session 中断后恢复 | 新 session 检测到 INTERRUPTED manifest，提示用户并可从断点继续 |
| 后端队列已有大量任务时 | 检测到 queue_depth 高时主动告知用户预计排队时间 |
