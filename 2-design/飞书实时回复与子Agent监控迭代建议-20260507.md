## 1. 结论摘要

这次两个 case 的核心问题相同：Agent 把普通 assistant 文本当成了飞书实时消息，但 OpenClaw 等聊天桥接平台会把普通 assistant 文本按 turn 缓冲，直到本轮工具调用链结束后才统一投递。因此，单文件和批量任务都会出现“过程阶段看似逐条输出，用户实际最后一并收到”的现象。

需要把“通知用户”从提示词约束升级为可验证的运行时协议：

1. 单文件链路的阶段进度必须调用 `message` tool 或 `cli notify`，不能只输出普通文本。
2. 批量链路的子 Agent 必须用 OpenClaw 官方 sub-agent 启动方式启动，并显式传入飞书会话上下文：`chat_id`、`message_id`、`reply_to_id`、`sender_id/ou 用户编码`、`bot_open_id`、群聊标识。
3. 子 Agent 启动后必须先发“监控已启动”确认消息；主 Agent 没收到确认时要回退到主 Agent 自行轮询或 CLI 通知。
4. 当前 OpenClaw 部署环境的 Skills/CLI 版本没有同步到最新仓库，运行记录里 `python -m cli notify --help` 返回 `NO_CLI_NOTIFY`。这会导致兜底通知能力缺失，必须作为 P0 部署问题处理。
5. **前置阻塞发现**：JSONL 运行记录证实**当前部署配置未向主 Agent 暴露 `message` tool**（Agent 思考过程明确写出 "I don't have a `message` tool available in my tool list"）。OpenClaw 体系中 `message` 是独立能力，需通过 tool policy 显式开启。这意味着适配器优先级 1 当前不可用，需运维调整配置。同时 CLI fallback（`cli notify`）因部署未同步也不可用，是真正的 P0 阻塞项。
6. 私聊（DM）场景与群聊共享同一根因（turn 缓冲），但路由方式不同（`open_id` vs `chat_id`），需独立分析（见 §5）。

## 2. 本地证据

### 2.1 运行上下文已包含飞书路由信息

`groupchat-session-be6b85df.jsonl` 中 OpenClaw runtime context 已注入以下信息：

```json
{
  "chat_id": "chat:oc_7aabd157ea9f5076bf5e695ffe638774",
  "message_id": "om_x100b50f596388480b3ef12df19f8329",
  "reply_to_id": "om_x100b50ae62371ca8b320c2fdf53a60b",
  "sender_id": "ou_b3100d4d8b404252c9fd0c965c445074",

  "sender": "王敏乐",
  "is_group_chat": true,
  "was_mentioned": true,
  "has_reply_context": true

```

这说明 OpenClaw 已经知道应该向哪个飞书群聊、哪条消息、哪个用户上下文回复。问题不是没有路由信息，而是主 Agent 没有把这些信息作为通知上下文传给实时消息工具或子 Agent。

### 2.2 单文件流程只输出了普通文本

同一运行记录里，Agent 在多个时间点生成了阶段文本：

| 时间 | Agent 文本 |
|------|------------|
| 13:05:40 | 收到了，来查一下这个文件的详情。 |
| 13:06:22 | 好的，找到飞书凭证了。先获取 token 和文件信息。 |
| 13:06:32 | 文件信息：`tv-report-1.wav`，开始下载。 |
| 13:06:37 | 文件下载完成，开始预检。 |

但这些记录都是 assistant `TEXT`，不是 `message` tool。工具调用序列中只有 `read`、`exec`、`memory_search`、`process` 等，没有 `message` tool 调用。因此这些阶段文本只能作为本轮最终回复的一部分被 OpenClaw bridge 投递，不能保证实时到达飞书。

### 2.3 兜底 CLI 在部署环境中不可用

运行记录里 Agent 检查通知 CLI：

```text
cd .../funasr-task-manager/3-dev/src/backend
python -m cli notify --help 2>/dev/null || echo "NO_CLI_NOTIFY"
```

返回：

```text
NO_CLI_NOTIFY
```

而本仓库当前已实现 `3-dev/src/backend/cli/commands/notify.py`，README 也已经写入 `python -m cli notify send --text "..."`。这说明 OpenClaw 实际运行的工作区或安装包不是最新版本。即使 Skill 写了 CLI fallback，部署环境也无法执行。

### 2.4 最后结果附件是手写飞书 API 发送的

由于没有 `message` tool，也没有 `cli notify`，Agent 最终直接用飞书 API 上传并发送文件。这证明飞书 token、chat id 和发送能力在环境里存在，但现在缺少统一封装，导致 Agent 临场拼 shell 和 curl，风险包括：

- token 暴露在命令历史或日志里；
- `chat_id`、`reply_to` 处理不一致；
- 文本、文件、线程回复逻辑分散；
- 无法统一重试、去重、审计和失败降级。

### 2.5 `message` tool 不在 Agent 工具列表中

这是最关键的前置发现。JSONL 第 50 行 Agent 思考过程中明确写出：

> "Since I don't have a `message` tool available in my tool list, I'll need to use the Feishu API directly..."

整个 session 的工具调用只涉及 3 种 toolName：

| toolName | 调用次数 | 用途 |
|----------|---------|------|
| `read` | 4 次 | 读 Skill 文档和结果交付规范 |
| `exec` | ~20 次 | 执行 shell 命令（curl、ffprobe、python 等） |
| `memory_search` | 1 次 | 搜索历史飞书凭据 |

**零次 `message` tool 调用**——不是模型"忘了调用"，而是工具列表中根本没有 `message` tool。

这意味着：

- `CHANNEL-NOTIFICATION.md` 中适配器优先级 1（`message` tool）在当前部署中不可用。
- 适配器优先级 2（`cli notify`）因部署未同步也不可用（§2.3）。
- Agent 被迫退化到手写 curl，等同于无适配器兜底。
- **在 OpenClaw 暴露 `message` tool 之前，`cli notify` 是唯一可行的实时通知路径。**

**🔴 阻塞**（§12 阻塞项 1）：**2026-05-07 实际诊断已确认**——当前 tools profile 为 `coding`，不含 `group:messaging`，主 Agent 和子 Agent 均无 `message` tool。修复：`openclaw config set tools.alsoAllow '["message"]'` + `tools.subagents.tools.allow '["message"]'`。

### 2.6 凭据明文暴露

JSONL 中发现以下敏感信息明文泄露：

| 敏感信息 | 出现位置 | 暴露次数 |
|----------|---------|---------|
| `tenant_access_token`（`t-g10457co...`） | `exec` 命令参数和返回值 | 10+ 次 |
| `appSecret`（`***REDACTED***`） | OpenClaw feishu channel 配置输出 | 1 次 |
| `appId`（`***REDACTED***`） | 同上 | 1 次 |

`appSecret` 的暴露尤其严重——它是飞书应用的长期凭据，不像 `tenant_access_token`（2h 有效）那样自动过期。Agent 通过 `cat ~/.openclaw/openclaw.json` 查看 feishu channel 配置时，把 `appSecret` 原样输出到了 session 记录中。**应视为已泄露。**

安全对策（**🔴 最高优先级**）：

- **已暴露的 `appSecret` 必须立即轮换**（见 §12 阻塞项 3）。
- `cli notify` 封装后，Agent 不需要直接接触 `appId`/`appSecret`，凭据管理在 CLI/部署层完成。
- **禁止 Agent 直接 `cat` OpenClaw 配置文件**。需要凭据状态时使用 `cli notify auth-check` 只返回 `authenticated: true/false`（不含密钥）。
- CLI 日志脱敏——不输出 `appSecret` 和 `tenant_access_token`。
- 轮换后的新 secret 不写入任何文档或 transcript。

## 3. Case 1：单文件群聊回复不实时

### 3.1 根因

单文件流程天然被拆成了多个短工具调用，因此 Agent 有机会在每步之间发送实时消息。但实际执行中，Agent 只是输出普通 assistant 文本，没有调用 `send_user_notice()` 的真实实现。

现有 `channel-intake`、`result-delivery`、`CHANNEL-NOTIFICATION.md` 已写明“必须调用 `send_user_notice()`”，但这仍是 prompt 级约束。只要模型没有实际调用 `message` tool 或 CLI/API，平台就无法把它当成实时通知。

### 3.2 修复原则

单文件链路必须变成以下硬时序：

```text
收到请求
  -> message tool: 收到文件，开始读取文件信息
  -> 获取飞书文件信息
  -> message tool: 文件信息确认，开始下载
  -> 下载文件
  -> message tool: 下载完成，开始预检
  -> ffprobe / 预检
  -> message tool: 预检通过，开始上传
  -> 上传到后端
  -> message tool: 任务已提交，开始转写
  -> 轮询或 SSE 监听
  -> message tool: 状态变化 / 完成 / 失败
  -> message tool: 发送结果摘要 + 附件
```

关键点：

- 普通 assistant 文本只用于最终总结，不计入“阶段通知已发送”。
- 每条阶段通知必须等工具返回成功或明确失败后再继续下一步。
- `message` tool 不可用时才使用 `cli notify`。
- `cli notify` 不可用时，必须在最终总结中标注“实时通知通道不可用”，并把这次运行计入缺陷统计。

### 3.3 建议改动

1. 在 `channel-intake` 增加“通知能力预检”步骤：
   - 检查当前 tool list 是否有 `message` tool；
   - 检查 `python -m cli notify --help` 是否存在；
   - 检查 CLI fallback 所需的 `FEISHU_CHAT_ID` 或 `FEISHU_REPLY_TO`；
   - 预检结果写入本轮自检。
2. 在 `result-delivery` 增加“状态通知实际发送记录”：
   - `notice_seq`
   - `notice_text`
   - `adapter`：`message_tool` / `cli_notify` / `assistant_text_fallback`
   - `delivered`：true / false
   - `message_id`
3. 在单文件任务开始时生成本轮 `notification_context`，后续所有阶段复用。
4. 将手写飞书 curl 禁止为默认路径，只允许在 `cli notify` 缺失且用户/部署方明确允许时作为临时应急。

## 4. Case 2：批量模式子 Agent 未执行监控发送

### 4.1 根因

当前 `local-batch-transcribe` 和 `batch-monitor` 文档描述了“主 Agent 调度 + 子 Agent 监控播报”，但委托协议只传了任务信息，没有传飞书渠道上下文，也没有定义 OpenClaw 官方 sub-agent 调用方式。

现有委托示例只有：

```yaml
task_group_ids: [...]
batch_id: ...
total_files: ...
poll_interval_sec: 30
timeout_sec: 3600
result_format: txt
output_dir: ...
```

这对监控任务本身够用，但对子 Agent 发送飞书消息不够。子 Agent 需要知道：

- 发送到哪个群聊：`chat_id`
- 回复到哪条用户消息或话题：`message_id` / `reply_to_id`
- 谁触发了任务：`sender_id` / `sender_open_id`
- 当前是否群聊：`is_group_chat`
- 是否要 @ 原用户：`sender_name` / `sender_id`
- 当前 runtime 是否给子 Agent 开放 `message` tool

### 4.2 推荐委托协议

主 Agent 启动子 Agent 时必须传完整结构：

```yaml
task:
  task_group_ids:
    - "01KR..."
  batch_id: "local-20260507-130000"
  total_files: 50
  poll_interval_sec: 30
  heartbeat_interval_sec: 60
  timeout_sec: 3600
  result_format: "txt"
  output_dir: "runtime/agent-local-batch/outputs/local-20260507-130000"

notification_context:
  channel: "feishu"
  chat_id: "oc_7aabd157ea9f5076bf5e695ffe638774"
  message_id: "om_x100b50f596388480b3ef12df19f8329"
  reply_to_id: "om_x100b50ae62371ca8b320c2fdf53a60b"
  sender_open_id: "ou_b3100d4d8b404252c9fd0c965c445074"
  sender_name: "王敏乐"
  bot_open_id: "ou_3b5c0d182d285d6956d9e459bb9d9eb6"
  conversation_label: "ASR测试"
  group_subject: "ASR测试"
  is_group_chat: true
  was_mentioned: true

notification_policy:
  adapter_priority:
    - "openclaw_message_tool"
    - "cli_notify"
  require_start_ack: true
  start_ack_timeout_sec: 5
  reply_in_thread: true
  mention_requester_on_error: true
  soft_fail_progress_notice: true
  strict_final_notice: true
```

### 4.3 子 Agent 启动握手

主 Agent 委托后不能直接释放，必须等待启动确认：

```text
主 Agent:
  1. 调用 OpenClaw sub-agent spawn / task 工具
  2. 将 task + notification_context + notification_policy 传入子 Agent
  3. 等待子 Agent 在 5 秒内发送“监控已启动”消息
  4. 收到 ack 后再向用户说明主 Agent 已释放
  5. 未收到 ack 时进入 fallback：主 Agent 自行轮询或 CLI notify
```

子 Agent 第一件事不是查任务状态，而是发送：

```text
批量转写监控已启动

批次：{batch_id}
任务组：{task_group_ids}
文件总数：{total_files}
轮询间隔：{poll_interval_sec}s
```

这条消息必须通过 `message` tool 或 CLI/API 发送。普通文本不算 ack。

### 4.4 子 Agent 工具权限

子 Agent 必须至少具备：

| 能力 | 用途 |
|------|------|
| `task-group status` | 查询任务组进度 |
| `task-group download` | 下载完成结果 |
| `message action=send` | 飞书实时进度通知 |
| `message filePath/path` 或 `cli notify send-file` | 发送结果附件 |
| sleep / timer | 控制轮询间隔 |

如果子 Agent 没有消息发送工具，它可以查状态，但不能承担“监控播报”职责，应立即失败并让主 Agent fallback。

### 4.5 OpenClaw sub-agent spawn 机制（🟡 已部分确认）

> **2026-05-07 实际诊断结论**：
> - ✅ `sessions_spawn` **已暴露**——`group:sessions` 在 `coding` profile 中，`sessions_spawn`、`sessions_yield`、`sessions_send` 等均可用。
> - ❌ 子 Agent **没有 `message` tool**——父 Agent 没有，子 Agent 自然也没有。修复需两步：(1) 主 Agent 加 `message`；(2) 子 Agent 显式 `tools.subagents.tools.allow '["message"]'`。
> - ⚠️ `maxSpawnDepth: 1`——不能再 spawn 孙子 Agent。
>
> 走路径 A 的前提是阻塞项 1（主 Agent `message`）和子 Agent `message` 配置均已解除。

**确认的 spawn 方式：`sessions_spawn` 工具**

官方文档（[Sub-agents](https://docs.openclaw.ai/subagents)、[Session tools](https://docs.openclaw.ai/concepts/session-tool/)）确认：

```json
{
  "name": "sessions_spawn",
  "arguments": {
    "task": "执行 batch-monitor 监控任务：\n- task_group_ids: [...]\n- batch_id: local-20260507-130000\n- notification_context: {chat_id: ..., sender_open_id: ...}\n...",
    "label": "batch-monitor-20260507",
    "thread": true
  }
}
```

**已确认的行为特性**：

| 特性 | 结论 | 来源 |
|------|------|------|
| 阻塞性 | **非阻塞**，立即返回 `{ status: "accepted", runId, childSessionKey }` | [Session tools](https://docs.openclaw.ai/concepts/session-tool/) |
| `message` tool 继承 | **条件可用**：`message` 不在默认限制列表中，但前提是父 Agent 已开放 `message`（当前诊断：父子均无 `message`，因 `coding` profile 不含 `group:messaging`；需 `tools.alsoAllow` 或切 `full` profile 后才可继承） | [Sub-agents — Tool policy](https://docs.openclaw.ai/subagents#tool-policy) |
| completion announce | **仅完成时**触发，不支持 mid-run announce | [Sub-agents — Announce](https://docs.openclaw.ai/subagents) |
| 中间进度通知 | 子 Agent 需**自行调用 `message` tool** | 推导 |
| 嵌套 spawn | 默认 `maxSpawnDepth: 1`，子 Agent 不能再 spawn | [Sub-agents — Nested](https://docs.openclaw.ai/subagents) |
| announce 可靠性 | **best-effort**，gateway 重启会丢失 pending announce | [Sub-agents](https://docs.openclaw.ai/subagents) |

**已知飞书限制**（[Issue #10773](https://github.com/openclaw/openclaw/issues/10773)）：子 Agent 在飞书话题线程中回复时，可能创建新消息而非在原话题内回复。但默认策略仍然是 **优先 reply 原消息线程**（与 §12.5 统一），以便将"收到→提交→进度→完成"收束到同一个话题。如果实测复现此 Bug，则子 Agent 退回发群主会话，并带 `batch_id` 标识。主 Agent 始终优先 reply 原消息。

**主 Agent 等待机制**：使用 `sessions_yield` 而非轮询：

> "`sessions_yield` intentionally ends the current turn so the next message can be the follow-up event you are waiting for."

**路径 B 和 C 保留为 Fallback**：

- 路径 B（session 内续写）：当 `sessions_spawn` 因 tool profile 限制不可用时使用。
- 路径 C（主 Agent 自行轮询 + `cli notify`）：当 `message` tool 也不可用时使用（已在 `local-batch-transcribe` Phase 5 fallback 中定义）。

## 5. Case 3：私聊单文件转写

> 证据来源：私聊截图（图 2），无对应 JSONL 运行记录。

### 5.1 场景描述

用户通过飞书私聊（DM）向机器人发送音频文件并要求转写。从图 2 截图可以看到，私聊中 Agent 同样经历了完整的处理链路（文件接收 → 下载 → 预检 → 上传 → 转写 → 结果交付），并包含任务摘要表格和结果附件发送。

### 5.2 与群聊的差异

| 维度 | 群聊（Case 1） | 私聊（Case 3） |
|------|---------------|---------------|
| 路由标识 | `chat_id`（群聊 ID） | `open_id`（用户 ID） |
| 飞书 API `receive_id_type` | `chat_id` | `open_id` |
| 回复方式 | 可回复原消息（thread reply） | 直接发消息或回复原消息 |
| @ 需求 | 可能需要 @ 触发用户 | 不需要 @ |
| `is_group_chat` | `true` | `false` |
| OpenClaw session key | `agent:main:feishu:group:...` | `agent:main:feishu:direct:...` |
| turn 缓冲行为 | 确认存在（图 1 实证） | **推测相同**（未有 JSONL 验证） |

### 5.3 根因推测

私聊与群聊共享相同的 OpenClaw bridge 投递机制，因此 turn 级缓冲行为预计一致。但私聊的用户感知可能稍好——因为私聊中只有用户和机器人两方，即使消息集中到达，用户也不会觉得"被插话打断"。

但核心问题不变：**在处理过程中，用户看不到中间进度**。对于较长音频（>5 分钟），用户会有"机器人是不是卡了"的焦虑。

### 5.4 私聊专属修复要点

1. **`notification_context` 适配**：私聊时 `is_group_chat = false`，`cli notify` 需使用 `open_id` 而非 `chat_id` 发送消息。
2. **`cli notify` 已支持**：现有 `notify.py` 的 `send` 命令支持 `--receive-id-type open_id` 参数（待验证）。
3. **模板调整**：私聊中不需要 @ 用户、不需要 `group_subject`，通知模板可以更简洁。
4. **回复策略**：私聊中建议直接发新消息而非回复原消息，避免消息链过长影响阅读。

## 6. 外部官方依据与本项目结论

### 6.1 OpenClaw

依据：

- [OpenClaw Sub-Agents](https://docs.openclaw.ai/tools/subagents)：sub-agent spawn 是非阻塞的，完成后会向 requester chat channel 宣告结果；文档也说明完成投递会解析 requester route。
- [OpenClaw Groups](https://docs.openclaw.ai/channels/groups)：OpenClaw 对群聊有 group policy、mention gating 和 group session；群聊上下文包含群聊、提及、路由等字段。
- OpenClaw groups 文档还说明群聊可受工具限制控制，且 group/channel tool restrictions 会叠加全局工具策略。

对本项目的结论：

- OpenClaw 的 completion announce 只能保证子 Agent 结束时的结果回传，不能替代中间进度播报。
- 飞书群聊里的可见阶段消息必须通过显式消息工具发出。
- 子 Agent 如果没有继承 requester route，主 Agent 必须显式传 `chat_id`、`message_id`、`reply_to_id` 和 `sender_id`。

### 6.2 飞书 / Lark Open Platform

依据：

- [飞书发送消息 API](https://open.feishu.cn/document/server-docs/im-v1/message/create)：发送群聊消息时需要目标接收 ID 和消息内容，实际 API 使用 `receive_id_type=chat_id`。
- [飞书回复消息 API](https://open.feishu.cn/document/server-docs/im-v1/message/reply)：回复已有消息时使用原消息 `message_id`。

对本项目的结论：

- `chat_id` 和 `message_id/reply_to_id` 是两类不同路由信息。群聊直接发消息要 `chat_id`，回复线程要原消息 ID。
- 子 Agent 不应猜测目标会话；必须由主 Agent 从 OpenClaw runtime context 中传入。
- `sender_id` / `ou_...` 不是群聊发送必需字段，但用于 @ 触发用户、权限审计、私聊兜底和错误定向通知。

### 6.3 Cursor Background Agents

依据：

- [Cursor Slack 文档](https://docs.cursor.com/slack)：Cursor 的 Slack 集成允许在 Slack 中通过 @Cursor 启动 Background Agents，并依赖 Slack 权限读取上下文、发布更新。

对本项目的结论：

- Cursor 的后台 Agent 与 Slack 绑定依赖平台集成和权限，不是普通子进程自然具备发消息能力。
- 类比到飞书，子 Agent 要向指定 channel/user 发消息，必须持有渠道工具和目标 channel/user ID。

### 6.4 Codex / OpenAI

依据：

- [Codex Cloud](https://developers.openai.com/codex/cloud)：Codex cloud 任务在独立云环境中后台运行，可以并行处理任务。
- [OpenAI Background mode](https://developers.openai.com/api/docs/guides/background)：长任务可后台启动，调用方通过轮询 response 状态获取进度。

对本项目的结论：

- 后台任务和主对话是解耦的。后台 worker 不天然知道应该向哪个聊天渠道发进度。
- 如果用 Codex/Codex-like 后台任务实现批量监控，必须额外提供一个通知通道，例如 webhook、MCP 工具、CLI notify 或飞书 API 凭据。

### 6.5 Claude Code

依据：

- [Claude Code Subagents](https://code.claude.com/docs/en/subagents)：subagent 有自己的上下文窗口、系统提示、工具权限和独立权限；适合隔离探索或执行特定任务。

对本项目的结论：

- Claude Code 的 subagent 模型强调独立上下文和工具权限，不能默认继承主对话里的飞书路由。
- 委托文本必须包含完整任务上下文和通知上下文，否则子 Agent 只能返回最终摘要，不能正确发送渠道进度。

### 6.6 Hermes

依据：

- [Hermes Agent 文档](https://hermes-agent.nousresearch.com/docs/)：Hermes 支持消息平台网关、定时任务和并行 subagent。
- [Hermes Subagent Delegation](https://hermes-agent.nousresearch.com/docs/user-guide/features/delegation)：`delegate_task` 会创建隔离上下文的 child agent；文档明确提醒子 Agent 没有父会话历史，父 Agent 必须传入子 Agent 所需的一切上下文；同时 leaf subagent 默认不能调用 `send_message`，且 `delegate_task` 是同步的，不是 durable background queue。

对本项目的结论：

- Hermes 的官方模型更明确地证明：子 Agent 不能靠“继承父上下文”获得飞书路由。
- 如果平台禁止子 Agent 发跨平台消息，就不能让该子 Agent承担“播报员”职责，只能让主 Agent 或 durable cron/background 任务持有消息工具。

### 6.7 OpenCode

依据：

- [OpenCode Agents](https://opencode.ai/docs/agents/)：OpenCode 区分 primary agents 和 subagents；subagents 可由 primary agent 自动或手动调用，并可配置 prompt、model 和权限。

对本项目的结论：

- OpenCode 也把子 Agent 视为带独立配置和权限的专用助手。
- 要让 OpenCode 子 Agent 发通知，需要在其 permissions/tools 中显式允许消息工具或外部 API 工具。

### 6.8 Factory Droid

依据：

- [Factory Custom Droids](https://factory.mintlify.app/cli/configuration/custom-droids)：custom droids 是可复用 subagents，拥有自己的 system prompt、model preference 和 tooling policy，并通过 Task tool 调用。
- [Factory CLI Overview](https://docs.factory.ai/cli/getting-started/overview)：Droid 有 Slack、Jira、Notion 等工程系统集成，但这些集成属于明确配置的工具能力。

对本项目的结论：

- Droid 的子 Agent 同样需要显式工具策略。没有消息工具或 Slack/飞书集成权限，子 Agent 不能凭空向 channel 发消息。
- “实时进度”应作为工具调用结果或平台集成事件，而不是子 Agent 的普通文本。

## 9. 风险与取舍

| 风险 | 影响 | 对策 |
|------|------|------|
| OpenClaw 子 Agent 无法继承 `message` tool | 子 Agent 不能播报 | 委托前验证工具权限；不满足则主 Agent fallback。 |
| **当前部署未向 Agent 暴露 `message` tool** | **适配器优先级 1 完全不可用** | **🔴 阻塞：需运维调整 tool policy 开启（§12 阻塞项 1）。** |
| 子 Agent 发错群聊或话题 | 用户体验和隐私风险 | 禁止猜测目标；必须使用 runtime 注入的 `chat_id/message_id/reply_to_id`。 |
| 频繁通知导致刷屏 | 群聊噪音 | 进度通知按 30 秒或状态变化限频；心跳 60 秒以上。 |
| CLI fallback 需要飞书凭据 | 部署成本和安全风险 | OpenClaw `message` tool 优先；CLI fallback 只作为兜底，并使用配置/环境变量，不手写 token。 |
| **凭据明文暴露在 JSONL 日志** | **appSecret 泄露，可被滥用** | **CLI 封装凭据管理；禁止 Agent 直接 cat 配置文件；轮换已暴露凭据。** |
| 子 Agent 生命周期不可靠 | 批量监控中断 | 启动 ack + watchdog + 主 Agent 补偿发送。 |
| prompt 约束仍被模型忽略 | 问题复发 | 引入运行时自检和测试，要求 session 中存在真实 `message`/`notify` 调用证据。 |
| **部署环境与仓库版本漂移** | **新功能不可用，问题反复** | **版本校验 + DEPLOY-SYNC.md + init 阶段检查。** |
| 私聊路由使用错误的 `receive_id_type` | 消息发送失败 | `notification_context.is_group_chat` 决定路由方式。 |

## 10. 验收场景

### 场景 A：单文件群聊转写

步骤：

1. 用户在飞书群聊中 @ 机器人，并回复一个音频文件要求转写。
2. Agent 发送“收到文件”实时消息。
3. Agent 下载文件前发送“开始下载”。
4. 下载完成后发送“下载完成，开始预检”。
5. 预检后发送“开始上传并提交”。
6. 任务提交后发送 task id。
7. 状态变化时发送进度。
8. 完成后发送摘要和附件。

通过标准：

- 飞书中阶段消息按实际处理时间分布；
- JSONL 中存在真实消息工具或 CLI notify 调用；
- 不再出现所有阶段消息在 turn 结束后集中投递。

### 场景 B：批量任务子 Agent 监控

步骤：

1. 用户发起本地批量转写。
2. 主 Agent 扫描、提交任务组。
3. 主 Agent 调用 OpenClaw sub-agent，并传入 `notification_context`。
4. 子 Agent 5 秒内发送“监控已启动”。
5. 主 Agent 释放并响应新用户消息。
6. 子 Agent 按进度变化发送通知。
7. 完成后下载结果并发送汇总。

通过标准：

- 子 Agent 的第一条消息可在飞书看到；
- 子 Agent 进度消息包含 `batch_id`；
- 主 Agent 不被批量轮询阻塞；
- 子 Agent 失败时 fallback 生效。

### 场景 C：私聊单文件转写

步骤：

### 一般决策项（五项已定）

#### ✅ 4. completion announce 能否承担进度播报？

**决策：不能。completion announce 只作为最终兜底，不承担进度播报。**

官方依据（[Sub-agents — Announce](https://docs.openclaw.ai/subagents)）：

> "On completion, the sub-agent announces a summary/result message back to the requester chat channel."

且 `sessions_spawn` 明确不接受 `target/channel/replyTo` 这类投递参数（[Sub-agents](https://docs.openclaw.ai/subagents)）。

具体规则：

- completion announce 只作为最终兜底——确保子 Agent 结束时至少有一条回传。
- 批量进度必须由子 Agent 执行中主动调用 `message` tool 或 CLI/API 发送。
- 如果子 Agent 没有主动发消息工具（`message` tool 和 `cli notify` 都不可用），就不启动"子 Agent 监控播报模式"，直接 fallback 到主 Agent 轮询。

#### ✅ 5. 群聊进度消息发群还是回复原消息线程？

**决策：优先回复原消息线程，必要时退回发群。**

OpenClaw 群聊默认 `messages.groupChat.visibleReplies: 'message_tool'`（[Groups](https://docs.openclaw.ai/channels/groups)），Agent 必须调用 `message` tool 才能在群聊中可见发言。这直接解释了 Case 1 的根因。

回复策略：

- 有 `reply_to_id` / `message_id`：走回复消息（reply），把"收到 → 提交 → 进度 → 完成"收束到同一个话题里，避免刷群干扰其他人。
- 没有 reply id 但有 `chat_id`：直接发群。
- 完成和异常可以同时保持在原线程里，并 @ 触发用户，保证可见性。

> **注意**：飞书话题线程有已知 Bug（[Issue #10773](https://github.com/openclaw/openclaw/issues/10773)），子 Agent 回复可能创建新消息而非在原话题内回复。如果实测遇到此问题，退回发群。

#### ✅ 6. 批量失败时是否 @ 触发用户？

**决策：异常和最终完成 @ 触发用户；普通进度和心跳不 @。**

| 通知类型 | 是否 @ 用户 | 说明 |
|---------|------------|------|
| 启动确认 | 不 @ | 除非群很大或用户明确要求 |
| 普通进度 | 不 @ | 避免打扰 |
| 心跳 | 不 @ | 仅告知"仍在运行" |
| 首次出现失败 | **@ 触发用户一次** | 确保感知 |
| 最终完成且有失败项 | **@ 触发用户** | 需要关注 |
| 全部成功完成 | 可 @ | 消息简洁；如在原线程回复也可不 @ |

#### ✅ 7. CLI fallback 是否允许持有飞书 app secret？

**决策：允许，但只作为部署兜底，凭据不能暴露给 Agent 文本层。**

适配器优先级不变：

1. OpenClaw `message` tool
2. CLI notify fallback
3. 普通 assistant 文本（仅本地终端可用）

CLI fallback 调飞书 API 必然需要 `app_id/app_secret` 或等价 token 获取能力。关键不是"绝对不能持有"，而是：

- 凭据放在受控配置文件或环境变量里。
- 配置文件权限 `0600`。
- **禁止 Agent 直接 `cat` OpenClaw 配置文件**。
- CLI 日志脱敏——不输出 `appSecret` 和 `tenant_access_token`。
- Session/transcript 不出现凭据明文。
- 轮换后的新 secret 不写入任何文档。

#### ✅ 8. 私聊是否需要区别通知频率？

**决策：频率策略默认一致，模板区别化。**

**一致的部分**（群聊 / 私聊共享）：

- 进度：状态变化或每 30 秒上限。
- 心跳：无变化 60 秒以上。
- 完成/异常：立即发。
- 不因私聊就高频刷屏。

**区别的部分**（私聊模板更简洁）：

- 私聊不需要 @ 用户。
- 私聊不需要群名、话题说明。
- 私聊模板可以更短，例如"3/20 已完成，预计还需 4 分钟"。
- 群聊模板要包含 `batch_id`，避免多个用户任务混淆。
