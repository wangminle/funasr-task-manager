# 并发吞吐量 RTF 基准测试方案

> 日期: 2026-04-08  
> 状态: 已实施

## 1. 问题背景

当前 benchmark 仅测量**单线程 RTF**——依次发送一个音频文件到服务器，测量 `处理耗时 / 音频时长`。  
这导致线程数不同的服务器（如 4 线程 vs 8 线程 vs 16 线程）测出的 RTF 几乎相同，  
调度器无法区分它们的真实吞吐能力，任务分配趋于均等分配。

**根本原因**: 单文件 benchmark 只使用了服务器的 1 个 worker 线程，无法反映其并发处理能力。

## 2. 方案概述

引入**双指标 benchmark 体系**：

| 指标 | 含义 | 用途 |
|------|------|------|
| `single_rtf` | 单线程 RTF = 单文件处理耗时 / 音频时长 | 单任务 ETA 估算 |
| `throughput_rtf` | 吞吐量 RTF = 并发满载时的 wall_clock / (N × 音频时长) | 调度器配额分配 |

### 吞吐量 RTF 的含义

`throughput_rtf` 反映服务器在满载并发下，**每秒能处理多少秒音频**的倒数。  
例如 `throughput_rtf = 0.003` 意味着服务器每秒可处理约 333 秒的音频。

### 调度器配额公式

```
speed(server) = 1 / throughput_rtf
quota(server) = total_tasks × speed(server) / Σ speeds
```

这样 8 线程服务器自然获得约 2× 于 4 线程服务器的任务量。

## 3. 梯度并发测试流程

### 样本分工

| 阶段 | 样本文件 | 选择原因 |
|------|---------|---------|
| Phase 1: single_rtf | `tv-report-1.wav` (长 WAV) | 较长音频 → 单线程测量更准确 |
| Phase 2: throughput_rtf | `test.mp4` (短 MP4) ×N | 短文件 ×N 并发 → 快速完成梯度测试 |

### 测试流程

```
对每台 ONLINE 服务器:
  1. 单线程测试 → single_rtf
     - 发送 tv-report-1.wav (一次)
     - single_rtf = elapsed / audio_duration
     
  2. 梯度并发测试 → throughput_rtf
     对 N ∈ [1, 2, 4, 8] (上限 = max_concurrency):
       - 同时建立 N 个 WebSocket 连接
       - 每个连接发送 test.mp4 的一份拷贝
       - 记录从第一个开始发送到最后一个收到结果的 wall_clock 时间 T
       - per_file_rtf(N) = T / audio_duration
       - throughput_rtf(N) = T / (N × audio_duration)
       - 记录各级别的结果用于分析
     
  3. 取最终 throughput_rtf = throughput_rtf(max_tested_N)
     - max_tested_N = min(max_concurrency, 8)
```

### 关键输出指标

```json
{
  "single_rtf": 0.024,
  "throughput_rtf": 0.003,
  "benchmark_concurrency": 8,
  "benchmark_samples": ["tv-report-1.wav", "test.mp4"],
  "concurrency_gradient": [
    {"concurrency": 1, "per_file_rtf": 0.024, "throughput_rtf": 0.024},
    {"concurrency": 2, "per_file_rtf": 0.025, "throughput_rtf": 0.0125},
    {"concurrency": 4, "per_file_rtf": 0.026, "throughput_rtf": 0.0065},
    {"concurrency": 8, "per_file_rtf": 0.028, "throughput_rtf": 0.0035}
  ]
}
```

## 4. 任务拆解

### 4.1 数据库迁移 (Alembic 003)

`server_instances` 表新增两列:

| 列名 | 类型 | 说明 |
|------|------|------|
| `throughput_rtf` | `Float`, nullable | 并发满载吞吐量 RTF |
| `benchmark_concurrency` | `SmallInteger`, nullable | 测量 throughput_rtf 时使用的并发数 |

保留现有 `rtf_baseline` 列存储 single_rtf（语义不变）。

### 4.2 模型层 (model + schema)

- **`ServerInstance`** 模型: 添加 `throughput_rtf`, `benchmark_concurrency` 映射列
- **`ServerResponse`** schema: 新增 `throughput_rtf`, `benchmark_concurrency` 字段
- **`ServerBenchmarkItem`** schema: 新增 `single_rtf`, `throughput_rtf`, `benchmark_concurrency`, `concurrency_gradient` 字段

### 4.3 Benchmark 服务 (server_benchmark.py)

- 新增 `benchmark_server_concurrent()` — 在指定并发度下发送 N 个 sample
- 新增 `benchmark_server_full()` — 编排完整的单线程 + 梯度并发测试
- 更新 `ServerBenchmarkResult` 数据类携带新指标

### 4.4 API 层 (servers.py)

- `POST /api/v1/servers/benchmark` — 调用新的 full benchmark，写入 `rtf_baseline` + `throughput_rtf` + `benchmark_concurrency`
- `POST /api/v1/servers/{id}/benchmark` — 同上（单台服务器）
- 响应体包含梯度测试详情

### 4.5 调度器 (scheduler.py)

- **`ServerProfile`**: 新增 `throughput_rtf` 字段
- **`_allocate_quotas()`**: 用 `throughput_rtf`（fallback `rtf_baseline`）计算 speed
- **`get_effective_rtf()`**: 仍用 `rtf_baseline` (single_rtf) + penalty 做单任务 ETA
- **`get_base_rtf()`**: 仍用 `rtf_baseline` 做纯处理速度比较
- 新增 **`get_throughput_speed()`**: 返回 `1 / throughput_rtf` 用于配额

### 4.6 CLI (commands/server.py)

- `server list`: 展示列新增 `吞吐量RTF`, `测试并发`
- `server benchmark`: 输出包含梯度测试详情

### 4.7 测试脚本

- 更新 `benchmark_servers_rtf.py` 支持并发测试模式
- 更新相关单元测试适配新字段

## 5. 向后兼容

- `rtf_baseline` 列保留不删，语义 = single_rtf
- `throughput_rtf` 为 nullable，调度器 fallback 到 `rtf_baseline / 1`（即假设并发度 = 1）
- 已有服务器首次 benchmark 后自动填充新字段
