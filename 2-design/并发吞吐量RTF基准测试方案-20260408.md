# 并发吞吐量 RTF 基准测试方案

> 日期: 2026-04-08  
> 最后更新: 2026-04-09  
> 状态: 已实施（含退化检测增强）

## 1. 问题背景

当前 benchmark 仅测量**单线程 RTF**——依次发送一个音频文件到服务器，测量 `处理耗时 / 音频时长`。  
这导致线程数不同的服务器（如 4 线程 vs 8 线程 vs 16 线程）测出的 RTF 几乎相同，  
调度器无法区分它们的真实吞吐能力，任务分配趋于均等分配。

**根本原因**: 单文件 benchmark 只使用了服务器的 1 个 worker 线程，无法反映其并发处理能力。

## 2. 方案概述

引入**双指标 benchmark 体系**：

| 指标 | 含义 | 用途 |
|------|------|------|
| `single_rtf` | 单线程 RTF = 单文件处理耗时 / 音频时长 | 单任务 ETA 估算 |
| `throughput_rtf` | 吞吐量 RTF = 并发满载时的 wall_clock / (N × 音频时长) | 调度器配额分配 |

### 吞吐量 RTF 的含义

`throughput_rtf` 反映服务器在满载并发下，**每秒能处理多少秒音频**的倒数。  
例如 `throughput_rtf = 0.003` 意味着服务器每秒可处理约 333 秒的音频。

### 调度器配额公式

```
speed(server) = 1 / throughput_rtf
quota(server) = total_tasks × speed(server) / Σ speeds
```

这样 8 线程服务器自然获得约 2× 于 4 线程服务器的任务量。

## 3. 梯度并发测试流程

### 样本分工

| 阶段 | 样本文件 | 选择原因 |
|------|---------|---------|
| Phase 1: single_rtf | `tv-report-1.wav` (长 WAV) | 较长音频 → 单线程测量更准确 |
| Phase 2: throughput_rtf | `test.mp4` (短 MP4) ×N | 短文件 ×N 并发 → 快速完成梯度测试 |

### 测试流程

```
对每台 ONLINE 服务器:
  1. 单线程测试 → single_rtf
     - 发送 tv-report-1.wav (一次)
     - single_rtf = elapsed / audio_duration
     
  2. 梯度并发测试 → throughput_rtf
     对 N ∈ [1, 2, 4, 8] (始终跑完整梯度，不受 max_concurrency 封顶):
       - 同时建立 N 个 WebSocket 连接
       - 每个连接发送 test.mp4 的一份拷贝
       - 记录从第一个开始发送到最后一个收到结果的 wall_clock 时间 T
       - per_file_rtf(N) = T / audio_duration
       - throughput_rtf(N) = T / (N × audio_duration)
       - 记录各级别的结果用于分析
     
  3. 退化检测取最终值（详见 §3.1）
     - 从 N=1 逐级验证，找到最高非退化并发级别
     - throughput_rtf = throughput_rtf(recommended_N)
     - recommended_N 同时写入 max_concurrency
```

### 3.1 退化检测算法（2026-04-09 增强）

**原方案**取 `throughput_rtf(max_tested_N)`，即无条件使用最高级别的结果。
**增强方案**使用退化检测逐级验证，避免以下问题：

- 服务器只有 2 个 worker 但配置了 `max_concurrency=4`，N=4 时吞吐不增
- 某次 benchmark 因瞬时负载得到较低推荐值，后续 benchmark 受 max_concurrency 封顶无法恢复

**算法流程**：

```
best = gradient[0]  # 从 N=1 开始

for i in 1..len(gradient):
  current = gradient[i]
  prev = gradient[i-1]
  
  # 条件 1: 吞吐量必须有 ≥10% 的改善
  improvement = 1 - current.throughput_rtf / prev.throughput_rtf
  if improvement < 0.10:
    break  # 退化点
  
  # 条件 2: 单文件 RTF 不能超过 single_rtf 的 2 倍
  if current.per_file_rtf > single_rtf × 2.0:
    break  # 退化点
  
  best = current

recommended_concurrency = best.concurrency
throughput_rtf = best.throughput_rtf
```

**关键设计决策**：

| 决策 | 说明 |
|------|------|
| 梯度不受 max_concurrency 封顶 | 防止螺旋下降：被降低的 max_concurrency 限制后续探测范围 |
| `THROUGHPUT_MIN_IMPROVEMENT = 0.10` | 吞吐量需有 ≥10% 改善才认为当前级别有效 |
| `PER_FILE_MAX_DEGRADATION = 2.0` | 单文件延迟不超过 single_rtf 的 2 倍 |
| recommended_concurrency → max_concurrency | benchmark 自动校准调度槽位数（仅在梯度完整时允许降级，见 §3.2） |

**退化检测示例**（2 worker 节点冒充 4 并发）：

| N | per_file_rtf | throughput_rtf | 改善幅度 | 判定 |
|---|-------------|---------------|---------|------|
| 1 | 0.024 | 0.024 | — | ✅ 基准 |
| 2 | 0.025 | 0.0125 | 47.9% | ✅ 接受 |
| 4 | 0.050 | 0.0125 | 0% | ❌ 退化 |

→ `recommended_concurrency = 2`，`throughput_rtf = 0.0125`

### 3.2 不完整梯度保护（2026-04-09）

如果梯度测试因超时或网络瞬时错误提前终止（例如 N=1 成功但 N=2 超时），`gradient_complete` 标记为 `false`。此时 `_apply_benchmark_result()` **只允许 `max_concurrency` 上调或维持**，拒绝基于不完整证据的降级。

原因：一次瞬时抖动不代表服务器并发能力下降，不应把原本 4/8 槽的节点永久降到 1 槽。

| 场景 | gradient_complete | recommended < current | 结果 |
|------|:-:|:-:|------|
| 完整梯度 + 退化检测降级 | true | 是 | 正常写入（可信降级） |
| 部分梯度 + 建议升级 | false | 否 | 正常写入（向上安全） |
| 部分梯度 + 建议降级 | false | 是 | **跳过写入**，保持原 max_concurrency |

### 关键输出指标

```json
{
  "single_rtf": 0.024,
  "throughput_rtf": 0.003,
  "benchmark_concurrency": 8,
  "recommended_concurrency": 8,
  "gradient_complete": true,
  "benchmark_samples": ["tv-report-1.wav", "test.mp4"],
  "concurrency_gradient": [
    {"concurrency": 1, "per_file_rtf": 0.024, "throughput_rtf": 0.024},
    {"concurrency": 2, "per_file_rtf": 0.025, "throughput_rtf": 0.0125},
    {"concurrency": 4, "per_file_rtf": 0.026, "throughput_rtf": 0.0065},
    {"concurrency": 8, "per_file_rtf": 0.028, "throughput_rtf": 0.0035}
  ]
}
```

## 4. 任务拆解与开发进展

| 列名 | 类型 | 说明 |

| `throughput_rtf` | `Float`, nullable | 并发满载吞吐量 RTF |
| `benchmark_concurrency` | `SmallInteger`, nullable | 测量 throughput_rtf 时使用的并发数 |

保留现有 `rtf_baseline` 列存储 single_rtf（语义不变）。

- **`ServerInstance`** 模型: 添加 `throughput_rtf`, `benchmark_concurrency` 映射列

- **`ServerBenchmarkItem`** schema: 新增 `single_rtf`, `throughput_rtf`, `benchmark_concurrency`, `recommended_concurrency`, `concurrency_gradient` 字段

### 4.3 Benchmark 服务 (server_benchmark.py) ✅

- 新增 `_benchmark_concurrent()` — 在指定并发度下发送 N 个 sample
- 新增 `benchmark_server_full()` — 编排完整的单线程 + 梯度并发测试
- 新增 `_detect_optimal_concurrency()` — 退化检测算法（2026-04-09）
- 更新 `ServerBenchmarkResult` 数据类携带新指标（含 `recommended_concurrency`）
- 梯度测试始终跑完整 `(1, 2, 4, 8)`，不受 `max_concurrency` 封顶（2026-04-09 修正）

### 4.4 API 层 (servers.py) ✅

- `POST /api/v1/servers/benchmark` — 调用新的 full benchmark，写入 `rtf_baseline` + `throughput_rtf` + `benchmark_concurrency` + `max_concurrency`
- `POST /api/v1/servers/{id}/benchmark` — 同上（单台服务器）
- 响应体包含梯度测试详情和 `gradient_complete` 标志
- 提取 `_apply_benchmark_result()` 统一处理 benchmark 结果写入（2026-04-09 重构）
- `_apply_benchmark_result()` 在 `gradient_complete=false` 时拒绝降级 `max_concurrency`（2026-04-09 增强）

### 4.5 调度器 (scheduler.py) ✅

- **`ServerProfile`**: 新增 `throughput_rtf` 字段
- **`_allocate_quotas()`**: 用 `throughput_rtf`（fallback `rtf_baseline`）计算 speed

- **`get_base_rtf()`**: 仍用 `rtf_baseline` 做纯处理速度比较

### 4.6 CLI (commands/server.py + transcribe.py) ✅

- `server list`: 展示列新增 `吞吐量RTF`, `测试并发`

### 4.7 测试脚本 ✅

- 更新 `benchmark_servers_rtf.py` 支持并发测试模式

### 4.8 运行时 RTF 保护 (task_runner.py) ✅

**设计不变量**: `rtf_baseline` 只由 benchmark 设置，运行时不回写。
