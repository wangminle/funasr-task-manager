# 讨论方案实现对照与优先级清单

> 日期：2026-04-03
> 目的：把 1-discussion 中已经讨论过的关键方案，与当前仓库中的真实落地情况做一次收口，并给出下一步最值得投入开发时间的顺序。

## 一、执行摘要

- 已经落地的主干能力主要有四类：调度主框架、多节点并发、批量任务组、实时进度与 CLI 批量转写。
- 当前最值得投入开发时间的缺口，不是推翻现有调度器重做，而是补齐超长文件处理能力、真实队列 ETA、调度可观测性和 benchmark 校准。
- 对当前项目而言，MLFQ、RR 这类经典操作系统调度方案不属于高价值缺口；即使现在没有实现，也不构成明显遗憾。

## 三、已实现能力的判断结论

### 3.1 调度主框架已经落地，不宜推翻重来

当前仓库已经具备以下能力：

- LPT 排序
- Earliest-Finish-Time 分配
- 基于 RTF 的异构服务器容量感知
- 并发惩罚因子
- 任务分发主循环

这和 [scheduling-strategy-review-20260330.md](../1-discussion/scheduling-strategy-review-20260330.md) 中“部分建议其实代码里已经存在，不应误判为从零开始”的判断是一致的。

### 3.2 批量任务和 CLI 批量提交已经跨过“串行瓶颈”阶段

[CLI串行问题分析-20260328.md](../1-discussion/beta测试记录/20260328-openclaw试用工具/CLI串行问题分析-20260328.md) 当时指出的问题是：CLI 采用“上传一个、等待一个、下载一个”的串行流，导致无法利用后端调度器的并发能力。

当前仓库已经把这个点基本补上：

- 支持批量创建任务
- 支持 task_group 聚合
- CLI 多文件自动切换到批量模式
- 支持 no-wait 提交

因此，这一项应归类为“已实现，后续主要补文档与交互细节”，而不是继续当成核心能力缺口。

## 四、按优先级排序的待办项

### 4.1 P0：长音频隐式分段并行

**当前状态**

- 目前仓库还没有真正进入“父任务外显、内部段级并行”的阶段。
- 现有实现更像“整文件转码后整体送入 ASR 服务”，而不是“切段后分散到多服务器执行”。

**为什么最值得做**

- 这是唯一一个直接针对超长文件 wall-clock 的结构性能力。
- 讨论文档已经明确给出收益目标：超长文件在三台服务器集群下 wall-clock 至少下降 25%。
- 这一能力一旦成立，后续 benchmark、调度器、进度系统的优化收益都会被放大，因为调度对象从“整文件”变成了“可并行的小段”。

**讨论出处**

- [超长音视频隐式分段并行方案（单任务外显）.md](../1-discussion/超长音视频隐式分段并行方案（单任务外显）.md)
  这里完整提出了单任务外显、内部 task_segments、静音切段、段级重试、统一合并的总体方案。
- [VAD分段并行转写方案（合并版）-20260328.md](../1-discussion/VAD分段并行转写方案（合并版）-20260328.md)
  这里把触发阈值、静音参数、段长、双层调度约束、风险缓解和实施步骤进一步收敛成可执行设计。

### 4.2 P1：真实队列 ETA + 排队位次贯通 + 调度决策落库

**当前状态**

- 这项能力不是从零开始，而是“底层已有计算，链路还没打通”。
- 当前 ETA 更接近“纯处理时长估算”，还不是“排队等待 + 处理时长”的完整 ETA。
- 调度器内部已有 estimated_start、estimated_finish、queue_position 等信息，但未完整落到任务记录与前端展示。

**为什么值得优先做**

- 这是对用户感知最强的一项基础设施改造。用户等待时首先问的是“还要多久”。
- 这项工作对现有架构侵入较低，因为核心计算已经存在，主要缺的是持久化、SSE 透出和前端消费。
- 调度决策落库后，后续才有条件回答“为什么某个任务排到了这台机器”“为什么 ETA 偏差大”。

**讨论出处**

- [scheduling-strategy-review-20260330.md](../1-discussion/scheduling-strategy-review-20260330.md)
  这里明确指出当前 ETA 不包含排队等待时间，建议把 ETA 修正提前到第一优先级，同时补 schedule_reason、queue_position_at_dispatch、estimated_start_at_dispatch、estimated_finish_at_dispatch 等决策可观测字段。

### 4.3 P2：benchmark 多次采样 / 并行校准

**当前状态**

- 当前系统已经有 benchmark 能力，也有运行时基于任务完成结果的 RTF 滚动更新。
- 但 benchmark 依然更接近“单次测量”，还没有充分解决共享 GPU、冷启动、顺序执行带来的偏差问题。

**为什么值得做**

- 调度器的判断质量，首先取决于输入数据是否可信。
- 如果 RTF 基线本身偏了，后面的 quota、EFT、ETA 都会跟着偏。
- 这项工作的价值不在直接提升用户界面，而在于提升整个调度系统的判断精度，是后续优化的地基。

**讨论出处**

- [scheduling-strategy-review-20260330.md](../1-discussion/scheduling-strategy-review-20260330.md)
  文档明确指出：共享 GPU 场景下，单次 benchmark 极易受时序差异和资源竞争影响，建议采用多次采样取中位数、并行 benchmark、运行时更快收敛等策略。

### 4.4 P3：分波调度

**当前状态**

- 当前轮询调度会尽快把可用槽位填满，但还没有形成明确的“按波次投放任务，再利用新完成结果重算下一波”的机制。

**为什么值得做**

- 这项能力的核心价值不在“更复杂”，而在“更及时地利用新信息”。
- 第一波任务完成后，调度器能拿到更真实的 RTF 反馈，再去决定下一波，整体比一次性全量决策更稳。
- 它也更利于短任务穿透，不至于长期被第一批长任务的粗估结果绑定住。

**讨论出处**

- [scheduling-strategy-review-20260330.md](../1-discussion/scheduling-strategy-review-20260330.md)
  文档把分波调度视为最高 ROI 的下一步之一，理由是实现成本相对可控，但能同时改善冷启动、短任务周转和资源动态适应。

### 4.5 P4：priority 调度

**当前状态**

- 这是当前讨论中“有价值，但不应先做”的项目。
- 问题不在调度代码是否能支持 priority，而在于产品语义尚未定清楚。

**为什么值得做但放在后面**

- 在有明确产品规则时，priority 能直接影响等待策略和资源分配，是有价值的。
- 但如果没有先定义“谁能设置优先级、优先级如何影响排队、是否会影响用户公平预期”，过早实现很容易变成技术功能先行、产品语义滞后。
- 因此，它应排在 ETA、benchmark、分波调度这些基础设施项之后。

**讨论出处**

- [funasr-scheduling-analysis-20260329.md](../1-discussion/funasr-scheduling-analysis-20260329.md)
  文档最早明确提出了 priority 调度和多级队列方向。
- [scheduling-strategy-review-20260330.md](../1-discussion/scheduling-strategy-review-20260330.md)
  文档进一步修正为：priority 可以做，但必须先定义清楚产品语义，并建议先做最小 MVP。

## 五、不建议作为当前重点投入的方案

### 5.1 MLFQ

**结论**

- 不是当前阶段的高价值缺口。
- 现在没做，并不遗憾。

**原因**

- 当前系统是多节点、多并发槽位、非抢占式的批量任务场景，和 MLFQ 最擅长解决的“单机时间片抢占”问题域并不一致。
- 对当前项目而言，MLFQ 很容易退化成复杂版的固定优先级多队列，但成本和调参复杂度都更高。

**讨论出处**

- [scheduling-strategy-review-20260330.md](../1-discussion/scheduling-strategy-review-20260330.md)
  评审文档已经明确把 MLFQ 从“强烈推荐”降级为“不建议当前阶段引入”。

### 5.2 RR

**结论**

- 不属于当前项目应继续追求的方向。

**原因**

- RR 的优势建立在任务可被时间片抢占的前提上。
- ASR 转写任务一旦开始通常不能被任意中断，RR 在这里没有真正可落地的时间片意义。

**讨论出处**

- [funasr-scheduling-analysis-20260329.md](../1-discussion/funasr-scheduling-analysis-20260329.md)
  该文档回顾了 RR 的经典适用场景。
- [scheduling-strategy-review-20260330.md](../1-discussion/scheduling-strategy-review-20260330.md)
  评审文档进一步指出：RR 和抢占式调度不适合当前非抢占式 ASR 任务。

## 六、建议的实际投入顺序

如果下一阶段只按“最值得投开发时间”的标准来排，建议顺序保持如下：

1. 长音频隐式分段并行
2. 真实队列 ETA + 排队位次贯通 + 调度决策落库
3. benchmark 多次采样 / 并行校准
4. 分波调度
5. priority 调度

这个顺序的核心逻辑是：

- 先解决超长文件的结构性性能瓶颈。
- 再解决用户最可感知的等待体验和调度可观测性。
- 再修正调度输入数据的可信度。
- 然后再进入更进一步的调度策略优化。
- 最后才是需要产品规则先定义清楚的优先级体系。
