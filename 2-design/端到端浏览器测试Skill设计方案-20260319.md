## 一、背景与目标

### 1.1 现状

项目已有的 `4-tests/scripts/e2e/` 是基于 pytest + httpx 的 API 级别端到端测试，覆盖了后端文件上传、任务创建、状态流转、结果获取等流程，但缺少以下路径：

- 真实浏览器中的文件拖拽/选择上传
- 前端 Element Plus 组件的交互（配置选项、提交按钮、进度展示）
- 完整的「用户视角」端到端链路（从打开页面到拿到转写结果）
- 面向不同规模的分层回归测试

### 1.2 目标

构建一个可重复执行的浏览器 E2E 测试 Skill，满足：

1. **模拟真实用户操作**：浏览器打开上传页 → 选择音视频文件 → 配置参数 → 提交转写 → 等待完成 → 验证结果
2. **分层回归**：smoke（日常快速）/ remote-standard（远端推荐标准）/ standard（本地较重合并前）/ full（发版前）四档
3. **结果可追溯**：每次测试产出结构化报告和工件归档
4. **Agent 可复用**：通过 Skill 标准化流程，AI Agent 可直接执行端到端测试

## 二、项目技术架构概览

| 层级 | 技术栈 |
| ---- | ------ |
| 前端 | Vue 3 + Vite + Element Plus，端口 5173 |

### 核心 API 端点

| 端点 | 方法 | 用途 |
| ---- | ---- | ---- |
| `/api/v1/files/upload` | POST | 上传音视频文件 |
| `/api/v1/tasks` | POST | 批量创建转写任务 |
| `/api/v1/tasks` | GET | 查询任务列表 |
| `/api/v1/tasks/{id}` | GET | 查询单个任务状态 |
| `/api/v1/tasks/{id}/result` | GET | 获取转写结果 (json/txt/srt) |
| `/api/v1/tasks/{id}/progress` | GET | SSE 进度推送 |
| `/api/v1/servers` | GET | 查询已注册 FunASR 服务器 |
| `/health` | GET | 健康检查 |

### 任务状态流转

```text
PENDING → PREPROCESSING → QUEUED → DISPATCHED → TRANSCRIBING → SUCCEEDED
                                                              ↘ FAILED
                                                              ↘ CANCELED
```

## 四、测试流程设计（7 个阶段）

```text
┌──────────┐  ┌──────────┐  ┌──────────┐  ┌──────────┐  ┌──────────┐  ┌──────────┐  ┌──────────┐
│ Phase 1  │→│ Phase 2  │→│ Phase 3  │→│ Phase 4  │→│ Phase 5  │→│ Phase 6  │→│ Phase 7  │
│ 目标选择 │  │ 批次生成 │  │ 环境预检 │  │ 环境准备 │  │ 浏览器   │  │ 分层验证 │  │ 报告归档 │
│          │  │          │  │          │  │          │  │ 模拟操作 │  │          │  │          │
└──────────┘  └──────────┘  └──────────┘  └──────────┘  └──────────┘  └──────────┘  └──────────┘
```

### Phase 1：选择测试目标

根据触发场景选择 Profile：

- 用户要求"快速验证" → `smoke`
- 用户要求"远端节点标准回归" / "带宽有限但想多跑几个文件" → `remote-standard`
- 用户要求"合并前检查" → `standard`
- 用户要求"发版回归" / "全量测试" → `full`

### Phase 2：生成素材批次

运行 `build_fixture_batch.py`，产出 fixture manifest JSON。

### Phase 3：环境预检

在启动浏览器前，必须逐项检查：

| 检查项 | 方法 | 期望结果 |
| ------ | ---- | -------- |
| 后端健康 | `GET http://localhost:8000/health` | `{"status": "ok"}` |
| 前端可达 | `GET http://localhost:5173` | HTTP 200 |
| FunASR 服务器 | `GET /api/v1/servers` | 至少 1 个已注册服务器 |
| 测试数据 | 检查 `7-data/assets/1-测试audioFiles/` | 包含音视频文件 |

任一检查失败则终止并报告缺失的前置条件。

### Phase 4：环境准备

- 后端：`http://localhost:8000`（Uvicorn）
- 前端：`http://localhost:5173`（Vite dev server）
- 浏览器自动化：Playwright（推荐安装在 `3-dev/src/frontend/`）
- 工件输出：`7-data/outputs/e2e/<timestamp>/`

### Phase 5：浏览器模拟操作

完整用户路径：

```text
1. 打开 http://localhost:5173/upload
2. 通过 input[type=file].setInputFiles() 注入批次中的所有文件
3. 等待文件出现在"待上传文件"表格中
4. 确认语言、标点、ITN 等选项配置
5. 点击"提交转写"按钮
6. 等待 "成功创建 N 个转写任务" 提示
7. 导航到 /tasks 任务列表页
8. 轮询等待所有任务到达终态（5–10 秒间隔）
9. 对每个 SUCCEEDED 任务获取 txt/json/srt 结果
```

**关键技术决策**：

- 使用 `setInputFiles()` 而非模拟拖拽事件，因为 Element Plus 的 `el-upload` 最终走同一个 file input，直接赋值更稳定
- 等待机制使用 Playwright 的 `waitForSelector` 和 `waitForResponse`，不使用固定 sleep
- 为了选择器稳定性，建议在前端添加 `data-testid` 属性

### Phase 6：分层验证

按严格度递进的三层断言：

#### 第一层：硬性通过条件（Hard Gate）

- 页面正常加载
- 文件成功上传
- 任务创建数量 = 文件数量
- 任务状态推进到终态
- 成功任务可下载结果

#### 第二层：结构性校验（Structural Gate）

- txt 结果非空
- json 结果中 `text` 字段非空
- 视频文件不因预处理失败而跳过

#### 第三层：语义校验（Semantic Gate）

- 如有维护的关键词基线，校验结果包含期望关键词
- 如无基线，记录转写文本并标记待人工审核
- 不要求精确文本匹配（ASR 模型输出有不确定性）

## 五、超时策略

| Profile | 单任务超时 | 整体超时 |
| ------- | ---------- | -------- |
| smoke | 5 分钟 | 10 分钟 |
| remote-standard | 8 分钟 | 20 分钟 |
| standard | 10 分钟 | 30 分钟 |
| full | 20 分钟 | 90 分钟 |

超时依据：大文件（>100MB）需经历上传 → ffmpeg 预处理转码 → WebSocket 传输 → ASR 推理，412MB 视频可能需要 15 分钟以上。

超时行为：截图当前页面状态，记录各任务当前 status，生成失败报告后退出。

## 十、设计决策记录

### Q1：为什么用 Playwright 而不是纯 API 测试？

用户明确要求"模拟拖入 webpage"的体验。浏览器测试能覆盖前端组件渲染、文件选择交互、进度展示等端到端路径，这些是 API 级 E2E 无法覆盖的。现有 `4-tests/scripts/e2e/` 已覆盖 API 层。

### Q2：为什么用 `setInputFiles()` 而不是模拟真实拖拽事件？

Element Plus 的 `el-upload` 组件最终都通过 `<input type="file">` 接收文件。直接赋值比合成 DragEvent 更稳定可靠。只在明确需要测试"仅拖拽行为"时才模拟真实拖拽。

### Q3：为什么分三个 Profile？

大文件（178MB、412MB）上传 + 转写耗时长，不适合频繁在远端链路上运行。smoke 让日常开发快速得到反馈；remote-standard 固定使用五个最小文件作为远端推荐标准回归；standard 在本地或资源较充足环境下提供更重覆盖；full 用于发版前的全量验证。

### Q4：为什么不做精确文本比对？

ASR 模型输出有不确定性，相同音频在不同时间可能产出略有差异的文本。启发式验证（非空、格式正确、关键词匹配）已足够发现大多数回归。在未建立基线前，不应把模型措辞微调当作失败。

### Q5：浏览器测试代码为什么放前端目录而非 4-tests？

Playwright 是 Node.js 工具，与前端 npm 依赖天然同目录。放在 `3-dev/src/frontend/tests/e2e/` 可以直接复用 Vite 配置和 npm 脚本，避免把 Node 浏览器测试强行塞进 Python pytest 目录。
