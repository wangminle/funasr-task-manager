# VAD 分段并行转写实施计划

> 日期：2026-04-24
>
> 范围：基于 `1-discussion/VAD分段-并行转写方案（合并版）-20260328.md` 与 `1-discussion/VAD分段-超长音视频隐式分段并行方案（单任务外显）.md`，评点既有可行性分析，并形成可执行的实施步骤和工作分解清单。
>
> 更新：2026-04-25 合并 `VAD切分策略改进与评估方案-20260425.md`，将实际长音频测试中发现的切分策略、用户控制参数和质量评估要求纳入本实施计划的优化阶段。

## 一、结论摘要

两份评估报告的主结论基本合理：VAD 分段并行转写方案具备实施条件，并且更适合按“合并版”落地。当前仓库已经具备以下关键基础：

- `ffmpeg` 查找与异步转码基础：`3-dev/src/backend/app/services/audio_preprocessor.py`
- 任务状态机：`PENDING/PREPROCESSING/QUEUED/DISPATCHED/TRANSCRIBING/SUCCEEDED/FAILED`
- 多服务器调度器：`3-dev/src/backend/app/services/scheduler.py`
- 后台任务执行链路：`3-dev/src/backend/app/services/task_runner.py`
- FunASR WebSocket adapter：`3-dev/src/backend/app/adapters/funasr_ws.py`
- 结果格式化基础：`3-dev/src/backend/app/services/result_formatter.py`
- Alembic migration 与 Repository 模式

但两份报告对实际改造难度略偏乐观。这个方案不是简单新增几个函数，而是要把当前“父任务就是执行单元”的执行链，扩展为“父任务外显、segment 内部执行”的双层模型。最大风险集中在 `task_runner.py` 的调度、执行、重试、进度聚合和最终合并。

建议采用“两段式落地”：

1. 先完成离线闭环：数据模型、canonical WAV、VAD 切分、结果合并、单元测试。
2. 再接入运行链路：段级调度、段级执行、父任务聚合、API 诊断字段、集成测试。

这样能先验证切分和合并算法，再处理高风险并发执行链。

## 二、对两份评估报告的评点

### 2.1 合理部分

两份报告对以下判断是成立的：

- 方案方向保守：不新增外部任务状态、不改变用户侧“一个文件 = 一个任务”的心智模型。
- 基础设施已经就绪：已有异步任务 runner、多服务器调度、WebSocket 转写、结果格式化、数据库迁移。
- 短文件零影响路径可保留：低于阈值的任务继续走现有整文件路径。
- 独立 `task_segments` 表优于复用 `tasks` 表：能够避免污染外部任务模型，并更容易表达 `keep_start_ms/keep_end_ms` 等段级字段。
- 最大改造点确实是 `task_runner.py`，其次是调度策略和结果合并。

### 2.2 需要修正的部分

#### 1. `ensure_wav()` 不等于 canonical WAV

当前 `ensure_wav()` 对 `.wav/.pcm` 会直接返回原文件，不保证输入已经是 `16kHz / mono / s16`。方案要求“统一转 16kHz mono WAV”，因此需要新增或改造 canonical WAV 逻辑，而不是直接复用 `ensure_wav()` 即可。

建议：

- 保留 `ensure_wav()` 的兼容职责。
- 新增 `ensure_canonical_wav()` 或扩展现有函数，确保所有进入 VAD 切分和 FunASR 的音频都是统一格式。

#### 2. 当前 `PREPROCESSING` 只是延迟状态，不是真预处理

`_promote_preprocessing_tasks()` 当前只是把符合时间条件的任务从 `PREPROCESSING` 推到 `QUEUED`。VAD 切分应插入这里，但需要注意：

- 短文件：继续 `PREPROCESSING -> QUEUED`
- 长文件：canonical WAV -> silencedetect -> plan -> split -> 写入 segments -> `QUEUED`
- 切分失败：父任务 `FAILED`

#### 3. 调度器不一定需要“大改”，更需要 work item 抽象

`schedule_batch()` 当前接收 `dict`，依赖字段主要是 `task_id` 和 `audio_duration_sec`。首版可以把 segment 包装成内部 work item，复用现有 LPT + EFT 算法。

建议不要一开始重写调度器，而是做最小抽象：

- 普通任务 work item：`work_id = task_id`
- 分段任务 work item：`work_id = segment_id`
- `ScheduleDecision.task_id` 后续可重命名或兼容为 `work_id`
- 调度前在 `task_runner` 过滤单父任务并发上限

#### 4. 当前重试是父任务级，与段级重试冲突

当前 `_retry_failed_tasks()` 会将 `FAILED` 父任务重新排队。分段后如果某个 segment 失败，不应直接把父任务置为 `FAILED` 并整体重试。

建议：

- 普通短任务继续沿用父任务级重试。
- 分段任务新增段级重试：`FAILED segment -> PENDING segment`。
- 只有当某段 `retry_count >= max_retry_count` 时，父任务才进入 `FAILED`。

#### 5. 父任务状态语义需要严格遵守现有状态机

现有状态机不允许 `QUEUED -> TRANSCRIBING`，必须经过 `DISPATCHED`。因此分段场景建议语义如下：

| 父任务状态 | 分段语义 |
|---|---|
| `PREPROCESSING` | canonical WAV、VAD 检测、切段、写入 segment 表 |
| `QUEUED` | 段清单已就绪，等待段级调度 |
| `DISPATCHED` | 至少一个段已分配服务器，但尚未开始或父任务聚合进入执行态 |
| `TRANSCRIBING` | 至少一个段正在执行或已有段完成但仍未全部完成 |
| `SUCCEEDED` | 所有段成功，且合并结果写入完成 |
| `FAILED` | 某段重试耗尽，或最终合并失败 |

#### 6. 段结果存储字段需统一

两份方案里出现了 `raw_result_path` 与 `raw_result_json` 两种表达。首版建议使用 `raw_result_json`，减少文件管理复杂度。若后续发现结果体过大，再迁移为 `raw_result_path`。

## 三、实施原则

- 短文件路径优先保持不变，所有新逻辑只在达到分段阈值时启用。
- 先写算法单测，再接入 runner。
- 不新增外部任务状态，不改变已有 API 的必填字段。
- 分段任务的父任务 `assigned_server_id` 保持 `null`，避免误导为单服务器执行。
- 段级并发限制在 `min(online_server_count, 3)`，防止单个超长任务吃满集群。
- 合并成功后可以清理段文件；失败时保留段文件和段结果用于排障。

### 阶段 3：Canonical WAV 与 VAD 切分

**目标**：实现可独立测试的音频切分闭环。

**涉及文件**：

- 修改：`3-dev/src/backend/app/services/audio_preprocessor.py`

**工作项**：

- 新增或改造 canonical WAV 函数，确保输出 `16kHz / mono / s16 WAV`。
- 实现 `silence_detect(wav_path)`：
  - 调用 `ffmpeg -af silencedetect=n=-35dB:d=0.8`
  - 解析 stderr 中的 `silence_start/silence_end/silence_duration`
- 实现 `plan_segments(duration_ms, silence_ranges, config)`：
  - 优先在目标段长附近选择静音点
  - 无静音点时按硬上限切分

- 实现 `split_wav_segments(wav_path, plan)`：
  - 使用 ffmpeg 输出每段 WAV
  - 文件名包含 `task_id` 或 `segment_index`
  - 原子写入，避免半成品被读取

**验收**：

- silencedetect 日志解析单测通过。
- 有静音点时按静音点切分。
- 无静音点时按硬上限切分。

- 短音频不产生不必要切分。

- 纯文本 fallback 可用。
- JSON/TXT/SRT 三种输出可生成。

### 阶段 5：调度 work item 抽象

**目标**：让普通任务和 segment 共用调度器，同时保留现有短文件行为。

**涉及文件**：

- 修改：`3-dev/src/backend/app/services/scheduler.py`
- 修改：`3-dev/src/backend/app/services/task_runner.py`
- 新增或修改测试：`4-tests/scripts/unit/test_scheduler.py`

**工作项**：

- 引入内部 work item 概念：
  - `kind = "task"` 或 `"segment"`
  - `work_id`
  - `parent_task_id`
  - `audio_duration_sec`
- 调度器继续按 `audio_duration_sec` 估算。
- `_dispatch_queued_tasks()` 在调度前筛选：
  - 普通 `QUEUED` task
  - 分段父任务下的 `PENDING` segment
- 对分段父任务应用并发上限：
  - `min(online_server_count, settings.max_parent_active_segments)`
- work stealing 不得突破单父任务并发上限。

**验收**：

- 普通任务调度测试不回归。
- 单父任务最多只会同时启动指定数量的 segment。
- 多个父任务和短任务混排时，短任务不会被长期饿死。

### 阶段 6：Runner 切分接入

**目标**：在 `PREPROCESSING` 阶段创建 segment 清单。

**涉及文件**：

- 修改：`3-dev/src/backend/app/services/task_runner.py`
- 修改：`3-dev/src/backend/app/storage/repository.py`
- 新增测试：`4-tests/scripts/unit/test_task_runner_segments.py`

**工作项**：

- `_promote_preprocessing_tasks()` 增加分支：
  - 无 duration 或 duration < 阈值：走现有路径。
  - duration >= 阈值：执行 canonical WAV、silencedetect、plan、split、写 segments。
- segment 创建成功后父任务进入 `QUEUED`。
- 切分失败时父任务进入 `FAILED`。
- 避免重复切分：
  - 如果父任务已有 segments，不重复创建。

**验收**：

- 短任务行为不变。
- 长任务会生成 segments。
- 重启 runner 后不会重复切段。
- 切分失败可清晰记录错误。

### 阶段 7：Runner 段级执行与重试

**目标**：让 segment 成为内部执行单元。

**涉及文件**：

- 修改：`3-dev/src/backend/app/services/task_runner.py`
- 修改：`3-dev/src/backend/app/storage/repository.py`

**工作项**：

- 新增 `_execute_segment(segment_id)`。
- 段开始时：
  - segment `DISPATCHED -> TRANSCRIBING`
  - 父任务必要时 `QUEUED -> DISPATCHED -> TRANSCRIBING`
- 段成功时：
  - 写入 `raw_result_json`
  - segment `SUCCEEDED`
  - 更新父任务 progress
  - 校准调度器 RTF，使用段长而不是父任务总时长
- 段失败时：
  - 记录 server failure
  - 未达重试上限：segment 回到 `PENDING`
  - 达重试上限：父任务 `FAILED`
- 父任务取消后，不再启动新 segment。

**验收**：

- 单段失败可以迁移到其他服务器重试。
- 某段重试耗尽后父任务失败。
- 父任务 progress 按已完成有效音频时长聚合。
- 多段并发完成不会重复触发父任务终态。

### 阶段 8：父任务合并与终态

**目标**：所有 segment 完成后合并并写入父任务最终结果。

**涉及文件**：

- 修改：`3-dev/src/backend/app/services/task_runner.py`
- 修改：`3-dev/src/backend/app/services/result_merger.py`
- 修改：`3-dev/src/backend/app/storage/file_manager.py`（如需段文件清理辅助）

**工作项**：

- 新增 `_maybe_finalize_segmented_task(task_id)`。
- 检查所有 segments：
  - 任一 `FAILED` 且重试耗尽：父任务失败。
  - 全部 `SUCCEEDED`：触发合并。
- 合并过程需要幂等保护：
  - 避免多个 segment 同时完成时重复合并。
  - 可使用内存锁加数据库状态二次检查。
- 合并成功：
  - 写 JSON/TXT/SRT。
  - 父任务 `SUCCEEDED`。
  - 触发 callback。
  - 可异步清理段文件。
- 合并失败：
  - 父任务 `FAILED`。
  - 保留段文件和段结果排障。

**验收**：

- 多段任务最终能生成完整结果。
- 合并只执行一次。
- callback 行为与普通任务一致。
- 失败时保留足够排障信息。

## 七、关键风险与缓解

| 风险 | 等级 | 缓解 |
|---|---|---|
| `task_runner.py` 改动范围过大 | 高 | 先做离线算法闭环，再接入 runner；每阶段有独立测试 |
| 段级重试与父任务级重试冲突 | 高 | 普通任务保留父级重试，分段任务使用段级重试 |
| `ensure_wav()` 不能保证 canonical 格式 | 中 | 新增 canonical WAV 逻辑 |
| silencedetect 对真实噪声不稳定 | 中 | 参数配置化，保留硬切兜底和 overlap |
| 多段同时完成导致重复合并 | 中 | 内存锁 + 数据库二次检查 |
| 单个长任务占满集群 | 中 | 单父任务活跃段上限 `min(online, 3)` |
| 合并后时间戳重复或乱序 | 中 | keep 区间过滤 + 单调性测试 |
| 长音频测试素材不足 | 中 | 用 ffmpeg 生成可控 20+ 分钟测试音频 |

## 八、首轮最小可交付范围

首轮建议不要直接追求完整 E2E，而是交付一个可验证的最小闭环：

- `TaskSegment` 模型和 migration。

## 九、2026-04-25 切分策略优化补充

### 9.1 背景

阶段 0–9 完成后，长音频测试暴露出切分策略还需要进一步优化：

1. 搜索窗口过窄：当 10–11 分钟窗口内无合适静音段时，当前策略会直接硬切，可能切断语音。
2. 缺少降级策略：硬切前未尝试寻找更短的句间停顿，例如 500–700ms 的弱静音。
3. 用户缺少控制入口：切分当前属于系统自动行为，CLI/API 未提供任务级开关。
4. 缺少质量评估闭环：还不能量化 VAD 切分对文本完整性、边界重复、漏字和时间戳连续性的影响。

本节作为 0424 主实施计划的优化阶段，不改变“父任务外显、segment 内部执行”的总体架构，只补强切分策略、任务级控制和评估方法。

### 9.2 优化任务一：切分策略渐进式搜索与降级

**目标**：`plan_segments()` 在找不到理想断点时，逐步扩大搜索窗口并降低静音阈值，硬切只作为最终兜底。

**渐进搜索策略**：

```text
从当前位置 pos 开始：
第 1 轮：[pos+600s, pos+660s]  找 >= 0.8s 的静音，取最长
第 2 轮：[pos+660s, pos+720s]  找 >= 0.8s 的静音，取最长
第 3 轮：[pos+720s, pos+780s]  找 >= 0.8s 的静音，取最长
降级轮：[pos+600s, pos+780s]  找 >= 0.3s 的静音，取最长
硬切：  pos+780s              最终兜底
```

**配置项调整**：

| 配置项 | 旧值 | 新值 | 说明 |
|---|---:|---:|---|
| `segment_target_duration_sec` | 600 | 600 | 不变，搜索起点 |
| `segment_max_duration_sec` | 660 | 780 | 扩展为 13 分钟硬切上限 |
| `segment_search_step_sec` | 新增 | 60 | 每轮搜索窗口宽度 |
| `segment_search_max_rounds` | 新增 | 3 | 渐进搜索最大轮数 |
| `segment_fallback_silence_sec` | 新增 | 0.3 | 降级轮最短静音阈值 |

**算法要点**：

- 第 1–N 轮按固定窗口查找满足主静音阈值的候选点。
- 每轮候选点以静音区间中点作为切点，并优先选择最长静音。
- 若主搜索失败，则在 `[target, maximum]` 整体范围内按降级静音阈值查找。
- 仍无候选点时才在 `pos + maximum` 硬切。
- 原有尾段合并、overlap 和 keep 区间语义保持不变。

**影响范围**：

- `3-dev/src/backend/app/services/audio_preprocessor.py`：重写或扩展 `plan_segments()`。
- `3-dev/src/backend/app/config.py`：新增渐进搜索配置项。
- `4-tests/scripts/unit/test_audio_segmentation.py`：补充各搜索轮次、降级轮和硬切场景测试。

**验收标准**：

- `plan_segments()` 支持 3 轮渐进搜索、降级搜索和硬切兜底。
- 配置项支持环境变量覆盖。
- 新增测试覆盖第 1/2/3 轮命中、降级命中、无静音硬切、尾段合并和短音频不切分。

### 9.3 优化任务二：CLI/API 增加 `auto_segment` 控制参数

**目标**：让用户在任务级别控制是否启用 VAD 切分，便于人工规避边界风险，也便于质量评估做 on/off 对照。

**参数定义**：

| 值 | 语义 |
|---|---|
| `auto` | 默认值，系统根据时长阈值自动决定是否切分 |
| `on` | 强制启用切分，即使音频低于默认阈值也执行 VAD 分析 |
| `off` | 强制关闭切分，整文件直接转写 |

**数据流**：

```text
CLI --auto-segment=off
  -> TaskItemCreate.options = {"auto_segment": "off", ...}
    -> Task.options_json = '{"auto_segment": "off", ...}'
      -> task_runner._promote_preprocessing_tasks() 读取 options_json
        -> needs_segmentation 判断时加入 auto_segment 策略
```

**改动点**：

- `3-dev/src/backend/app/schemas/task.py`：`TaskItemCreate.options` 支持 `auto_segment` 字段。
- `3-dev/src/backend/app/api/tasks.py`：`create_tasks()` 继续透传 options。
- `3-dev/src/backend/app/services/task_runner.py`：`_promote_preprocessing_tasks()` 读取 `options_json` 并参与是否切分判断。
- `3-dev/src/backend/cli/commands/transcribe.py`：新增 `--auto-segment` 选项。
- `3-dev/src/backend/cli/commands/task.py`：`create` 命令新增 `--auto-segment` 选项。
- `3-dev/src/backend/cli/api_client.py`：确保 `create_tasks()` options 透传。

**验收标准**：

- CLI `--auto-segment=auto|on|off` 端到端可用。
- API `options.auto_segment` 可写入、可读取、可生效。
- `off` 明确跳过 VAD 分析和物理切段。
- `on` 明确绕过时长阈值，但仍遵守切分算法本身的安全边界。
- 未传参时保持现有自动切分行为。

### 9.4 优化任务三：切分前后输出质量对比评估

**目标**：建立可重复的质量评估闭环，量化 VAD 切分对转写结果的影响。

**评估方法**：

1. 选取 3 个长音频文件分别做两组转写：
   - 基准组：`auto_segment=off`，整文件直接转写。
   - 实验组：`auto_segment=on`，VAD 切分后并行转写再合并。
2. 对比维度：
   - 文本完整性：基准组字数 vs 实验组字数，计算缺失/多余比例。
   - 切分边界质量：切分点附近是否有重复句、断句、漏字。
   - 时间戳连续性：实验组时间戳是否单调递增、无异常跳变。
   - 整体相似度：使用编辑距离、BLEU 或其他轻量文本相似度指标。

**输出物**：

- 质量评估脚本。
- 对比评估报告，按项目规范写入 `4-tests/batch-testing/outputs/benchmark/`，文件名带 `-YYYYMMDD` 后缀。

**验收标准**：

- 评估脚本可指定同一音频的 `auto_segment=off/on` 两组任务。
- 输出报告包含字数差异、相似度、时间戳单调性和边界片段检查结果。
- 评估结果能够反向指导 `segment_silence_noise_db`、`segment_silence_min_duration`、`segment_fallback_silence_sec` 和窗口参数调优。

### 9.5 优先级与依赖

```text
任务一：切分策略改进
  -> 任务二：CLI/API auto_segment 参数
    -> 任务三：质量评估脚本与报告
```

任务一优先级最高，因为它直接影响后续基准和质量评估结果。任务二用于提供可控对照入口，任务三依赖前两项完成后再执行。

### 9.6 优化任务四：三级分段切分力度（10m/20m/30m）

**背景**：当前分段策略固定以 10 分钟为切分目标。但实际使用中，部分用户偏好处理较长音频且不希望频繁切分——频繁切分在边界处可能引入微小的质量损失。同时，库中大量长文件时，较大的切分粒度能减少 segment 数量和合并开销。

**目标**：让用户通过 CLI/API 选择 10 分钟、20 分钟或 30 分钟三个切分级别，系统据此选择对应的参数预设。默认仍为 10 分钟级别。

**三级参数预设**：

| 级别 | `segment_target_duration_sec` | `segment_max_duration_sec` | 渐进搜索区间 | `segment_min_file_duration_sec` |
|------|---:|---:|---|---:|
| `10m`（默认） | 600 | 780 | 10→11→12→13 分钟 | 600 |
| `20m` | 1200 | 1380 | 20→21→22→23 分钟 | 1200 |
| `30m` | 1800 | 1980 | 30→31→32→33 分钟 | 1800 |

> 注：`segment_search_step_sec=60`、`segment_search_max_rounds=3`、`segment_fallback_silence_sec=0.3` 在三个级别中保持不变。渐进搜索和降级逻辑完全复用，只是起点和上限不同。

**参数定义**：

| 值 | 语义 |
|---|---|
| `10m` | 默认值，10 分钟级别切分（现有行为） |
| `20m` | 20 分钟级别切分，减少切分次数 |
| `30m` | 30 分钟级别切分，最低切分频率 |

**数据流**：

```text
CLI --segment-level=20m
  → TaskCreateRequest.segment_level = "20m"
    → Task.options_json = '{"segment_level": "20m", ...}'
      → task_runner._promote_preprocessing_tasks() 读取 options_json
        → 根据 segment_level 选择参数预设传入 plan_segments()
```

**改动点**：

1. `config.py`：新增 `SEGMENT_LEVEL_PRESETS` 字典，定义三级参数预设。
2. `schemas/task.py`：`TaskCreateRequest` 新增 `segment_level` 字段（`Literal["10m", "20m", "30m"]`，默认 `"10m"`）。
3. `api/tasks.py`：`create_tasks()` 透传 `segment_level` 到 `options_json`。
4. `task_runner.py`：`_promote_preprocessing_tasks()` 根据 `segment_level` 查找参数预设，传入 `_create_segments_for_task()`。`_create_segments_for_task()` 接受可选的 target/max 参数，覆盖全局配置。
5. `audio_preprocessor.py`：`plan_segments()` 已支持参数化调用，无需修改。
6. `cli/commands/transcribe.py` + `cli/commands/task.py`：新增 `--segment-level` 选项。
7. `cli/api_client.py`：`create_tasks()` 透传 `segment_level`。

**可行性分析**：

- **算法零改动**：`_find_best_cut()` 和 `plan_segments()` 已完全参数化，三个级别只是传入不同的 `target_ms`/`maximum_ms`，核心逻辑完全复用。
- **测试低风险**：已有的 12 个 `TestFindBestCut` + 20+ 个 `TestPlanSegments` 用例已充分验证渐进搜索和降级逻辑，20m/30m 级别只是不同参数组合，可补充少量参数化测试。
- **与 `auto_segment` 正交**：`segment_level` 控制"切多大"，`auto_segment` 控制"切不切"，两者独立。`auto_segment=off` 时 `segment_level` 被忽略。

**验收标准**：

- [x] CLI `--segment-level=10m|20m|30m` 端到端可用。
- [x] API `segment_level` 字段可写入、可读取、可生效。
- [x] 20m 级别：20 分钟音频不切分，40 分钟音频切为 2 段，切点在 20~23 分钟区间。
- [x] 30m 级别：30 分钟音频不切分，60 分钟音频切为 2 段，切点在 30~33 分钟区间。
- [x] 默认行为（未传参）保持 10m 级别不变。
- [x] `auto_segment=off` 时忽略 `segment_level`。
