## C-1：批量结果导出同名 stem 覆盖（P1）

### 问题描述

当批次中存在相同 stem 但不同扩展名的文件（如 `tv-report-1.wav` 和 `tv-report-1.mp4`）时，导出结果写到同一路径，后写的文件静默覆盖前一个，**导致数据丢失**。

影响两处：

- `cli/commands/transcribe.py:274` — `_run_batch` 即时下载
- `cli/commands/task.py:229-234` — `task result --group` 多格式下载

### 解决方案

**采纳审查工程师建议：保留原始文件扩展名，而非计数器去重。**

原方案（计数器）生成 `tv-report-1_result.txt` / `tv-report-1_1_result.txt`，有两个缺点：
- 顺序依赖：同批次重跑，任务返回顺序不同会导致 `_1` 后缀落到不同文件
- 不可回溯：无法从文件名判断对应的是 .wav 还是 .mp4

改用**保留原始扩展名**方案：

```python
# 用完整文件名（含扩展名）作为 stem 基础
# tv-report-1.wav → tv-report-1.wav_result.txt
# tv-report-1.mp4 → tv-report-1.mp4_result.txt
stem = file_name_map.get(tid, tid[:12])  # 这里用完整文件名而非 Path.stem
dest = output_dir / f"{stem}_result{suffix}"
```

优点：确定性、可回溯、与人工排查习惯一致。

### 需要改动的文件

- `cli/commands/transcribe.py` — `_run_batch` 下载段
- `cli/commands/task.py` — `_download_group_results` 函数

### 回归测试

- 单元测试：验证同 stem 不同扩展名文件输出不互相覆盖
- 手动回归：使用含 `tv-report-1.wav` + `tv-report-1.mp4` 的批次确认输出完整

- [ ] 修复 `transcribe.py`

## C-2B：ffprobe 缺失时无时长兜底估算 + 文案修正（P1）

### 问题描述

**行为链路**（ffprobe 缺失时）：

1. `metadata.py:39` — 返回 `MediaMetadata(error="ffprobe not found...")`
2. `upload.py:85` — 仅打印 warning，`duration_sec` 保持 `None`
3. `task_runner.py:177-178` — `audio_duration` 回落到 `0.0`

```python
audio_duration = 0.0
if task.file and task.file.duration_sec:
    audio_duration = task.file.duration_sec
```

结果：所有任务被视为"零时长"，LPT 调度退化为仅 `overhead`（5s），完全失去容量感知。

**文案声称有 fallback，实际不存在**：

- `diagnostics.py:109` — `"will use file-size heuristic"`
- `README.md:26` — `"ffmpeg（可选，用于音频元信息提取和时长估算）"`

### 解决方案

**1. 在 `upload.py` 中补时长粗估（queue hint，非精确 duration）**

```python
def _estimate_duration_from_size(size_bytes: int, filename: str) -> float | None:
    ext = Path(filename).suffix.lower()
    bitrate_map = {
        ".wav": 32_000, ".flac": 24_000,
        ".mp3": 16_000, ".m4a": 16_000, ".aac": 16_000, ".ogg": 16_000,
        ".mp4": 20_000, ".mkv": 20_000, ".webm": 20_000, ".avi": 20_000, ".mov": 20_000,
    }
    bps = bitrate_map.get(ext)
    if bps and size_bytes > 0:
        return size_bytes / bps
    return None
```

> 注意：此估算只作为调度排序的粗略参考（queue hint），不应被视为精确时长。MP4 视频的音频占比差异大，估算偏差可能达到 2-5 倍。

**2. 修正文案**

- `diagnostics.py:109` → `"ffprobe not found; audio duration will use file-size heuristic (rough estimate, may be inaccurate for video files)"`
- `README.md:26` → 配合 D-2 重写

## S-1：LPT 调度负载严重不均 + RTF 可信度分析（P2，需分析后决策）

### 背景

从多轮测试中观察到一致的负载不均现象：

**E2E 测试汇总（5 任务 × 5 次 = 25 个任务）：**

| 服务器 | RTF | 分配任务数 | 占比 |
|--------|-----|-----------|------|
| funasr-10095 | 0.124 | 20 | 80% |
| funasr-10097 | 0.656 | 5 | 20% |
| funasr-10096 | 0.737 | **0** | 0% |

**量化课程批次（8 个任务）：**

| 服务器 | 分配任务数 | 占比 | 耗时范围 |
|--------|-----------|------|---------|
| funasr-10095 | 7 | 87.5% | 29s - 197s |
| funasr-10097 | 1 | 12.5% | 21s |
| funasr-10096 | 0 | 0% | — |

### 算法推演（8 任务案例）

以 8 任务量化课程为例，精确推演 LPT + EFT 算法行为。

**服务器参数**（均为 `max_concurrency=4`）：

| 服务器 | RTF | 处理公式 |
|--------|-----|---------|
| funasr-10095 | 0.124 | p = dur × 0.124 + 5 |
| funasr-10097 | 0.656 | p = dur × 0.656 + 5 |
| funasr-10096 | 0.737 | p = dur × 0.737 + 5 |

从实际耗时反推音频时长（第4集在 10095 上耗时 197s → 音频 ≈ (197-5)/0.124 ≈ 1548s），**LPT 降序排列后逐个分配**：

| 轮次 | 任务 | 音频估长 | 分配到 | 选择原因 |
|------|------|---------|--------|---------|
| 1 | 第4集 | ~1548s | **10095 slot-0** | 0+197=197 < 0+1146=1146(10096) |
| 2 | 第6集 | ~1065s | **10095 slot-1** | 0+137=137 < 0+789=789(10096) |
| 3 | 第5集 | ~871s | **10095 slot-2** | 0+113=113 < 0+647=647(10096) |
| 4 | 第7集 | ~613s | **10095 slot-3** | 0+81=81 < 0+457=457(10096) |
| 5 | 第3集 | ~516s | **10095 slot-3** | 81+69=150 < 0+385=385(10096) |
| 6 | 第2集 | ~274s | **10095 slot-2** | 113+39=152 < 0+207=207(10096) |
| 7 | 第8集 | ~194s | **10097 slot-0** | 0+132=132 < 137+29=166(10095) |
| 8 | 第1集 | ~24s | **10097 slot-1** | 0+21=21 < 150+8=158(10095) |

**结果**：10095 获得 6 个，10097 获得 2 个，10096 获得 0 个。算法的 makespan 约 197s。

> 注：实际观测是 7:1:0 而非 6:2:0，可能因实际音频时长、penalty_factor、或 RTF 微调导致少量偏差，但模式一致。

### 关键发现：RTF 可信度问题

**三台服务器部署在同一台物理主机**（`100.116.250.20`，端口 10095/10096/10097）。

这意味着 RTF 差异（0.124 vs 0.656 vs 0.737，相差 5-6 倍）**极可能不反映真实硬件差异**，而是来自：

1. **benchmark 时序差异**：先跑的节点独占 GPU，RTF 偏低；后跑的节点受前一个干扰，RTF 偏高
2. **GPU 资源竞争**：三个 FunASR 实例共享同一 GPU，相互干扰导致不同时间段的性能波动
3. **样本偏差**：benchmark 只发一个短音频，单次测量随机性大

如果三台服务器的实际处理能力接近（都在 RTF 0.3-0.5 范围），那当前算法基于 **失真的 RTF 输入** 做出的"把 87% 任务堆到一台"的决策就不是最优的，而是被错误数据误导了。

### 算法本身是否有问题？

**从纯 makespan 优化角度**：算法在给定 RTF 输入下的行为是**数学正确**的。如果 10095 真的比 10096 快 6 倍，把大部分任务给 10095 确实能最小化完成时间。

**但存在两个结构性盲点**：

1. **不建模批内并发惩罚**：`get_effective_rtf()` 中的 `penalty_factor * running_tasks` 使用的是**调度前的** `running_tasks`，在批量调度循环中不递增。当 4 个任务同时跑在同一 GPU 上时，实际 RTF 会上升，但调度器不感知这一点。

2. **RTF 收敛陷阱**：被饥饿的节点永远得不到任务 → 永远无法通过实际完成来更新 RTF → RTF 永远是旧 benchmark 值 → 继续被饥饿。这是一个正反馈死循环。

### 解决方案：按速度比例预分配名额（本轮实施）

经讨论，决定在本轮修改调度算法，采用**按速度比例预分配 + 最低保障**策略：

**核心改动**：在 `scheduler.py` 的 `schedule_batch` 中，分配前先按各服务器速度比例计算名额，**每台 ONLINE 服务器至少分 1 个任务**，然后在名额约束下执行 LPT + EFT 分配。

**第一步：按速度比例计算名额**

```python
def _allocate_quotas(self, task_count: int, servers: list[ServerProfile]) -> dict[str, int]:
    speeds = {s.server_id: 1.0 / max(self.get_base_rtf(s), 0.01) for s in servers}
    total_speed = sum(speeds.values())
    quotas = {}
    remaining = task_count
    for sid, spd in speeds.items():
        q = max(1, round(task_count * spd / total_speed))
        quotas[sid] = q
        remaining -= q
    # 溢出/不足修正
    if remaining > 0:
        fastest = max(speeds, key=speeds.get)
        quotas[fastest] += remaining
    elif remaining < 0:
        for sid in sorted(speeds, key=speeds.get):
            take = min(-remaining, quotas[sid] - 1)
            quotas[sid] -= take
            remaining += take
            if remaining >= 0:
                break
    return quotas
```

**第二步：在名额约束下做 LPT + EFT**

保留 LPT 排序和 EFT 选 slot 逻辑，但每台服务器达到名额上限后跳过：

```python
quotas = self._allocate_quotas(len(tasks), online_servers)
assigned_count = {s.server_id: 0 for s in online_servers}

for task_info in task_estimates:  # 已 LPT 排序
    eligible_slots = [s for s in slots
                      if assigned_count[s.server_id] < quotas[s.server_id]]
    if not eligible_slots:
        eligible_slots = slots  # fallback
    best_slot = min(eligible_slots, key=lambda s: ...)
    assigned_count[best_slot.server_id] += 1
```

**8 任务案例的预期效果**：

速度比 `1/0.124 : 1/0.656 : 1/0.737 ≈ 8.1 : 1.5 : 1.4`，归一化后 `73% : 14% : 13%`。
8 个任务按比例 → `6 : 1 : 1`（每台至少 1 个）。

| 服务器 | 当前分配 | 改后分配 |
|--------|---------|---------|
| funasr-10095 | 7 | 6 |
| funasr-10097 | 1 | 1 |
| funasr-10096 | **0** | **1** |

makespan 影响极小：最短任务（~24s 音频）在 10096 上约 23s，远低于 10095 上最长任务的 197s。
